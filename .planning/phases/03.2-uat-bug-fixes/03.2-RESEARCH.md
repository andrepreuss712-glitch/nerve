## Architecture Patterns

### DB Migration Pattern (for UX-07)
The existing migration block in `app.py` (lines 60–89) uses:
```python
for col, typedef in [
    ('preferred_language', "VARCHAR(10) DEFAULT 'de'"),
]:
    try:
        conn.execute(text(f'ALTER TABLE users ADD COLUMN {col} {typedef}'))
        conn.commit()
        print(f"[DB] Migration: added users.{col}")
    except Exception:
        pass
```
This pattern is idempotent — SQLite silently fails if column already exists, matching the existing approach.

## Bug-by-Bug Findings

### BUG-01: Anruf-Button / Training start broken

**What was found:** `startTraining()` in `training.html` line 468 — correctly calls `POST /training/start` via fetch. The `training_start()` route in `routes/training.py` is complete and correct. No obvious server-side blocker.

**Likely root cause:** The `t-callBtn` is `disabled` by default and only enabled when `t-profileSelect` has a value (`checkReady()`). If no profile exists, the button stays disabled. Also, `startFreizeichen()` uses `AudioContext` — browsers require a user gesture before allowing `AudioContext`. Since the button click IS a user gesture, this should work. The `AudioContext` in `startFreizeichen` may be blocked if the page loaded with a suspended context.

**Fix direction:** Verify `checkReady()` is wired to profile select `onchange` (it is, line 239). The `AudioContext` issue is benign — `catch` handles it. The real issue reported may be that the PROFILE dropdown is empty (no profiles seeded). Check that profiles exist for the org. If profiles exist and button is still disabled, add `console.log` to `checkReady()`. No code change required unless the profile select population is broken.

**Confidence:** MEDIUM — cannot reproduce without live session, but code path looks correct.

### BUG-02 / BUG-05: Microphone not working in /training and /live

**What was found (training):** PTT mic in training.html uses `getUserMedia()` lazily — only when user presses the PTT button (`t-pttBtn`). This is correct browser API usage. The button calls `startRec()` which calls `navigator.mediaDevices.getUserMedia({audio:true})`. This ONLY works over HTTPS or localhost.

**What was found (live):** `app.js` has NO `getUserMedia()` call at all. The live mic (`/live`) is server-driven — PyAudio runs on the Hetzner VPS, capturing audio from the server's microphone (which doesn't exist on a VPS). This is the fundamental architecture: `live_session.py` runs `pyaudio.open()` on the server. On VPS without audio hardware, this fails silently.

**Critical finding:** The live mic architecture is server-side PyAudio, not browser WebRTC. INFRA-05 says "PyAudio is NOT in server-requirements" — meaning the VPS intentionally has no audio capture. The live mic simply cannot work on VPS with current architecture. The browser shows no mic permission dialog for `/live` because there is no `getUserMedia()` call.

**For training mic (BUG-05):** Should work over HTTPS on getnerve.app. If it doesn't, check: (1) HTTPS is active (it is per Phase 3), (2) browser has not blocked mic permission globally, (3) `navigator.mediaDevices` is defined (yes, requires HTTPS). No code change likely needed — this is a browser permission issue.

**For live mic (BUG-02):** This is an architectural constraint. The VPS has no audio. Options: (a) Accept this as known limitation — live mode demo only on VPS, (b) implement browser-based mic capture via WebRTC and stream to Deepgram client-side then send transcripts to server. Option (b) is out of scope for this phase per CONTEXT.md. The timer not starting (BUG-03) is a consequence — no transcripts arrive because no mic works.

**Confidence:** HIGH for training PTT (code is correct, HTTPS + permissions). HIGH for live mic being server-side PyAudio by design.

### BUG-03: Live session timer doesn't start

**What was found:** `startSessionTimer()` in `app.js` line 13 is called inside the `socket.on('transcript')` handler (line 140) only when a `final` transcript arrives. Since no transcripts arrive on VPS (no server mic), the timer never starts.

**Fix direction:** Call `startSessionTimer()` on `socket.on('connect')` instead — or on a button click. This makes the timer start as soon as the Socket.IO connection is established, not waiting for first transcript. This is an independent fix from the mic issue.

**Code location:** `static/app.js` line 120 — the `socket.on('connect')` handler. Add `startSessionTimer()` there.

**Confidence:** HIGH.

### BUG-04: Language switch partially applied — Polish UI strings visible after language change

**What was found:** `selectLanguage(code)` in `training.html` line 450:
```javascript
function selectLanguage(code) {
  currentLanguage = code;
  tUI = (TRAINING_LANGUAGES[code] || TRAINING_LANGUAGES['de']).ui;
  // Updates: t-lang-btn active class, t-sendBtn textContent, t-helpBtn textContent
}
```
This only updates the send button and help button. It does NOT update:
- The "■ Beenden" button in `t-chat-header` (hardcoded in HTML as `onclick="endTraining()"` with text "■ Beenden")
- Any dynamic strings that use `tUI.*` are fine — they read `tUI` at call time
- BUT: the initial render at page load sets `currentLanguage = 'de'` and `tUI = TRAINING_LANGUAGES['de'].ui`

**The Polish bug "Wyślij →" and "Co powinienem powiedzieć?" appearing:** These ARE the Polish translations (`senden` and `hilfe_btn`). They appear because when language is switched to 'pl' BEFORE starting a training, `selectLanguage('pl')` is called and updates the buttons. The bug report says they show when they shouldn't — possible interpretation: the SEND button shows Polish text even when German is selected, meaning `selectLanguage` is being called with wrong code.

**More likely root cause:** After `startTraining()` returns, line 495: `if (data.ui) tUI = data.ui;` — this replaces `tUI` with the server-returned UI dict. The server returns `lang_config['ui']` based on `sprache` param. If the language buttons are clicked but `currentLanguage` is not sent correctly to the server, there's a mismatch. But looking at the code, `currentLanguage` is sent as `sprache` in the POST body (line 481). This looks correct.

**Actual gap:** The "Beenden" button text in the chat header is hardcoded in HTML: `<button class="btn btn-danger btn-sm" onclick="endTraining()">■ Beenden</button>` — it is never updated by `selectLanguage()`. This should use `tUI.beenden` after training starts. Similarly, the `t-sendBtn` initial text "Senden →" is hardcoded in HTML.

**Fix:** In `initChat()` (line 533), update all translatable button texts using the current `tUI`:
- `t-sendBtn`: `tUI.senden`
- The Beenden button: `tUI.beenden`

**Confidence:** HIGH (code confirmed).

### BUG-06: Auswertung shows fake coaching when no conversation occurred

**What was found:**

**Training side:** `endTraining()` calls `POST /training/end` unconditionally. No word or time check. `trainingSecs` is a JS variable tracking session seconds. `sessionMeta` tracks conversation turns. The server `training_end()` route calls `generate_scoring()` with whatever `history` exists — if only the opening KI message is in history (0 user turns), it will still attempt scoring.

**Live side:** `beenden()` in `app.js` line 257 calls `POST /api/beenden` then shows postcall. The `words` variable counts words from transcripts. `sessionSeconds` tracks time.

**Fix locations:**
- Training: In `endTraining()` before the fetch, check `trainingSecs < 10 || (message count from history == 0)`. Count user messages by tracking `msgCnt` or checking for any `addOutMsg` calls.
- Live: In `beenden()` after stopping timer, check `sessionSeconds < 10 || words < 20`. If below threshold, show placeholder message instead of postcall overlay.

**Confidence:** HIGH.

## DESIGN-14: DSGVO Banner Overlap

**What was found:**
- `.dsgvo-banner` CSS: `position:fixed; bottom:52px; left:50%; transform:translateX(-50%); z-index:300`
- `.footer` CSS: `padding:8px 20px; border-top:1px solid var(--glass-border); background:var(--page-bg); display:flex; flex-shrink:0`
- The footer is ~36px tall. DSGVO banner sits at `bottom:52px` which places it above the footer.
- `.panel-sprachanalyse` has `max-height:110px` and contains `.metric-circles` with 44px circles.
- The emotion circles are INSIDE the flex layout (`.main`), NOT position:fixed. They flow above the footer naturally.

**Overlap scenario:** On smaller screens or when the banner is wide, the banner at `bottom:52px` can overlap the emotion circles row which is the last row of `.main` before the footer. The `.main` section is `flex:1; overflow:hidden` — its bottom edge is just above the footer.

**Fix approach:**

## Common Pitfalls

### Pitfall 1: Kompakt overlay breaks Socket.IO
**What goes wrong:** If the fix for UX-09 used `window.open()` instead of CSS overlay, the Socket.IO connection in the new window would not exist — all real-time data stops.
**How to avoid:** Keep everything in same window. CSS `position:fixed` overlay only. Verified by the locked decision in CONTEXT.md.
**Warning signs:** Coaching panel goes silent, timer stops in compact mode.

### Pitfall 2: DB migration not running on VPS restart
**What goes wrong:** `app.py` migration block runs inside a `with engine.connect() as conn` block at startup. If the migration is added to the list but there is a syntax error in the new tuple, the entire migration loop fails silently (caught by outer `except`).
**How to avoid:** Add new column tuple to the EXISTING list inside the established `for col, typedef in [...]` block. Test locally first.
**Warning signs:** `preferred_language` column missing after deploy, settings save returns 500.

### Pitfall 3: `selectLanguage()` update misses training-started state
**What goes wrong:** `selectLanguage()` updates `tUI` global but if called AFTER `startTraining()` (while in chat phase), the server has already set `tUI = data.ui`. A subsequent language button click would update `tUI` but the training has already started with a different language for the AI persona.
**How to avoid:** Hide language selection buttons during active training (they are on the setup phase which is hidden anyway — this is not an issue in practice).

### Pitfall 4: Custom select breaks scenario delete button logic
**What goes wrong:** `renderScenarioDropdown()` populates the scenario `<select>`. The delete button logic reads `sel.value`. If the native select is replaced with a hidden input, JS that reads `document.getElementById('t-scenarioSelect').value` continues to work — but `onchange` events on the hidden input are not fired by programmatic `.value =` assignment.
**How to avoid:** After setting the hidden input value programmatically in the custom dropdown, manually call `onScenarioChange()`.

### Pitfall 5: Timer starts too early (BUG-03 fix)
**What goes wrong:** Moving `startSessionTimer()` to `socket.on('connect')` means the timer starts even before the user has spoken anything. This is actually the desired behavior per the UAT requirement ("timer counts up from session start"). But it means the timer runs even when the session is paused.
**How to avoid:** The `togglePause()` handler should also stop/resume the timer. Check if `stopSessionTimer()` is called on pause — it is not currently. Add timer pause logic to `togglePause()`.
