## Decisions Made

- Using JS to add/remove `paddingBottom` rather than CSS approach ŌĆö the DSGVO banner and `.panel-sprachanalyse` are not siblings/ancestors in the DOM, so CSS selectors cannot target them from the banner. JS is tied to the exact show/hide lifecycle.
- Hidden `<input type="hidden" id="t-scenarioSelect">` rather than renaming the id ŌĆö all downstream code reading `.value` on `#t-scenarioSelect` continues working without modification.
- `window._pendingDeleteId` rather than closure ŌĆö the delete modal confirm button cannot carry closure scope, so a window-level variable bridges the modal show and the async delete call.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Updated saveScenario() auto-select for custom dropdown**
- **Found during:** Task 2 (custom dropdown implementation)
- **Issue:** `saveScenario()` directly set `sel.value = data.id` on the old native select. With the hidden input pattern, this still sets the value but does NOT update the visible display label.
- **Fix:** Updated auto-select code to call `selectScenarioOption(String(data.id), label)` which updates both the hidden input and the visible label.
- **Files modified:** `templates/training.html`
- **Committed in:** `c5a41b9` (Task 2 commit)

**2. [Rule 1 - Bug] Added selectScenarioOption() call in confirmDeleteScenario() to reset display**
- **Found during:** Task 2 (delete modal logic)
- **Issue:** After deleting a scenario the original code only called `loadScenarios()`. With custom dropdown, the display label would still show the deleted scenario's name.
- **Fix:** Added `selectScenarioOption('', 'ŌĆö Kein Szenario ŌĆö')` before `loadScenarios()` in `confirmDeleteScenario()`.
- **Files modified:** `templates/training.html`
- **Committed in:** `c5a41b9` (Task 2 commit)
