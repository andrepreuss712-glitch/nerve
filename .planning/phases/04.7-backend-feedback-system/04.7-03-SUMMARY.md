# Phase 04.7 Plan 03: Daten-Tracking haerten Summary

**One-liner:** Granulare EWB-Klick-Persistenz via `ObjectionEvent` Tabelle — jeder EWB-Button-Klick wird als eigene Row gespeichert, gesammelt im Session-State, persistiert in `api_beenden` vor dem Audit-Log-Eintrag.

## What Was Built

### Task 1 — ObjectionEvent Modell + ewb_clicks In-Session-Sammlung (commit 5a24bcd)

- `database/models.py`: `class ObjectionEvent(Base)` mit `id`, `user_id`, `org_id`, `conversation_log_id`, `einwand_typ`, `success`, `created_at` — kein `avg_deal_value` (Pitfall 2 vermieden, `users.avg_deal_wert` bleibt unveraendert)
- `services/live_session.py`: `state['ewb_clicks'] = []` als initiales Listenelement im state-Dict
- `services/live_session.py`: `reset_session()` loescht `state['ewb_clicks']` via bestehendem `state_lock`-Block
- `services/live_session.py`: `record_ewb_click(einwand_typ, success=False)` Helper — thread-safe Append via `state_lock`

### Task 2 — EWB-Trigger record_ewb_click + Persistenz in api_beenden (commit 9a67ce2)

- `routes/app_routes.py`: `api_ewb_trigger` ruft `record_ewb_click(einwand_typ=einwand_typ, success=False)` nach erfolgreicher Haiku-Antwort
- `routes/app_routes.py`: `api_beenden` — ObjectionEvent Bulk-Insert aus `state['ewb_clicks']` VOR `log_action('session_end', ...)`:
  - liest `ewb_clicks` thread-safe via `ls.state_lock`
  - iteriert und fuegt `ObjectionEvent(...)` via `db_conv.add()` ein
  - `db_conv.commit()` nur wenn mindestens 1 Click vorhanden
