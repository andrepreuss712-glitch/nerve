<decisions>
## Implementation Decisions

### Infrastruktur

**D-01: Email-Provider = Resend**
- `pip install resend`, API-Key als `RESEND_API_KEY` in `.env` und `/etc/nerve/.env` auf Prod.
- EU-Region (Resend unterstützt EU-Datenresidenz).
- 3 initiale Templates: Willkommen nach Signup, Feedback-Bestätigung bei Übernahme in Planung, Passwort-Reset (für Nicht-OAuth-User).
- Absender: `noreply@getnerve.app` (Account/System-Emails) + `feedback@getnerve.app` (Feedback-Antworten).
- DNS (MX/SPF/DKIM) für beide Adressen: **bereits bei Cloudflare eingerichtet** (André 2026-04-08). Phase kann live testen.

**D-02: Superadmin-Detection = DB-Flag + ENV-Seed**
- Neue Spalte `users.is_superadmin BOOLEAN DEFAULT FALSE`.
- ENV-Var `SUPERADMIN_EMAIL` als initialer Seed: beim Login prüft `_login_user`, ob User-Email == `SUPERADMIN_EMAIL` und `is_superadmin=False` → setzt Flag auf True (einmalig, idempotent).
- Decorator `@superadmin_required` prüft `g.user.is_superadmin`.
- Rationale: ENV einfach für Bootstrap (André = erster Superadmin), DB-Flag flexibel für Zukunft (mehrere Trusted Admins ohne Re-Deploy).

**D-03: Screenshot-Upload = Lokal auf VPS**
- Pfad: `/opt/nerve/uploads/feedback/{uuid}.{ext}` (ext = png/jpg/webp).
- Max 5 MB, validiert serverseitig via Flask `MAX_CONTENT_LENGTH` + MIME-Check.
- `screenshot_path` in `feedback` Tabelle speichert relativen Pfad (`feedback/{uuid}.png`), Absolute via Config.
- Geliefert via eigene Admin-Route `GET /admin/feedback/screenshot/<uuid>` mit `@superadmin_required` Gate.
- `deploy.sh` muss `uploads/feedback/*` excluden (keine Überschreibung von User-Uploads beim Deploy).
- **Migration-Pfad für später:** Hetzner Object Storage in Phase 04.14 oder 04.16 falls nötig.

### Admin-UI

**D-04: Flask-Admin Theme = Bootstrap4 Default (MVP)**
- `Admin(app, name='NERVE Admin', template_mode='bootstrap4')` — Standard-Theme, kein Custom-CSS.
- Polish/NERVE-Look kommt in Phase 04.14 (CRM erweitert Admin-UI).
- Custom-Views nur wo unvermeidbar: KPI-Dashboard, Feedback-Notification-Popup, Planungs-Liste.

**D-05: Admin-Bereich = Separate `/admin` Route**
- Flask-Admin mountet unter `/admin` (Standard).
- Kein Sidebar-Eintrag in User-App — klare Trennung.
- Login über bestehenden Auth-Flow (User muss `is_superadmin=True` haben, sonst 403).

### Daten-Level

**D-06: Bestehende Tabellen werden in RESEARCH-Phase geprüft**
- Codebase-Scout hat ergeben: `Session`, `FeedbackEvent`, `Phrase`, `BillingEvent` existieren bereits in `database/models.py`.
- **Research-Task:** RESEARCH.md muss dokumentieren welche Felder da sind, was fehlt, welche Migration nötig ist. Vor Planung essentiell, damit Plans keine Duplikate erzeugen.
- Insbesondere: `Session.result`, `Session.mode`, `FeedbackEvent` vs neue `feedback` Tabelle aus Briefing — wird die bestehende `FeedbackEvent` umgebaut/erweitert oder eine neue parallele Tabelle angelegt?

**D-07: Cold Call Audit-Log-Granularität**
- `objection_events` Tabelle loggt jeden EWB-Button-Klick einzeln (Typ + Erfolg).
- `audit_log` loggt nur Session-Aggregate: Session-Start (Modus), Session-Ende (Dauer + 6 Klicks / 4 Erfolge als JSON in `details`).
- Kein Transkript in beiden Tabellen (DSGVO-konform gemäß CLAUDE.md-Policy).
- Audit-Log ist immutable: keine DELETE/UPDATE-Routen, App-seitig enforced + SQLite-Trigger als Defense-in-Depth.

### Workflow

**D-08: Plan-Split = ~6 atomare Plans**
- Plan 01: **Foundation** — `is_superadmin` Flag, `@superadmin_required` Decorator, ENV-Seed im Login, Flask-Admin Setup mit `/admin` Route (gated), minimaler Smoke-View.
- Plan 02: **Audit-Log** — `audit_log` Tabelle, Immutable-Trigger, `log_action()` Helper, Wire-up in bestehende Routes (Login, Session-Start/Ende, Profil-Änderung).
- Plan 03: **Daten-Tracking härten** — Session-Felder prüfen (aus RESEARCH), `objection_events` Tabelle, `phrases` Erweiterung, `avg_deal_value` in `user_profiles`, EWB-Tracking persistieren.
- Plan 04: **Feedback-Buttons + API** — User-Seite: grüner Sidebar-Button, Modal (Typ + Text + Screenshot), Quick-Ratings nach Training/Live, `POST /api/feedback`-Endpoints, neue `feedback` Tabelle (oder Erweiterung von `FeedbackEvent` — entscheidet RESEARCH).
- Plan 05: **Admin-Dashboard + Ticket-System** — Flask-Admin ModelViews (User, Feedback, AuditLog, Session), KPI-Dashboard Custom-View, Feedback-Notification-Popup beim Login, Ticket-Status-Flow, Planungs-Liste + `planning_feedback_link`, "In Planung übernehmen" → Resend-Email-Trigger.
- Plan 06: **Session-History (Analytics-Umbau)** — bestehende Analytics-Seite zu chronologischer Session-Liste umbauen, Detail-View, Filter, alte Charts entfernen.
- **Reihenfolge (Waves):** 01 ist Voraussetzung → Wave 1. 02+03 können parallel → Wave 2. 04 braucht 01 → Wave 3. 05 braucht 02+03+04 → Wave 4. 06 braucht 01, sonst unabhängig → Wave 2 oder Wave 3.

### Email-Templates (Pflicht in Plan 05 oder eigenem Sub-Task)

Drei initiale Resend-Templates:
1. **Welcome** (getriggert beim Signup, bestehende + OAuth-User): kurze Begrüßung, Link zur App, Ansprechpartner.
2. **Feedback-in-Planung** (getriggert beim "In Planung übernehmen" Klick): Briefing-Text aus 4.3c, Ton "kurz, ehrlich, wertschätzend".
3. **Passwort-Reset** (für Nicht-OAuth-User, zukunftssicher): Token-Link, 1h gültig.

### Claude's Discretion

- Genaues Schema-Delta vs bestehenden `FeedbackEvent` — entscheidet RESEARCH.
- Ob Audit-Log SQLite-Trigger oder nur App-seitig durchgesetzt wird (Pro/Contra im Plan).
- Flask-Admin CustomView-Template für KPI-Dashboard (HTML-Struktur).
- Pagination-Größen für Admin-Tabellen.
- Cookie/Session-Lifetime für Admin-Bereich (gleich wie User oder strenger?).
- Screenshot-Bild-Format-Konvertierung (JPEG-Compression? WebP?).
- Exakte `notification_sent` Duplikat-Schutz-Logik.

<specifics>
## Specific Ideas

- Flask-Admin Version: `flask-admin>=2.0` (Bootstrap4 Template)
- Resend Python-SDK: `resend>=0.7`
- Screenshot-Upload Frontend: `FormData` + `fetch`, keine Extra-Library
- Feedback-Button CSS: bereits im Briefing spezifiziert (grün #00D4AA, subtle glow animation)
- Planungs-Liste UI: einfache HTML-Tabelle mit Status-Dropdown, keine Drag-and-Drop
- Audit-Log: `log_action(user_id, action, target_type, target_id, details)` als Helper in `services/audit.py`
- Feedback-Notification-Popup: einmaliger Check beim `/dashboard` oder `/admin` Load, kein WebSocket-Push
- `beforeunload` Hook für "Es wartet noch Feedback" nur im `/admin` Bereich, nicht in User-App

</specifics>
