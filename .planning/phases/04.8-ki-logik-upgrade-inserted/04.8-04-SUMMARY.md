# Phase 04.8 Plan 04: Active Hint Orchestration + /api/ergebnis Extension Summary

**One-liner:** Per-cycle readiness scoring and active-hint selection wired into `analyse_loop`, fed by 8 new score-signal flags returned by `analysiere_mit_claude`, surfaced via an extended `/api/ergebnis` payload that preserves every legacy key for zero-migration client compatibility.

## What Was Built

### Task 0 — 8 score signal flags in analysiere_mit_claude prompt (`services/claude_service.py`)

Extended `SYSTEM_PROMPT_BASE` with a new "Phase 04.8 — Score-Signale" section documenting 8 optional boolean flags: `einwand_geloest`, `detailfrage`, `budget_erwaehnt`, `naechster_schritt`, `zustimmung`, `konkurrenz`, `zeitdruck_kunde`, `monosyllabisch`. All flags are optional with explicit "bei Unsicherheit false" instruction. No existing schema keys modified. Downstream parser uses `ergebnis.get(key)` already, so missing flags degrade gracefully.

### Task 1 — Readiness + active_hint orchestration in analyse_loop (`services/claude_service.py`)

Added a new try-wrapped block at the end of the analyse_loop try body (after P02 phase-classifier and P03 cold-call-inference blocks, before the outer `except`). Every cycle the block:

1. Reads `score_factors_seen`, `current_phase`, `cold_call_inference` from `ls.state` under `state_lock`.
2. Tallies incremental factor counts from the latest `ergebnis` (`kaufsignal`, `einwand_offen`/`einwand_geloest`, and the 7 new flags).
3. Calls `compute_readiness_score({'score_factors_seen': factors}, [])` → `(score, bucket)`.
4. Builds a candidate list in priority order: critical fehler (P1) → closing+kb_delta kaufsignal (P2) → einwand+gegenargument_1 (P3) → cold_call recommended_next (P4) → tipp (P5). Text fields capped at 120 chars (T-04.8.4-01 mitigation).
5. Calls `select_active_hint(candidates)` for the winner.
6. Calls `dynamic_ewb_buttons(current_phase, base_buttons)` — base falls back to a derived list from the active profile's `einwaende`.
7. Writes `score_factors_seen`, `readiness_score`, `readiness_bucket`, `kaufbereitschaft` (legacy mirror), `active_hint`, `ewb_buttons` to `ls.state` under `state_lock`, and mirrors `ls.kaufbereitschaft = score` for module-global consumers.

Wrapped in try/except — orchestration errors never crash analyse_loop.

### Task 2 — /api/ergebnis payload extension (`routes/app_routes.py::api_ergebnis`)

Added 8 new keys inside the existing `with ls.state_lock:` block:
- `current_phase`, `current_phase_name`, `phase_confidence`
- `readiness_score`, `readiness_bucket`
- `active_hint`
- `ewb_buttons`
- `cold_call_inference`

All legacy keys preserved: `version`, `aktiv`, `ergebnis`, `line_id`, `kaufbereitschaft`, `ewb_top2`. Backward compat guaranteed — existing client JS that reads only legacy keys continues to work unchanged. Defaults match the P01 state schema (readiness_score=30, readiness_bucket='cold', current_phase=1, phase_confidence=0.0).

## Deviations from Plan

**1. [Rule 1 - Bug] Plan example referenced `ergebnis.get('gegenargument')` but schema uses `gegenargument_1`/`gegenargument_2`**
- **Found during:** Task 1 candidate-list construction.
- **Issue:** The plan's snippet built the einwand candidate using `ergebnis.get('gegenargument')`, but `SYSTEM_PROMPT_BASE` never returns that key — it returns `gegenargument_1` and `gegenargument_2`. Using the plan's key verbatim would have made the einwand candidate never trigger.
- **Fix:** Used `ergebnis.get('gegenargument_1')` (the primary counter-argument) for the einwand candidate text.
- **Files modified:** services/claude_service.py
- **Commit:** 2292a03

**2. [Rule 3 - Missing helper] `ls.get_active_profile_ewbs` does not exist**
- **Found during:** Task 1 base_buttons fallback.
- **Issue:** The plan-example `hasattr` guard covered this case but only returned `None` fallback, which would make `dynamic_ewb_buttons` always return empty on unknown phases.
- **Fix:** Inlined a safe derivation from `ls.get_active_profile()` — extracts the `einwaende` list from profile data and maps each entry to its `einwand` text (or `kategorie` as fallback), filtering empties. Wrapped in try/except so profile-load failures fall back to `None`.
- **Files modified:** services/claude_service.py
- **Commit:** 2292a03

**3. [Documentation] kaufbereitschaft legacy key is now a mirror of readiness_score**
- **Note:** Per plan spec, `kaufbereitschaft` is no longer updated by the original deterministic delta logic (`ls.update_kaufbereitschaft(delta)` earlier in the loop still runs but the value it writes is overwritten at the end of the cycle by the orchestration block). This matches RESEARCH Q2 R2 — the new readiness_score is the source of truth, and the legacy key mirrors it so pre-04.8 clients see consistent values. Not a bug, but worth flagging for future debugging: the einwand `-3`/`-5` delta logic at line ~692 is effectively shadowed and could be removed in a later cleanup plan.
- No code change.

## STRIDE Mitigations Implemented

- **T-04.8.4-01 (Info Disclosure — active_hint text in response):** All candidate text fields capped at 120 chars via `str(...)[:120]` before being added to the candidate list. No raw transcript ever forwarded — only Haiku-derived fields (`kritischer_fehler`, `gegenargument_1`, `tipp`, `recommended_next`).
- **T-04.8.4-02 (Tampering — score_factors_seen unbounded growth):** Factor dict keys restricted to a hardcoded allowlist (`kaufsignal`, `einwand_offen`, `einwand_geloest`, plus the 7 new flags) inside the tally block. Dict is reset per session via P01 state schema reset. No user input flows into the key set.
- **T-04.8.4-03 (DoS — per-cycle candidate build):** Accepted. Candidate list construction is O(1) — at most 5 append ops per cycle, `select_active_hint` is a pure `min()` over ≤5 items. `dynamic_ewb_buttons` is a dict lookup + list copy (≤6 items).
- **T-04.8.4-04 (Elevation — /api/ergebnis without login):** Existing `@login_required` decorator on `api_ergebnis` unchanged. Route-level auth unaffected.
