# Phase 04.8 Plan 05: Client-Side Rendering of Active Hint + Phase/Readiness + Dynamic EWB Summary

**One-liner:** Additive DOM (single hint slot + phase/readiness badges) and four new render functions wired into the `/api/ergebnis` polling tick, so every Phase 04.8 server field reaches the UI within 500 ms without touching the existing `#ai` card flow, PiP layout, or Phase 04.3 CSS tokens.

### Task 2 — Render functions + polling wiring (`static/app.js`)

Added four functions in a new block above `triggerEwb()`:

- **`renderActiveHint(hint)`** — If `hint` is null/undefined, hides all three slots and empties their innerHTML. Otherwise picks a symbol (`⚠` for red, `★` for gold, `●` default) and writes a single `.hint-{type}` div with `escHtml(hint.text)` to all three mirrors. XSS-safe.
- **`renderPhaseBadge(phase, name, confidence)`** — Guards against undefined phase. Writes `Phase N · Name`, toggles `.low-conf` when confidence < 0.6.
- **`renderReadinessBadge(score, bucket)`** — Guards against non-numeric score. Writes `NN%` and sets `className = readiness-badge bucket-{bucket}`.
- **`renderDynamicEwbButtons(buttons)`** — Accepts array of strings OR `{label|typ|text}` objects, writes HTML to both `#ewbBar` and `#kp-ewbBar`. No-op on empty/invalid input. Uses `escHtml` + existing `triggerEwb()` click handler for drop-in compatibility with legacy behaviour.

Polling tick (`setInterval` fetching `/api/ergebnis`) extended **after** the existing `data.ewb_top2` legacy handler:

```js
renderActiveHint(data.active_hint);
renderPhaseBadge(data.current_phase, data.current_phase_name, data.phase_confidence);
renderReadinessBadge(data.readiness_score, data.readiness_bucket);
if (data.ewb_buttons && Array.isArray(data.ewb_buttons) && data.ewb_buttons.length) {
  renderDynamicEwbButtons(data.ewb_buttons);
}
```

Legacy `renderPipEwbButtons(data.ewb_top2)` path left intact as fallback. `zeigeKarte` / `#ai` card-append flow unchanged.

## Deviations from Plan

**1. [Rule 3 - Blocking issue] Existing `renderEwbButtons()` takes no arguments**
- **Found during:** Task 2 `renderDynamicEwbButtons` implementation.
- **Issue:** Plan example said `renderDynamicEwbButtons` should "reuse existing renderEwbButtons signature — it accepts a list and writes to both bars." Source inspection showed `renderEwbButtons()` at app.js:155 takes zero arguments and reads `window._profileEinwaende` directly. Calling it with a list would have silently discarded the argument.
- **Fix:** Inlined the button HTML generation inside `renderDynamicEwbButtons`, matching the exact same DOM output pattern (same `.ewb-btn`, same `triggerEwb()` onclick, same `escHtml`). No refactor of the legacy no-arg fn — it remains the fallback renderer for profile einwaende.
- **Files modified:** static/app.js
- **Commit:** 367fc8a

**2. [Rule 2 - Missing critical functionality] Defensive guards on undefined fields**
- **Found during:** Task 2 review.
- **Issue:** Plan snippet blindly wrote `el.textContent = ...` even if `phase` or `score` were undefined (e.g. during the first poll cycle before the server populated state). Would have rendered "Phase undefined · undefined" and "undefined%".
- **Fix:** Added early returns: `renderPhaseBadge` skips render if `phase == null`; `renderReadinessBadge` skips if `score` is not a number. Legacy clients hitting a pre-04.8 server endpoint render unchanged.
- **Files modified:** static/app.js
- **Commit:** 367fc8a

**3. [Rule 2 - Robustness] Flexible `buttons` input shape**
- **Found during:** Task 2.
- **Issue:** Plan assumed `data.ewb_buttons` is a list of strings, but server-side `dynamic_ewb_buttons` may return richer objects in the future (P04 summary mentioned "up to 6 buttons" without locking shape).
- **Fix:** `renderDynamicEwbButtons` accepts strings OR objects with `label|typ|text` fields and normalises to labels. Empty/invalid → no-op (legacy renderer stays in effect).
- **Files modified:** static/app.js
- **Commit:** 367fc8a

## STRIDE Mitigations Implemented

- **T-04.8.5-01 (Tampering/XSS — `renderActiveHint` innerHTML):** All user-derived text goes through existing `escHtml()` helper before template insertion. `hint.type`/`hint.color` are whitelisted via CSS class lookup (`hint-${hint.type||'tipp'}`) — unknown types fall through to the neutral `hint-tipp` style with no code execution surface.
- **T-04.8.5-02 (Information Disclosure — phase badge):** Accepted per plan. Seller's own screen only.
- **T-04.8.5-03 (DoS — 500 ms polling + DOM writes):** Accepted. Writes are constant-time (`innerHTML =` on ≤3 elements, no layout thrash beyond what the legacy tick already does). All renderers early-exit if target elements are missing.
