## Decisions (LOCKED)

### D-01: Phasen-Erkennung via zusätzlichem Haiku-Call im analyse_loop
- **Wie:** Neuer Haiku-Prompt klassifiziert die aktuelle Gesprächsphase (1-6) aus Transcript + vorheriger Phase + session_mode.
- **Input:** Letzte ~20 Transcript-Zeilen, aktuelle Phase, Zeit im Gespräch.
- **Output:** Phase-Nummer + Konfidenz.
- **Throttling:** analog zu EWB-Ranking, Default alle 2-3 analyse_loop-Zyklen (Planner kalibriert). Bei Phasen-Wechsel sofort erneut prüfen.
- **Persistiert in:** `state['current_phase']` (int 1-6), `state['phase_confidence']` (float), `state['phase_changed_at']` (timestamp).
- **Begründung:** Robust gegen Formulierungsvielfalt und Dialekte, deterministische Heuristiken wären zu brüchig. Throttling hält Kosten im Rahmen.

### D-02: Kaufbereitschafts-Score als deterministisches Python-Regelwerk
- **Wo:** `services/ki_logik.py::compute_readiness_score(state, transcript_window)`.
- **Input:** Transcript-Fenster + state (Redeanteil, Gesprächsdauer, erkannte Einwände, Score-History).
- **Rules:** Die exakten Faktoren +/- Werte und Schwellen (0-30/31-60/61-80/81-100) aus dem Briefing werden 1:1 als Python-Konstanten abgebildet.
- **Update-Cadence:** Jeder analyse_loop-Cycle (kein Extra-Throttling — reine Python-Arithmetik).
- **Persistiert in:** `state['readiness_score']` (0-100 int), `state['readiness_bucket']` ('cold'|'warm'|'hot'|'closing').
- **Begründung:** Briefing liefert vollständige Formel → Python ist erklärbar, testbar, kostenlos. Kein Haiku-Call nötig.

### D-03: Hinweis-Priorisierung server-seitig, Client rendert nur top
- **Wo:** `services/ki_logik.py::select_active_hint(candidates)` — nimmt alle aktuellen Hint-Kandidaten und wählt den mit höchster Priorität.
- **Prio-Ordnung (aus Briefing):** 1 Kritischer Fehler, 2 Kaufsignal, 3 Einwand, 4 Phasen-Hinweis, 5 Tipp.
- **State-Feld:** `state['active_hint']` = `{type, priority, text, color, source}` — ein Objekt, nicht Liste.
- **Client:** `/api/ergebnis` liefert `active_hint`. JS rendert den Einen, fertig. Keine Prio-Logik im JS.
- **Begründung:** Matcht bestehende Server-zentrale Architektur, hält Business-Logik in Python, Client bleibt dumm.

### D-04: Neues Modul `services/ki_logik.py`
- **Layout:**
  ```
  services/ki_logik.py
    - detect_phase(transcript_window, current_phase, elapsed_s, mode) -> (int, float)
    - compute_readiness_score(state, transcript_window) -> (int, bucket_str)
    - select_active_hint(candidates: list) -> dict | None
    - infer_cold_call_context(seller_transcript, current_phase) -> dict
    - dynamic_ewb_buttons(phase: int, base_buttons: list) -> list
  ```
- **Wer ruft auf:** `services/live_session.py::analyse_loop` orchestriert alle fünf Funktionen.
- **Was bleibt in claude_service.py:** Bestehende Haiku-Calls (`analysiere_mit_claude`, EWB-Ranking). Neue Haiku-Calls für phase detection und cold-call inference werden dort als Helper-Funktionen ergänzt (`classify_phase`, `infer_customer_state`), von ki_logik.py aufgerufen.
- **Begründung:** Saubere Trennung Business-Logik ↔ Prompt-Engineering. Ein Modul, nicht Package — Solo-Founder-freundlich.

### D-05: Cold-Call-Inferenz als eigener Haiku-Call (meeting mode: skip)
- **Trigger:** Nur wenn `session_mode == 'coldcall'`.
- **Wie:** Haiku-Prompt bekommt Vertriebler-Transcript + aktuelle Phase, gibt inferierten Kunden-State zurück (`{likely_customer_action, confidence, recommended_next}`). Inferenz-Regel-Tabelle aus Briefing als Few-Shot-Beispiele.
- **Throttling:** Gleicher Rhythmus wie Phasen-Erkennung.
- **Persistiert in:** `state['cold_call_inference']`.

### D-06: Dynamische EWB-Buttons pro Phase
- **Wie:** `dynamic_ewb_buttons(phase, base)` liefert phasen-spezifisches Button-Set aus Briefing (Phase 1 Opener → "Keine Zeit/Wer sind Sie?/..." etc.).
- **Fallback:** Bei unbekannter Phase → `base_buttons` aus Profil (heutiges Verhalten).
- **Integration:** Ersetzt/erweitert bestehenden `state['ewb_top2']` durch `state['ewb_buttons']` mit bis zu 6 Buttons. Client-Rendering muss angepasst werden (existierendes `renderEwbButtons` kann beide Bars füllen).
- **Wichtig:** EWB-Ranking-Call (Phase 04.2.1) kann entfallen ODER die 2 wahrscheinlichsten aus dem Phasen-Set picken. Researcher empfiehlt.

### D-07: Button-Klick → sofortiger Kontext-Snapshot
- **Wann Vertriebler Button drückt:** Server erhält `{button_label, phase, readiness_score, session_mode}` und generiert präzise Reaktions-Empfehlung via bestehenden `api_frage`-artigen Haiku-Call.
- **Bestehender Endpoint:** EWB-Trigger aus Phase 04.2 (`api_frage` Pattern). Nur Prompt erweitern um Phase + Score + Inferenz.
- **Persistiert in ObjectionEvent** (Phase 04.7) für Post-Call-Analyse.
