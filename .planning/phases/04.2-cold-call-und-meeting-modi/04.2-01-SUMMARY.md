Backend infrastructure for Cold Call and Meeting modes: mode-aware Deepgram connections (diarize toggle), EWB trigger endpoint with Claude Haiku, session_mode DB column with migration, and EWB logging in quick_action_log.

## What Was Built

### Task 1: DB migration + Deepgram mode-aware connections

**database/models.py** — Added `session_mode` column to `ConversationLog`:
```python
session_mode = Column(String(20), default='meeting')  # 'cold_call' or 'meeting'
```

**app.py** — Added `_migrate()` section for `conversation_logs.session_mode`:
```python
for col, typedef in [('session_mode', "VARCHAR(20) DEFAULT 'meeting'")]:
    conn.execute(text(f'ALTER TABLE conversation_logs ADD COLUMN {col} {typedef}'))
```

**services/deepgram_service.py** — Four changes:
1. Added `_session_modes = {}` dict alongside `_deepgram_sessions`
2. `_open_deepgram_connection(sid, mode='meeting')` — mode parameter added
3. `diarize=(mode == 'meeting')` — Cold Call gets `False`, Meeting gets `True`
4. `handle_start_live_session(data=None, sid=None)` — reads `data.get('mode', 'meeting')` from Socket.IO payload
5. `_close_deepgram_connection` cleans up `_session_modes` alongside `_deepgram_sessions`

### Task 2: EWB trigger endpoint + session_mode in api_beenden

**routes/app_routes.py** — Two changes:

1. `/api/ewb_trigger` POST endpoint:
   - Accepts `einwand_typ` from request body
   - Pulls last 3 transcript segments as context
   - Looks up matching `gegenargument` from active profile's `einwaende` list
   - Calls Claude Haiku (`claude-haiku-4-5-20251001`) with objection-specific prompt
   - Logs to `quick_action_log` with `typ='ewb'`
   - Returns `{'ok': True, 'antwort': ..., 'einwand_typ': ...}`

2. `api_beenden` session_mode persistence:
   - Reads `session_mode` from POST body (`req_data.get('session_mode', 'meeting')`)
   - Passes `session_mode=session_mode` to `ConversationLog(...)` constructor

## Decisions Made

- `session_mode` defaults to `'meeting'` everywhere for full backward compatibility — existing frontend calls without the field are treated as meeting mode
- EWB endpoint mirrors `api_frage` pattern (same Haiku model, same logging pattern) but with an objection-specific German prompt that ends with an open question
- `_session_modes` dict parallels `_deepgram_sessions` for potential future use (e.g., conditionally processing only speaker-0 audio in cold call mode at the application layer)
