### CR-01: `generate_scoring` crashes on malformed Claude JSON response

**Files modified:** `services/training_service.py`
**Commit:** c6e3f51
**Applied fix:** Added `if start == -1 or end <= start: raise ValueError(...)` guard before `json.loads` in `generate_scoring`, and wrapped the call in `try/except (json.JSONDecodeError, ValueError)`. Applied the same pattern to `_generate_live_preview` with a graceful fallback return of `{'momente': [], 'zusammenfassung': ''}` instead of raising.

### CR-02: `generated_personality` from frontend is silently ignored by backend

**Files modified:** `routes/training.py`
**Commit:** 263ff81
**Applied fix:** Two changes in `training_start`: (1) replaced the no-op `data.get('personality_type_id', type(None)) or data.get('personality_type_id')` with `data.get('personality_type_id') or None` and added `generated_personality = data.get('generated_personality')`; (2) added a new block after the `personality_type_id` loading section that populates `personality_data`, `startstimmung`, and `personality_hidden` from `generated_personality` when no `personality_type_id` is supplied. This also covers IN-03 (the no-op parse pattern).

### WR-02: TTS called unconditionally in `/training/respond` despite `voice_available` flag

### WR-03: No authorization check on custom personality deletion

**Files modified:** `routes/training.py`
**Commit:** 57955ba
**Applied fix:** Replaced `db.get(PersonalityType, personality_type_id)` with a `db.query(PersonalityType).filter(PersonalityType.id == personality_type_id, (PersonalityType.is_custom == False) | (PersonalityType.user_id == g.user.id)).first()` query that enforces ownership — system personalities (is_custom=False) are accessible by all, custom ones only by their creator.
