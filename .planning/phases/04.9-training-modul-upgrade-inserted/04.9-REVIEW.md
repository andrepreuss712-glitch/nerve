# Phase 04.9: Code Review Report

**Reviewed:** 2026-04-10
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 04.9 introduces personality-driven training with mood tracking, six seeded customer archetypes, scenario selection, and new analytics endpoints. The core feature is well-structured — the mood JSON fallback in `generate_response_with_mood` is a good pattern. However, two critical issues were found: an unguarded `json.loads` call that will crash on any malformed Claude response in `generate_scoring`, and a `generated_personality` payload sent from the frontend that the backend silently ignores — the user receives no error but the personality has no effect on the session. Five warnings cover race conditions on session dict access, TTS being called regardless of the `voice_available` flag in `/training/respond`, and missing access control on the personality deletion path.

## Critical Issues

### CR-01: `generate_scoring` crashes on malformed Claude JSON response

**File:** `services/training_service.py:819`
**Issue:** `json.loads(text[start:end])` is called without any exception handling. If Claude returns a non-JSON response, or the braces are mismatched, this raises `json.JSONDecodeError` which propagates uncaught through `training_end()`. The caller at `routes/training.py:447` has a `try/except` block but it only catches the error after the unhandled exception already crashes the scoring path — it returns the hardcoded fallback dict without the `punkte` key, which would then cause a `KeyError` at line 467 (`scoring.get('gesamt_score', 0)` is safe, but the upstream `result['punkte']` at line 824 is never set in the fallback). Additionally `_generate_live_preview` at line 867 has the exact same unguarded `json.loads`.

```python
# services/training_service.py:817 — fix generate_scoring
text   = response.content[0].text.strip()
start  = text.find('{')
end    = text.rfind('}') + 1
if start == -1 or end <= start:
    raise ValueError(f"No JSON braces in Claude response: {text[:100]}")
try:
    result = json.loads(text[start:end])
except (json.JSONDecodeError, ValueError) as e:
    raise ValueError(f"Scoring JSON parse failed: {e}") from e

# services/training_service.py:865 — fix _generate_live_preview
start = text.find('{')
end   = text.rfind('}') + 1
if start == -1 or end <= start:
    return {'momente': [], 'zusammenfassung': ''}
try:
    return json.loads(text[start:end])
except (json.JSONDecodeError, ValueError):
    return {'momente': [], 'zusammenfassung': ''}
```

### CR-02: `generated_personality` from frontend is silently ignored by backend

**File:** `routes/training.py:98` / `templates/training.html:840-842`
**Issue:** The frontend sends `generated_personality` in the start payload when the user clicks "Diesen Kunden nehmen" after generating a random personality (line 840-842 of training.html). The backend at `routes/training.py:98` reads `personality_type_id` from the request but never reads `data.get('generated_personality')`. The result is that a user who generates a custom personality, confirms it, and starts training actually gets a session with `personality_data = None` — the personality they selected is silently discarded and the standard prompt path is used instead. This is a silent feature failure, not a crash, but it ships a broken user-facing feature.

```python
# routes/training.py — after line 107, add:
generated_personality = data.get('generated_personality')

# Then after line 139 in the personality_type_id loading block:
if personality_type_id is None and generated_personality and isinstance(generated_personality, dict):
    # Use the unsaved generated personality directly in this session
    personality_data = generated_personality.get('attribute') or {}
    personality_data['name'] = generated_personality.get('name', 'Generiert')
    personality_data['icon'] = generated_personality.get('icon', '\U0001F464')
    personality_data['kurzbeschreibung'] = generated_personality.get('kurzbeschreibung', '')
    startstimmung = personality_data.get('startstimmung', 0)
    if schwierigkeit == 'schwer':
        personality_hidden = True
```

## Warnings

### WR-01: Session dict mutated outside the lock in `/training/respond`

**File:** `routes/training.py:291-305`
**Issue:** The lock is acquired to get a reference to `session`, then released. All subsequent mutations (`session['history'].append(...)`, `session['sekretaerin_ueberwunden'] = True`, `session['phase'] = 'kunde'`, etc.) happen on the dict reference without holding the lock. If two concurrent requests arrive (e.g., user double-submits), both can enter this code simultaneously and corrupt the history list or phase state. The session dict and list mutations at lines 301-407 are all unprotected.

```python
# Acquire the lock for the full mutation block, or use a per-user lock.
# Minimal fix: copy what you need under the lock, then do mutations
# under the same lock or use a per-session RLock.
with _sessions_lock:
    session = _sessions.get(g.user.id)
    if not session:
        return jsonify({'error': 'Keine aktive Session'}), 400
    # do history append here inside the lock
    session['history'].append({...})
```

### WR-02: TTS called unconditionally in `/training/respond` despite `voice_available` flag

**File:** `routes/training.py:371`
**Issue:** `text_to_speech(...)` is called on every `/training/respond` call regardless of `session.get('voice_available', True)`. The `/training/start` endpoint correctly gates TTS on `voice_available` (line 209), but `/training/respond` always calls TTS. This means users who exceeded the voice limit still consume ElevenLabs API credits on every subsequent message, and the cost tracker logs it against their quota.

```python
# routes/training.py:370-373 — add the guard
voice_id  = persona['voice_female']['id'] if is_sek else persona['voice_male']['id']
audio_b64 = None
if session.get('voice_available', True):
    audio_bytes = text_to_speech(kunde_antwort, voice_id, lang_config['elevenlabs_model'])
    if audio_bytes:
        audio_b64 = base64.b64encode(audio_bytes).decode('utf-8')
```

### WR-03: No authorization check on custom personality deletion

**File:** `routes/training.py:712-740` (`api_training_personality_save`)

The more pressing issue: `api_training_personalities` at line 641 returns custom types filtered by `user_id == g.user.id`, but a caller who knows a `personality_type_id` can pass it to `/training/start` and use any custom personality (even one belonging to a different user in the same org), because the load at line 130 uses `db.get(PersonalityType, personality_type_id)` with no `user_id` or `org_id` filter.

### WR-04: `stimmung_history` uses mutated `session` reference after lock release

**File:** `routes/training.py:339-347`
**Issue:** After the lock is released (line 291-292), the code rebuilds the mood and then re-acquires the lock to write `_sessions[g.user.id]['stimmung']` and `stimmung_history`. However, `'turn': len(session['history'])` on line 342 reads from the stale `session` reference (snapshot before lock). If a concurrent request added entries to `history` between the two lock acquisitions, the `turn` count will be wrong. This is a minor data-integrity issue in the analytics, not a crash.

**Fix:** Read `len(_sessions[g.user.id]['history'])` inside the second lock acquisition to get the current count.

### WR-05: `training_end` uses `db_log` without `try/finally` for entire block

**File:** `routes/training.py:512-575`
**Issue:** `db_log = get_session()` is created at line 512. The `try/finally` block at line 569-575 closes it, but if any exception occurs _before_ the `try` block is entered (unreachable in this case), `db_log` would leak. More practically: the `db_pts` session at line 582 is closed at line 596 inside a `try/except`, but if `db_pts.get(UserModel, ...)` raises before the `db_pts.close()` line, the session leaks. The project pattern requires manual close via try-finally (per CLAUDE.md), not bare try-except.

```python
# routes/training.py:582 — use try/finally pattern
try:
    db_pts = get_session()
    try:
        train_user = db_pts.get(UserModel, g.user.id)
        if train_user:
            # ... mutations ...
            db_pts.commit()
    finally:
        db_pts.close()
except Exception as ex:
    print(f"[Points] Training-Punkte Fehler: {ex}")
```

### IN-03: `personality_type_id` parsing uses confusing no-op pattern

**File:** `routes/training.py:98`
**Issue:** `data.get('personality_type_id', type(None)) or data.get('personality_type_id')` is a no-op expression — `type(None)` is the `NoneType` class (truthy), so the `or` branch is never taken. The effective result is identical to `data.get('personality_type_id')`. This indicates a misunderstanding; the intent was likely `data.get('personality_type_id') or None`.

```python
# routes/training.py:98 — simplify to:
personality_type_id = data.get('personality_type_id') or None
```
