# Phase 04.9 Plan 03: Backend API Wiring — Personality Endpoints + Mood-Tracked Training Flow Summary

3 new personality API endpoints (list/generate/save), scenario list fix to include system scenarios, training_start/respond/end fully wired for personality selection, mood tracking per turn, hangup signals, personality reveal at end, and ConversationLog persistence.

## What Was Built

### Task 1: Personality API endpoints (list, generate, save) + scenario list fix

**routes/training.py imports extended:**
- Added `PersonalityType`, `ConversationLog` to `from database.models import`
- Added `generate_response_with_mood`, `build_personality_prompt` to `from services.training_service import`

**GET `/api/training/personalities`:**
- Returns system types (is_custom=False) + caller's custom types (user_id=g.user.id)
- Scoped query prevents cross-user custom type visibility (T-04.9-08)
- Returns: id, name, icon, kurzbeschreibung, is_custom, startstimmung, kommentar

**POST `/api/training/personalities/generate`:**
- Calls `claude-haiku-4-5-20251001` with DACH-B2B persona generation prompt
- Returns unsaved personality JSON — user must call /save to persist
- JSON extracted via find('{') / rfind('}') pattern (consistent with codebase)

**POST `/api/training/personalities/save`:**
- Creates PersonalityType with `user_id=g.user.id`, `org_id=g.org.id`, `is_custom=True`
- Input truncated: name→100, icon→10, kurzbeschreibung→300 (T-04.9-06)
- user_id always from g.user.id session, never from request body (T-04.9-05)

**`training_scenarios_list()` query fixed:**
- Changed from `filter_by(org_id=g.org.id)` to OR filter including `erstellt_von == None`
- System scenarios now appear alongside org-scoped scenarios

### Task 2: training_start / training_respond / training_end for personality + mood

**`training_start()` updates:**
- Reads `personality_type_id` from request JSON (int-cast with fallback)
- Loads `PersonalityType` from DB; builds `personality_data` dict with name/icon/attributes
- D-05: sets `personality_hidden=True` when `schwierigkeit='schwer'` + `is_custom=True`
- Scenario lookup extended to include system scenarios (org_id OR erstellt_von IS NULL)
- When personality present: calls `build_personality_prompt()` instead of `build_customer_prompt()`
- Session dict gains: `personality_type_id`, `personality_data`, `stimmung`, `stimmung_history`, `personality_hidden`, `aufgelegt`, `scenario_id`, `szenario_data`
- Response: conditionally exposes personality_name/icon/stimmung; hides all when `personality_hidden=True` (T-04.9-07)

**`training_respond()` updates:**
- When `session.personality_data` present and not sekretaerin: calls `generate_response_with_mood()`
- Rebuilds system prompt with `current_mood` each turn via `build_personality_prompt()`
- Updates session `stimmung` + `stimmung_history` atomically via `_sessions_lock`
- Sets `session['aufgelegt']=True` if Claude signals hangup
- Response includes `aufgelegt`, `letzte_chance`, `schwierigkeit`
- D-07: `stimmung` returned in response only when `schwierigkeit='leicht'`; `None` otherwise

**`training_end()` updates:**
- Result includes `stimmung_history` from session
- D-05: `personality_revealed` dict (name/icon/kurzbeschreibung) added when `personality_hidden` was True
- After `db_log.flush()`: sets `log_entry.personality_type_id` and `log_entry.stimmung_history` (JSON)
