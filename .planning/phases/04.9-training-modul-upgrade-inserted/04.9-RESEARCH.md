## Summary

Phase 04.9 upgrades the existing training module from a simple difficulty-based simulation to a full personality-driven AI sparring partner. The existing code is well-structured and provides excellent building blocks — the main work is extending the prompt engineering, adding a new DB table, refactoring the setup flow UI, and wiring mood tracking through the respond/end endpoints.

The core technical challenge is Claude returning **structured JSON** (mood + text) instead of plain text on every `/training/respond` call. This requires updating `generate_response()` in `training_service.py` to return a dict, updating `/training/respond` to parse it and store mood history in session state, and conditionally surfacing the mood indicator in the chat UI based on difficulty level.

The scoring upgrade (6th category + mood curve) is a contained change to `generate_scoring()` and the scoring render function in `training.html`. Chart.js is already loaded on the training page via CDN.

**Primary recommendation:** Implement in waves — (1) DB + personality types data seeding, (2) Setup-flow UI (type cards + scenario step), (3) `generate_response()` → JSON + mood tracking, (4) Auflege-Logik + UX, (5) Scoring upgrade with mood curve.

## Common Pitfalls

### Pitfall 1: generate_response() JSON Parse Failure
**What goes wrong:** Claude occasionally returns text + JSON instead of JSON only. If the `{` is not found or JSON is malformed, the `json.loads()` call throws and the entire respond endpoint crashes.
**Why it happens:** Claude doesn't always follow JSON-only instructions perfectly, especially Haiku.
**How to avoid:** Wrap in try-except. On parse failure, fall back to treating response as plain text with `neue_stimmung = current_mood` (no change). Log with `[Training]` prefix.
**Warning signs:** 500 errors on `/training/respond` after first few turns.

### Pitfall 2: Mood State Lost on Page Refresh
**What goes wrong:** In-memory `_sessions` is wiped on server restart or user refresh. If user refreshes mid-training, stimmung_history is lost.
**Why it happens:** Sessions are not persisted to DB until `/training/end`.
**How to avoid:** This is the existing behavior for all session state — not a regression. Document as known limitation. Users cannot resume mid-training (same as now).

### Pitfall 3: org_id Filtering Excludes System Scenarios
**What goes wrong:** `filter_by(org_id=g.org.id)` returns zero system scenarios (org_id=NULL) for every org.
**Why it happens:** System scenarios use NULL org_id, filter_by with a specific org_id never matches NULL.
**How to avoid:** Use SQLAlchemy `or_` filter: `(org_id == g.org.id) | (org_id == None)`.

### Pitfall 4: Experte Hidden Type Leaks in API Response
**What goes wrong:** `/training/start` returns `personality_data` in the JSON response. If the frontend logs it or devtools shows it, the type is revealed before scoring.
**Why it happens:** Easy to accidentally include full personality dict in response.
**How to avoid:** When `personality_hidden=True` (Experte + random type), return only `{'typ_verborgen': true}` to frontend. Full personality_data stays in server-side `_sessions` dict only.

### Pitfall 5: Chart.js Canvas Already In Use
**What goes wrong:** If the scoring section is re-rendered (user trains multiple times without page refresh), `new Chart()` on the same `<canvas id="t-mood-chart">` throws "Canvas is already in use".
**Why it happens:** Chart.js holds a reference to the canvas element. Re-creating Chart on the same canvas errors.
**How to avoid:** Before `new Chart()`, call `Chart.getChart('t-mood-chart')?.destroy()` first. Same pattern used for trend chart in training analytics section.

### Pitfall 6: Stimmungs-Delta Not Matching Difficulty
**What goes wrong:** Experte mode gets warnings (letzte_chance) when it should not. Einsteiger gets no mood indicator.
**Why it happens:** Difficulty-based behavior is encoded in the Claude system prompt AND in the frontend rendering logic — both must agree.
**How to avoid:** Single source of truth: the `schwierigkeit` value in the session dict controls the system prompt rules AND the frontend JS constants. Pass `schwierigkeit` back in every `/training/respond` response so frontend always has current difficulty context.

### Pitfall 7: SCHWIERIGKEITEN Dict Key Mapping
**What goes wrong:** Existing code uses 'leicht'/'mittel'/'schwer'/'sekretaerin' as internal keys. New UI shows 'Einsteiger'/'Fortgeschritten'/'Experte'. If mapping isn't applied consistently, the wrong difficulty config is used.
**Why it happens:** D-04 adds a label rename without changing internal keys.
**How to avoid:** Keep existing internal keys ('leicht'/'mittel'/'schwer'). Only change the `label` values in `SCHWIERIGKEITEN` dict and in the training.html diff card display. All backend logic continues to use 'leicht'/'mittel'/'schwer'.

## Code Examples

### Pattern: Claude JSON Response with Mood

```python
# Source: extended from existing generate_response() in training_service.py
def generate_response_with_mood(
    conversation_history: list,
    system_prompt: str,
    current_mood: int,
    schwierigkeit: str = 'mittel',
) -> dict:
    """Returns dict with text, neue_stimmung, aufgelegt, letzte_chance."""
    messages = []
    for msg in conversation_history:
        role = "assistant" if msg['speaker'] == 'kunde' else "user"
        messages.append({"role": role, "content": msg['text']})
    if not messages:
        messages = [{"role": "user", "content": "(Telefon klingelt. Geh ran.)"}]

    response = claude_client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=300,
        system=system_prompt,
        messages=messages
    )
    text = response.content[0].text.strip()
    start = text.find('{')
    end   = text.rfind('}') + 1
    if start == -1:
        # Fallback: no JSON found, treat as plain text response
        return {'text': text, 'neue_stimmung': current_mood,
                'aufgelegt': False, 'letzte_chance': False}
    try:
        return json.loads(text[start:end])
    except Exception:
        return {'text': text, 'neue_stimmung': current_mood,
                'aufgelegt': False, 'letzte_chance': False}
```

1. **Persönlichkeitstyp bei Szenario-Auswahl: Welche Kombination ist sinnvoll?**

   - What's unclear: Should the UI show a warning if an incompatible combo is chosen (e.g., Freundlicher Ja-Sager + Aggressiver Scenario)? Or purely free choice?
   - Recommendation: No validation — free choice per D-06 and D-13. User learns from any combo.

2. **Zufällig generieren: Wird die generierte Persönlichkeit vor dem Training gezeigt?**
   - What we know: D-02 says a "Briefing" is shown ("Dein Kunde: IT-Leiter, 52..."). D-05 says in Experte mode with random type, personality is hidden.
   - What's unclear: Does "Zufällig generieren" in Einsteiger/Fortgeschritten show the briefing? Does it hide in Experte?
   - Recommendation: Show briefing in Einsteiger/Fortgeschritten. In Experte: selecting "Zufällig generieren" auto-triggers hidden mode per D-05. Briefing shown after scoring.

3. **API Kosten für "Zufällig generieren":**
   - What we know: Haiku is the model for all live training operations (CLAUDE.md constraint).
   - What's unclear: The generation prompt needs enough context to produce a realistic personality. 1 Haiku call per generation (~500 tokens) at ~$0.00025 per call is negligible.
   - Recommendation: Use Haiku, one call, ~300 token output. Fine.

4. **Wie wird Szenario-org_id gehandelt für System-Szenarien?**
   - What we know: Current `training_scenarios_list()` filters by `org_id=g.org.id`. System scenarios have `erstellt_von=None` but ALSO have `org_id=g.org.id` (they are created per-org by the seed function).
   - What's unclear: Are system scenarios seeded once globally (org_id=NULL) or seeded per org?
   - Recommendation: Seed with `org_id=NULL` and fix the query to use `or_` filter. This avoids N duplicate rows for N orgs.
