### Pattern 1: Hook-Writes (nicht-blockierend)

**Quelle:** `services/claude_service.py:97` — `_write_ft_assistant_event` ist der Referenz-Pattern aus Phase 04.7.1.

```python
# NEU in services/cost_tracker.py
def log_api_cost(provider: str, model: str, user_id: int | None,
                 units: float, unit_type: str) -> None:
    """
    Schreibt einen api_cost_log Entry. MUSS NIEMALS raisen —
    API-Calls dürfen nie wegen Logging-Fehler crashen.
    Wechselkurs und ApiRate werden beim Schreiben eingefroren.
    """
    try:
        from database.db import SessionLocal
        from database.models import ApiCostLog, ApiRate, ExchangeRate
        db = SessionLocal()
        try:
            rate = (db.query(ApiRate)
                      .filter_by(provider=provider, model=model, unit_type=unit_type, active=True)
                      .first())
            if not rate:
                return  # kein Preis → kein Log (Warning im Dashboard via last_checked_at)
            # Wechselkurs freeze
            if rate.currency == 'USD':
                fx = db.query(ExchangeRate).filter_by(currency_pair='USD_EUR').order_by(ExchangeRate.date.desc()).first()
                fx_rate = fx.rate if fx else 0.92  # Fallback
            else:
                fx_rate = 1.0
            cost_eur = float(rate.price_per_unit) * units * fx_rate
            db.add(ApiCostLog(
                provider=provider, model=model, user_id=user_id,
                units=units, unit_type=unit_type,
                rate_applied=rate.price_per_unit, fx_rate_applied=fx_rate,
                cost_eur=round(cost_eur, 6),
            ))
            db.commit()
        finally:
            db.close()
    except Exception as e:
        print(f"[CostTracker] log failed ({provider}/{model}): {e}")
```

- `services/claude_service.py:449, 476, 492` — nach jedem `claude_client.messages.create(...)` die `msg.usage.input_tokens` + `msg.usage.output_tokens` lesen und zwei `log_api_cost`-Calls schreiben (input + output). `[VERIFIED: anthropic SDK response object hat .usage mit input_tokens/output_tokens]`

```python
def superadmin_required(f):
    @wraps(f)
    def wrapper(*args, **kwargs):
        if not getattr(g, 'user', None) or not getattr(g.user, 'is_superadmin', False):
```

### Pattern 4: DB-Migration

`app.py::_migrate()` (Zeile 47) ist der Pattern. Kein Alembic, nur `CREATE TABLE IF NOT EXISTS` via `Base.metadata.create_all(engine)` am Startup + gezielte `ALTER TABLE` mit Try/Except für neue Spalten auf bestehenden Tabellen.

Für Phase 04.7.2:
- `Base.metadata.create_all(engine)` erzeugt die 6 neuen Tabellen automatisch beim Start (sobald Models in `database/models.py` registriert sind).
- `_migrate()` muss nur ergänzt werden für: Seed der FixedCosts (D-10 Defaults) + Seed der initialen ApiRates (Briefing S. nennt Preise).
- KEIN ALTER auf bestehenden Tabellen nötig — alle neuen Tabellen sind standalone.

### Anti-Patterns

- **NICHT** per Alembic migrieren — Projekt verwendet kein Alembic.
- **NICHT** API-Calls synchron auf DB warten lassen — Hook-Writes müssen try/except-ummantelt sein.
- **NICHT** Wechselkurse beim Lesen anwenden — beim Schreiben freezen (D-02, steuerlich korrekt).
- **NICHT** `rolle='owner'` statt `is_superadmin` verwenden — ein zahlender Owner seiner eigenen Org darf Andrés Privatzahlen nie sehen (D-08).
- **NICHT** Tax-Behandlung (DE_19/EU_RC/DRITTLAND) aus Land raten — Stripe `automatic_tax` und `tax_amounts` sind die Source of Truth. Eigene Logik nur als Fallback-Classifier.
- **NICHT** `invoice.paid` für Revenue-Log verwenden — das Event existiert schon für Fair-Use-Reset. `invoice.payment_succeeded` ist der kanonische Revenue-Trigger.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| PDF-Generierung | Eigener ReportLab-Code | WeasyPrint | Deutsche Zahlenformate, Tabellen, Page-Breaks — alles gratis via CSS. |
| USt-Berechnung pro Invoice | Land→Satz-Tabelle selbst pflegen | Stripe `automatic_tax` + `invoice.lines.tax_amounts` | Stripe pflegt Tax-Regeln pro Land/Produkt, inkl. Reverse-Charge EU B2B. |
| Wechselkurs-Abfrage | EZB-XML-Parser | Frankfurter API (JSON) | EZB-Daten, kein Key, kein Quota, 15-Min-TTL-Cache empfohlen. |
| CSV-Export | String-concat | `csv.writer` stdlib | Quoting, Escaping, UTF-8-BOM für Excel-Kompatibilität. |
| Deutsche Zahlenformatierung | `str.replace('.',',')` | Jinja-Filter + `Babel.numbers.format_currency` ODER eigener Filter mit `{:,.2f}`.replace — hier reicht eigener Filter, keine neue Dependency. | Projekt hat kein Babel. Eigener Filter ~6 Zeilen. |
| Scheduler | `while True: sleep(86400)` in Thread | APScheduler (BackgroundScheduler) oder systemd-timer | Safe shutdown, jitter, persistence. |
| Tab-State | React/Vue | Vanilla JS + `window.location.hash` | D-09 lock, Stack-Constraint. |

## Common Pitfalls

### Pitfall 1: Stripe automatic_tax ohne Tax Registrations
**What goes wrong:** `invoice.lines.tax_amounts` bleibt leer, alle `ust_cents=0`, Revenue-Dashboard zeigt falsche Zahlen.
**Why it happens:** `automatic_tax: true` im Code aktiviert die Berechnung, aber Stripe braucht zusätzlich eine aktive Tax Registration im Dashboard für DE (und später andere Länder). Sonst ignoriert Stripe.
**How to avoid:** Human-Task im Plan: "Stripe-Dashboard → Tax → Register for Germany". Vor dem Deploy der Webhook-Erweiterung verifizieren.
**Warning signs:** Test-Invoice hat `total == subtotal` und `tax_amounts: []`.
`[CITED: Stripe docs — Tax requires registrations before amounts populate]`

### Pitfall 2: Wechselkurs nachträglich geändert
**What goes wrong:** Historische api_cost_log-Einträge bekommen rückwirkend neue EUR-Werte, steuerlich falsch.
**Why it happens:** Naive Implementierung holt `get_current_rate()` beim Dashboard-Rendering statt beim Schreiben.
**How to avoid:** `api_cost_log` hat Spalten `rate_applied` UND `fx_rate_applied` — beides beim Schreiben gefroren. Lese-Aggregationen verwenden `cost_eur` aus der Tabelle, nicht neu berechnen.
**Warning signs:** Dashboard-Summen ändern sich nach Kurs-Update.

### Pitfall 3: Reverse-Charge §13b falsch gemeldet
**What goes wrong:** USt-VA ist fehlerhaft, Finanzamt meldet sich.
**Why it happens:** Für US-Drittland-API-Anbieter (Anthropic, Deepgram, ElevenLabs) muss ich als Leistungsempfänger die USt selbst berechnen (KZ 84 = Bemessungsgrundlage, KZ 85 = daraus 19% USt) UND gleichzeitig als Vorsteuer abziehen (KZ 67). Effekt: netto 0, aber Meldepflicht.
**How to avoid:** `eur_calculator.py` muss diesen Split explizit berechnen: `rc_bemessung = sum(api_cost_log.cost_eur for us_providers)` → `rc_ust = rc_bemessung * 0.19` → beides in KZ 84/85 UND KZ 67. Unit-Test mit Fixture.
**Warning signs:** KZ 67 leer, KZ 85 leer, aber api_cost_log hat Einträge.
`[ASSUMED: §13b UStG für US-Drittland-SaaS = sonstige Leistung B2B am Empfängerort → Reverse Charge. Briefing S.192 bestätigt dies im Beispiel. Sollte durch count.tax final verifiziert werden bevor es in USt-VA geht.]`

### Pitfall 4: Chart.js Annotation-Plugin nicht geladen
**What goes wrong:** Preisänderungs-Marker im Chart erscheinen nicht.
**Why it happens:** Chart.js 4.4.0 core kommt ohne annotations. Plugin muss separat geladen werden.
**How to avoid:** Entweder (a) zusätzliches CDN-Script `chartjs-plugin-annotation@3` laden, ODER (b) einfachere Lösung: Preisänderungen als separates Dataset (vertikale Linien) mit `type: 'line'`, `pointRadius: 0`, `borderColor: 'red'`. Option (b) vermeidet neue Dependency.

### Pitfall 5: WeasyPrint auf Windows-Dev-Maschine
**What goes wrong:** `pip install weasyprint` wirft bei Import OSError für cairo/pango DLLs.
**Why it happens:** WeasyPrint braucht native Pango/Cairo/GDK-PixBuf. MSYS2 ist der offizielle Weg; GTK3-Installer ist veraltet und bricht.
**How to avoid:** Dokumentiere im Plan: Dev-Install via `pacman -S mingw-w64-x86_64-pango` in MSYS2. Fallback: PDF nur auf VPS generieren, lokaler Test via HTML-Preview-Endpoint `/admin/dashboard/eur.html?period=2026-03` (selbes Jinja-Template, aber ohne PDF-Render).
`[CITED: github.com/Kozea/WeasyPrint issue #2105]`

### Pitfall 6: Per-User API-Attribution im Live-Modus
**What goes wrong:** `api_cost_log.user_id` ist NULL für Live-Analyse-Calls → Kunden-Profitabilität (Tab 4) zeigt alle Kosten als "unattributed".
**Why it happens:** `analyse_loop()` läuft im Background-Thread ohne Flask `g.user`. State muss aus `live_session.state` gelesen werden (genau wie `_write_ft_assistant_event` es macht: `ls.state.get('user_id')`).
**How to avoid:** `cost_tracker.log_api_cost()` ruft intern `live_session.state.get('user_id')` falls nichts übergeben. Für Training-TTS: `g.user.id` verfügbar (HTTP-Context). Für EWB-Ranking (auch Background): live_session.state.

### Pitfall 7: MRR-Berechnung vs. Revenue-Einnahmen
**What goes wrong:** Übersicht-Tab KPI-Karte "MRR" zeigt März-Einnahmen statt MRR (= Summe aktiver monatlicher Subscriptions).
**Why it happens:** MRR = `sum(org.plan_price for org where subscription_status='active')`, NICHT `sum(revenue_log.netto where month=current)`. Das sind zwei verschiedene Metriken.
**How to avoid:** Explizit zwei KPIs trennen: "MRR" (forward-looking aus Organisation-Tabelle) vs. "Einnahmen diesen Monat" (backward-looking aus revenue_log).

### Frankfurter-Client mit Cache + Fallback
```python
# services/exchange_rates.py
import requests
from datetime import date, timedelta
from database.db import SessionLocal
from database.models import ExchangeRate

FRANKFURTER_URL = "https://api.frankfurter.dev/v1/latest"

def fetch_usd_eur() -> float | None:
    try:
        r = requests.get(FRANKFURTER_URL, params={'base': 'USD', 'symbols': 'EUR'}, timeout=5)
        r.raise_for_status()
        return float(r.json()['rates']['EUR'])
    except Exception as e:
        print(f"[FX] Frankfurter fetch failed: {e}")
        return None

def update_daily_rate():
    """Cron-Entrypoint. Schreibt Tagesrate. Bei API-Down: skip (letzter Kurs bleibt aktuell)."""
    rate = fetch_usd_eur()
    if rate is None:
        return
    db = SessionLocal()
    try:
        existing = db.query(ExchangeRate).filter_by(date=date.today(), currency_pair='USD_EUR').first()
        if existing:
            return
        db.add(ExchangeRate(date=date.today(), currency_pair='USD_EUR', rate=rate, source='frankfurter'))
        db.commit()
    finally:
        db.close()

def get_current_rate(currency_pair: str = 'USD_EUR') -> float:
    """Holt den neuesten Kurs aus DB. Fallback: 0.92 (USD→EUR approximation)."""
    db = SessionLocal()
    try:
        row = db.query(ExchangeRate).filter_by(currency_pair=currency_pair).order_by(ExchangeRate.date.desc()).first()
        return float(row.rate) if row else 0.92
    finally:
        db.close()
```
*Source: Frankfurter API docs — https://frankfurter.dev/v1/*

### Jinja de_currency Filter
```python
# app.py (bei anderen Filtern)
def de_currency(value):
    if value is None:
        return '—'
    try:
        n = float(value)
    except (TypeError, ValueError):
        return str(value)
    formatted = f"{n:,.2f}"
    # US "1,234.56" → DE "1.234,56"
    return formatted.replace(',', '§').replace('.', ',').replace('§', '.') + '€'

app.jinja_env.filters['de_currency'] = de_currency
```
