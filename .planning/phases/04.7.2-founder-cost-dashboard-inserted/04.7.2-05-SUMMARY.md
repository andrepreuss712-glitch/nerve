# Phase 04.7.2 Plan 05: Ausgaben-Tab + Kunden-Tab Summary

Tab Ausgaben (EÜR-Kategorien + Anbieter-Split + 30-Tage-Chart + API-Rate-CRUD + Fixed-Cost-CRUD) und Tab Kunden (Org-Profitability mit Status-Ampel + User-Drilldown mit 4-Wochen-Trend) voll implementiert. classify_margin und compute_org_profitability via 5 Unit-Tests abgesichert. Gesamt 10/10 Tests grün (5 neue profitability + 5 unveränderte auth).

## What Was Built

### routes/admin_dashboard.py (+~360 Zeilen)

**Tab Ausgaben Endpoints:**

| Route | Method | Returns |
|---|---|---|
| `/admin/dashboard/ausgaben` | GET | JSON: by_provider[], fixed_costs[], daily_30[], api_rates[], api_total, fixed_total, grand_total |
| `/admin/dashboard/api_rates/<id>/mark_checked` | POST | `{ok, last_checked}` |
| `/admin/dashboard/api_rates/<id>/new_price` | POST | `{ok, impact_eur_per_month}` — deaktiviert alten Rate, legt neuen an, schreibt PriceChangeLog |
| `/admin/dashboard/fixed_costs` | POST | `{ok, id}` — create |
| `/admin/dashboard/fixed_costs/<id>` | POST | `{ok, ...}` — `_action` ∈ {update, delete, toggle} |

**Helpers:**
- `EUR_CATEGORY_BY_PROVIDER` dict (anthropic/deepgram/elevenlabs → Fremdleistungen Z.26, stripe → Geldverkehr Z.57)
- `classify_margin(pct) -> str` — `>70 healthy`, `50..70 warn`, `<50 critical`
- `compute_org_profitability(db, org_id, start, end) -> dict` — `{revenue_eur, api_cost_eur, margin_pct, status}`, div-by-zero-safe

**Tab Kunden Endpoints:**

| Route | Method | Returns |
|---|---|---|
| `/admin/dashboard/kunden` | GET | JSON: orgs[] (sortiert critical first) + summary{healthy, warn, critical} |
| `/admin/dashboard/kunden/<org_id>/users` | GET | JSON: users[] mit total_cost_eur, by_provider, weekly_trend (28 Tage / 4 Wochen) |

Alle 9 Endpoints: `@login_required + @superadmin_required`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `Organisation.plan_key` existiert nicht**
- **Found during:** Task 3 Test-Schreiben
- **Issue:** Plan-Test-Code nutzte `Organisation(..., plan_key='solo', ...)`. Column existiert nur auf RevenueLog. Das gleiche Problem war bereits in Plan 04 bei `test_admin_dashboard_auth.py` gefixt.
- **Fix:** Test-Helper `_mk_org()` nutzt `plan='solo'`. `kunden_page` JSON-Output nutzt `getattr(org, 'plan', '')` statt `plan_key`.
- **Files modified:** `routes/admin_dashboard.py`, `tests/test_profitability.py`
- **Commit:** 4e16688, 4c0c0cd

**2. [Rule 2 - Correctness] XSS-Schutz via esc() im innerHTML**
- **Found during:** Task 2+3 Template
- **Issue:** Plan-Templates nutzten direktes `${item.label}` in Template-Literals die in `innerHTML` landen. Threat T-04.7.2-05-04 fordert escapeHtml. Org-Name und User-Email stammen aus kontrollierten Quellen, aber der explizite Schutz ist ein Correctness-Requirement aus dem Threat-Model.
- **Fix:** `esc()` helper in beiden Templates eingeführt und auf alle dynamischen Strings angewendet.
- **Files modified:** `templates/admin/_tab_ausgaben.html`, `templates/admin/_tab_kunden.html`
- **Commits:** 0df35db, 4c0c0cd

**3. [Rule 2 - Correctness] `get_current_rate` try/except im api_rate_new_price**
- **Found during:** Task 1 Implementation
- **Issue:** Plan importiert `from services.exchange_rates import get_current_rate` und ruft direkt auf. Wenn Frankfurter-Scheduler nicht geladen oder Service fehlt, wirft der Handler 500 statt den Preis trotzdem zu speichern.
- **Fix:** try/except um import+call, Fallback auf 0.92 (USD) bzw. 1.0 (EUR). Impact-Berechnung ist eh eine Heuristik.
- **Files modified:** `routes/admin_dashboard.py`
- **Commit:** 4e16688

**4. [Rule 2 - Correctness] FixedCost cycle-Whitelist vor `fc =` Constructor**
- **Found during:** Task 1 Implementation
- **Issue:** Plan-Snippet prüft `fc.cycle not in (...)` NACH Construct. `abort(400)` wirkt, aber die Validierung sollte vor DB-Touch passieren.
- **Fix:** cycle-Whitelist vor Constructor-Call gezogen, Decimal-Parsing in try/except gewrappt.
- **Files modified:** `routes/admin_dashboard.py`
- **Commit:** 4e16688
