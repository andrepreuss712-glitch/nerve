<decisions>
## Implementation Decisions

### Scope (D-01)
- **M2a + M2b komplett in dieser Phase.**
- M2a: Tabs Übersicht/Einnahmen/Ausgaben/Kunden + DB-Modelle (ApiCostLog, ApiRate, FixedCost, RevenueLog, ExchangeRate, PriceChangeLog) + Wechselkurs.
- M2b: Tab EÜR (vollständige Berechnung mit USt-Voranmeldung), Tab Export (PDF + CSVs + DATEV-Stub).
- **Defer auf 04.7.3 oder später (Milestone 3 laut Briefing):** automatische monatliche Datenlieferung an Steuerberater, Belege-Archiv (GoBD), automatischer API-Rechnungs-Download, ZUGFeRD, DATEV Unternehmen Online.

### API-Cost-Tracking (D-02)
- **Per-Call Logging in Hooks.** Jeder Claude/Deepgram/ElevenLabs-Call schreibt nach `api_cost_log` mit `cost_eur` (berechnet aus aktueller `ApiRate` × Tokens/Minuten/Zeichen × aktueller `ExchangeRate`).
- Hook-Stellen:
  - `services/claude_service.py` — alle Anthropic-Calls (Haiku live, Sonnet post-call, EWB-Ranking)
  - `services/deepgram_service.py` — STT-Minuten beim Session-Ende
  - `services/elevenlabs_service.py` (oder wo TTS lebt) — Zeichen pro Synthese
- Wechselkurs wird zum Zeitpunkt des Eintrags eingefroren — nachträgliche Kursänderungen verändern keine bestehenden Buchungen (steuerlich korrekt).

### Stripe Revenue Tracking (D-03)
- **Stripe Tax + `invoice.payment_succeeded` Webhook.**
- Bestehender Webhook in `routes/payments.py:64` wird erweitert: liest `invoice.lines.data[].tax_amounts` für echte USt-Splits, `customer.address.country` für Land, schreibt `RevenueLog` mit `netto`/`ust`/`brutto`/`country`/`tax_treatment` (DE_19 / EU_RC / DRITTLAND).
- `automatic_tax: { enabled: true }` muss bei Checkout-Session-Erstellung aktiviert werden.

### PDF-Engine (D-04)
- **WeasyPrint** für EÜR-PDF-Export.
- HTML-Template in `templates/admin/eur_pdf.html` mit deutschem Zahlenformat (`{{ value | de_currency }}` Filter).
- DIN A4, deutsche Locale.

### Wechselkurs-Quelle (D-05)
- **Frankfurter API** (`https://api.frankfurter.app/latest?from=USD&to=EUR`).
- Kostenlos, kein API-Key, EZB-Daten.
- Täglicher Cron schreibt `exchange_rates` (date, currency_pair, rate, source='frankfurter').
- Fallback bei API-Down: letzten bekannten Kurs verwenden + Warning loggen.

### Preismonitor (D-06)
- **Manuell + Reminder.** Kein Cron, kein Claude-Web-Fetch in dieser Phase.
- Spalte `last_checked_at` auf `api_rates`. UI in Tab Ausgaben zeigt Warning wenn `now() - last_checked_at > 30 Tage`.
- "Jetzt geprüft" Button setzt `last_checked_at = now()`.
- Manueller "+ Neuen Preis eintragen" Flow für Preisänderungen → schreibt `price_change_log` Entry mit `old_rate`/`new_rate`/`impact_eur_per_month`.

### DATEV-Export (D-07)
- **Stub jetzt, voll später.** Tab Export hat Button "DATEV-Buchungsstapel (CSV)" der eine simple CSV mit `Datum,Konto,Gegenkonto,Betrag,Buchungstext` generiert — nicht DATEV-Format-konform.
- Voll-Implementierung (Pflichtfelder: BU-Schlüssel, Belegnummer, S/H, etc.) wartet bis count.tax explizit das Format anfragt.

### Admin-Gate (D-08)
- **`@superadmin_required`** aus `services/auth_decorators.py:5` (eingeführt in Phase 04.7).
- Gate basiert auf `User.is_superadmin` Bool-Column (`database/models.py:62`), NICHT auf `rolle`.
- **Begründung:** Das Founder Cost Dashboard zeigt Andrés private Einnahmen, Ausgaben, API-Kosten und Kunden-Profitabilität. Kein zahlender Kunde — auch keiner mit `rolle='owner'` seiner eigenen Org — darf das jemals sehen. `is_superadmin` ist exklusiv für André reserviert.
- Neuer Blueprint `routes/admin_dashboard.py` — jede Route mit `@superadmin_required` dekoriert.
- Pattern-Referenz: `routes/admin_views.py` (Flask-Admin Setup aus Phase 04.7) verwendet denselben Mechanismus.

### Tabs-Struktur (D-09)
- 6 Tabs unter `/admin/dashboard` Route.
- Tab-Switching client-side (kein Page-Reload) — Vanilla-JS, kein Framework (passt zu bestehendem Stack).
- Tab-State in URL-Hash (`#einnahmen`) für Deep-Links.

### Fixe Betriebskosten (D-10)
- `fixed_costs` Tabelle mit Briefing-Defaults seeden (Hetzner 4€, Domain 1.25€, Kontist 4.95€, count.tax 150€, Homeoffice 6€/Tag).
- CRUD-UI im Ausgaben-Tab (unten).
- Homeoffice ist Sonderfall: `cycle='per_day'`, separater "Tage diesen Monat" Input → multipliziert.

### Charts (D-11)
- **Chart.js** wiederverwenden — bereits in `templates/dashboard.html` eingebunden (Phase 04.6).
- Charts: MRR+Kosten 12-Monate Combo, Marge-Trend 12-Monate Line, Tageskosten 30-Tage Bar.
- Rote Markierungen bei Preisänderungen via `annotation`-Plugin oder manuelle Datasets.

### EÜR-Berechnung (D-12)
- Berechnung serverseitig in `services/eur_calculator.py`.
- Input: Zeitraum (Monat/Quartal/Jahr) + bestehende `revenue_log` + `api_cost_log` + `fixed_costs`.
- Output: structured dict mit Anlage-EÜR-Zeilen (Z.14, Z.16, Z.17, Z.19, Z.26, Z.52, Z.57, Z.65, Z.72, Z.75, Z.77).
- USt-Voranmeldung: KZ 81/21/45/84/85/66/67 als separater Block.
- Reverse-Charge-Logik (§13b): API-Anbieter sind US-Drittland → KZ 84/85 + KZ 67 (Effekt netto 0, Meldepflicht).

### Kunden-Profitabilität (D-13)
- Pro User: Umsatz (aus `revenue_log` JOIN org_id) − Summe `api_cost_log` JOIN user_id im Zeitraum = Marge.
- Status: ✅ >70%, ⚠️ 50-70%, 🔴 <50%.
- Detail-View pro User mit Kosten-Aufschlüsselung nach Anbieter + 4-Wochen-Trend.

### Claude's Discretion
- Konkrete CSS/Layout-Implementierung der KPI-Karten (Briefing zeigt ASCII, Designsprache aus `static/nerve.css` übernehmen).
- Datenbank-Index-Strategie für `api_cost_log` (vermutlich `(user_id, created_at)` und `(provider, created_at)`).
- Caching-Strategie für teure Aggregationen (Übersichts-KPIs, Trend-Charts) — z.B. Materialized Views oder einfacher In-Memory-Cache mit 5-Min-TTL.
- Test-Strategie (pytest existiert seit 04.7.1).
