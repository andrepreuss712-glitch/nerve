## Summary

Phase 04.10 adds three distinct feature clusters to the existing training module: (1) realistic audio simulation replacing Web Audio API oscillators with pre-loaded MP3/OGG files, (2) a proper secretary/boss two-voice flow with three secretary personality types and a full transfer sequence, and (3) UX polish including a linear setup flow, a proper hangup popup with quick-stats, a user-initiated hangup button, and NERVE-glass-styled scenario cards.

All work touches exactly three files: `services/training_service.py` (secretary type prompts, SCHWIERIGKEITEN restructure, audio config), `routes/training.py` (accept `anruf_typ` + `sekretaerin_typ_id` in training_start, extend training_respond with transfer-sequence signal), and `templates/training.html` (audio file playback replacing oscillators, new setup flow sections, hangup overlay/popup, scenario card redesign). No new routes, no new DB tables, no new Python packages are required.

The existing `[DURCHGESTELLT]` tag mechanism in training.py is the load-bearing integration point for the transfer. The `confirmEndTraining()` function and the `t-hangup-overlay` div from 04.9 are the reuse anchors for the new hangup popup. The `_sessions[user_id]` in-memory dict already carries `sekretaerin_ueberwunden` — this is what feeds the popup's quick-stats.

**Primary recommendation:** Build in four tightly-scoped waves: (1) audio file infrastructure, (2) secretary types + transfer sequence backend, (3) hangup popup + user hangup button, (4) setup flow linear rebuild + scenario card CSS.

### Pattern 1: HTML5 Audio Playback (replacing oscillators)
**What:** Use `new Audio(src)` for one-shot sounds; use `audio.loop = true` for looping tones; stop via `audio.pause(); audio.currentTime = 0`.
**When to use:** All 5 audio signals in training.html.
**Key constraint:** Browser autoplay policy requires audio playback to happen inside a user-gesture event handler OR after the user has already interacted. The "Anrufen" button click IS a user gesture, so audio started in `startTraining()` will not be blocked. [VERIFIED: Chrome/MDN autoplay policy docs — the existing code already handles this with AudioContext.resume()]

```javascript
// Source: MDN Web Docs — HTMLAudioElement
// Replace startFreizeichen() / stopFreizeichen() with:
let activeAudio = null;

function playAudioLoop(src) {
  stopActiveAudio();
  activeAudio = new Audio(src);
  activeAudio.loop = true;
  activeAudio.volume = 0.6;
  activeAudio.play().catch(e => console.warn('[Audio]', e));
}

function playAudioOnce(src, onended) {
  const a = new Audio(src);
  a.volume = 0.6;
  if (onended) a.onended = onended;
  a.play().catch(e => console.warn('[Audio]', e));
  return a;
}

function stopActiveAudio() {
  if (activeAudio) {
    activeAudio.pause();
    activeAudio.currentTime = 0;
    activeAudio = null;
  }
}
```

### Anti-Patterns to Avoid
- **Playing audio in a non-gesture context:** Audio started asynchronously (e.g., in a setTimeout that fires 10 minutes after the last gesture) will be blocked by Chrome autoplay. Keep all audio calls within the call flow that began with the "Anrufen" button press. [VERIFIED: Chrome autoplay policy]
- **Modifying SCHWIERIGKEITEN dict to add secretary logic:** The current dict has `'sekretaerin'` as a 4th key — this should become a flag on the existing 3 difficulty levels, not a 4th difficulty. D-12 says "Sekretaerin + Schwierigkeitsgrad frei kombinierbar". The `sekretaerin` key in SCHWIERIGKEITEN must be removed from the diff-grid rendering (or kept but hidden) while the new "Anruf-Typ" section handles the on/off switch.
- **Adding DB tables for secretary types:** Secretary types are simple enough for Python constants (SEKRETAERIN_TYPES dict in training_service.py), same as SCHWIERIGKEITEN. No new DB table or migration needed.
- **Breaking existing sekretaerin mode:** The `'sekretaerin'` entry in SCHWIERIGKEITEN is currently selectable as a difficulty. D-11/D-13 move this to a separate "Anruf-Typ" step. The backend `hat_sekretaerin` flag logic in `training_start` must be preserved — only the UI trigger changes.

## Common Pitfalls

### Pitfall 1: SCHWIERIGKEITEN dict has `'sekretaerin'` as 4th key
**What goes wrong:** The existing `SCHWIERIGKEITEN` dict has a `'sekretaerin'` entry which is rendered as a 4th difficulty card in the UI. D-13 moves Anruf-Typ to a separate step. If the `sekretaerin` difficulty card is not removed from the diff-grid, users will see a confusing duplicate.
**Root cause:** The `sekretaerin` mode was originally a difficulty variant. 04.10 promotes it to a first-class call type.
**How to avoid:** Remove `'sekretaerin'` from the `{% for key, diff in schwierigkeiten.items() %}` Jinja loop either by filtering it out in the template (`{% if key != 'sekretaerin' %}`) or by removing it from `SCHWIERIGKEITEN` (but keeping the logic that `hat_sekretaerin` checks `diff.get('sekretaerin', False)` on the selected diff). The safest approach: keep `SCHWIERIGKEITEN` as-is in Python, filter it in the template, and add `hat_sekretaerin` as a separate flag driven by the new `anruf_typ` request field.

### Pitfall 2: `hat_sekretaerin` flag source change
**What goes wrong:** Currently `hat_sekretaerin = diff.get('sekretaerin', False)` in training_start reads from the selected difficulty. After D-11, `anruf_typ = 'sekretaerin'` is a separate request field. If both paths exist simultaneously, they conflict.
**Root cause:** Mechanical refactor of how `hat_sekretaerin` is determined.
**How to avoid:** In `training_start`, replace `hat_sekretaerin = diff.get('sekretaerin', False)` with `hat_sekretaerin = (data.get('anruf_typ') == 'sekretaerin')`. The SCHWIERIGKEITEN entry for `'sekretaerin'` can stay or be removed — it's no longer the source of truth.

### Pitfall 3: Audio file 404s blocking the training start
**What goes wrong:** If `freizeichen_de.mp3` doesn't exist in `static/audio/`, `new Audio(src).play()` returns a rejected promise. If not caught, it silently breaks the audio without breaking the training flow — but the user sees no dial tone.
**Root cause:** Audio files must be placed before the feature is deployed.
**How to avoid:** `.play().catch(e => console.warn('[Audio]', e))` (already shown in Pattern 1). The catch must NOT rethrow — audio failure is non-fatal. Document the audio files as a Wave 0 prerequisite.

### Pitfall 4: Transfer sequence timing race condition
**What goes wrong:** `training_respond` returns `durchgestellt=True` AND a `chef_text` + `chef_audio` in the same response. The frontend currently processes both immediately (lines 412-431 in training.py, rendered in training.html). The new audio sequence (wartemusik -> klingeln -> verbindungston) must run BEFORE the chef message is displayed, but the data is already available.
**Root cause:** Data arrives before animation completes.
**How to avoid:** When `durchgestellt=True` is detected in the JS response handler, stash `data.chef_text` / `data.chef_audio` in a variable, play the audio sequence with `await sleep()`, THEN render the chef message. The response structure does not need to change.

### Pitfall 5: Hangup popup vs existing hangup overlay conflict
**What goes wrong:** The existing `t-hangup-overlay` (red flash) is used in the `aufgelegt` handler at line 1078-1093. The new popup (D-08) replaces this flow. If both exist, the user sees the red flash AND the popup.
**Root cause:** 04.10 obsoletes the 2-second timeout -> confirmEndTraining pattern.
**How to avoid:** Remove the `setTimeout(() => confirmEndTraining(), 2000)` in the aufgelegt handler. Replace with: red flash (optional), then `handleAufgelegt()` which plays besetztzeichen and shows the popup.

### Pitfall 6: `loadPersonalityTypes()` called on DOMContentLoaded race
**What goes wrong:** In linear flow, `loadPersonalityTypes()` is called on page load instead of on modus selection. If the fetch fails, the Kundentyp section shows empty. This was previously hidden so failure was invisible.
**Root cause:** Linear flow exposes the section earlier.
**How to avoid:** Show a loading state (`t-typ-cards` innerHTML = "Laedt...") and handle fetch errors gracefully — show a retry button or a message.

def build_sekretaerin_type_prompt(persona: dict, sekretaerin_typ: str,

    lang    = TRAINING_LANGUAGES.get(sprache, TRAINING_LANGUAGES['de'])
    typ     = SEKRETAERIN_TYPES.get(sekretaerin_typ, SEKRETAERIN_TYPES['blockerin'])
    # Difficulty modifiers
    geduld_map = {'leicht': 'Du bist vergleichsweise geduldig (3-4 Austausche).',
                  'mittel': 'Du brauchst 2-3 Austausche.',
                  'schwer': 'Du hast wenig Geduld. 1-2 Austausche maximal.'}
    geduld = geduld_map.get(schwierigkeit, geduld_map['mittel'])
    prompt = typ['prompt_basis'].format(
        chef_name=persona['chef_nachname'],
        firma=persona['firma'],
        sek_name=persona['sek_name'],
    )
    prompt += f"\n\nGEDULD: {geduld}"

    prompt += "\n\nWICHTIG: Wenn du durchstellst, antworte EXAKT mit dem Text:\n\"Einen Moment bitte, ich verbinde Sie. [DURCHGESTELLT]\""
    return prompt

### Audio Manager JS (training.html)
```javascript
// Source: MDN HTMLAudioElement; replaces startFreizeichen/stopFreizeichen
const AUDIO_FILES = {
  klingeln:       '/static/audio/klingeln.mp3',
  besetztzeichen: '/static/audio/besetztzeichen.mp3',
  verbindungston: '/static/audio/verbindungston.mp3',
  wartemusik:     '/static/audio/wartemusik.mp3',
};
// Language-specific freizeichen fetched from TRAINING_LANGUAGES[lang].audio_freizeichen

let _loopAudio = null;

function audioPlayLoop(src) {
  audioStop();
  _loopAudio = new Audio(src);
  _loopAudio.loop = true;
  _loopAudio.volume = 0.5;
  _loopAudio.play().catch(e => console.warn('[Audio]', e));
}

function audioPlayOnce(src) {
  return new Promise(resolve => {
    const a = new Audio(src);
    a.volume = 0.5;
    a.onended = resolve;
    a.onerror = resolve;  // don't block on error
    a.play().catch(() => resolve());
  });
}

function audioStop() {
  if (_loopAudio) { _loopAudio.pause(); _loopAudio.currentTime = 0; _loopAudio = null; }
}
```

## Open Questions (RESOLVED)

1. **Audio file acquisition** (RESOLVED)
   - What we know: ~13 MP3 files needed, paths defined, playback code straightforward
   - What's unclear: Actual files must be sourced/recorded before the wave that replaces oscillators
   - Recommendation: Source from freesound.org (CC0), or generate synthetic versions with a simple Python script using scipy for authentic telephone tones. This is a Wave 0 prerequisite.
   - **Resolution:** Plan 02 Task 1 generates all 13 MP3 files via Python script using numpy + wave module (built-in). Script kept as permanent tool at `tools/generate_audio.py` for reproducibility.

2. **Sekretaerin difficulty: should `schwer` be hardcoded to 'blockerin' or user-selectable?** (RESOLVED)
   - What we know: D-12 says "frei kombinierbar" (all combinations allowed)
   - What's unclear: Whether Experte+Helferin makes training too easy
   - Recommendation: Allow all combinations per D-12. Claude's Discretion can adjust geduld parameter.
   - **Resolution:** All combinations allowed per D-12. Plan 01 Task 1 implements geduld_map with leicht/mittel/schwer modifiers applied to all secretary types equally.

3. **Transfer sequence when voice is unavailable** (RESOLVED)
   - What we know: `voice_available=False` degrades to text-only. The audio sequence still plays.
   - What's unclear: Should the transfer sequence (wartemusik->klingeln->verbindungston) still play in text-only mode?
   - Recommendation: Yes — telephone sounds are UI feedback, not TTS. They should play regardless of `voice_available`.
   - **Resolution:** Yes, audio sequence plays regardless of voice_available flag. Plan 02 Task 2 acceptance criteria explicitly require transfer sequence to play in both voice and text-only modes.
