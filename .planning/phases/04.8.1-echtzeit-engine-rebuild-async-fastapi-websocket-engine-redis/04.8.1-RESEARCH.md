## Summary

Das Kernproblem ist technisch eindeutig: `eventlet.monkey_patch()` kollidiert mit den nativen Threads des Deepgram SDK — das SDK öffnet eigene WebSocket-Verbindungen in separaten Threads, die unter eventlet nicht korrekt funktionieren. Die Lösung ist eine Split-Architektur: Flask (gthread, sync, kein eventlet) behält Web-UI und Auth; ein neuer FastAPI+uvicorn Service (async, native asyncio) übernimmt den gesamten Real-Time-Stack.

Die Technologieentscheidungen sind optimal für diesen Use Case: FastAPI mit nativen WebSockets ersetzt sowohl Flask-SocketIO als auch das HTTP-Polling. Das Deepgram SDK v3.10.0 (bereits installiert) hat einen `AsyncListenWebSocketClient` via `client.listen.asyncwebsocket.v("1")` — dieser ist nativ asyncio-kompatibel und hat keine eventlet-Konflikte. Redis als Bridge ist der Standard-Ansatz für Microservice-Kommunikation unter Python, und `redis-py` 7.4.0 enthält async-Support via `from redis import asyncio as aioredis` ohne separate Library.

Die größte Planungsherausforderung ist die **Session-Identity-Brücke**: Die neue Engine muss wissen, welcher User mit welcher Session verbindet — ohne auf Flask-Session-Cookies zuzugreifen. Das Standard-Muster ist: Flask stellt einen kurzlebigen Token aus (UUID im Redis mit TTL), den der Browser beim WebSocket-Handshake als Query-Parameter übergibt.

**Primary recommendation:** FastAPI `@app.websocket("/ws/{session_token}")` + AsyncDeepgramClient (`asyncwebsocket.v("1")`) + `redis-py` async pub/sub + systemd `nerve-rt.service` auf Port 8001, nginx `/ws/` → 8001.

### Pattern 1: FastAPI WebSocket Endpunkt

**Was:** Browser verbindet auf `/ws/{session_token}` mit einem kurzlebigen Token (UUID), den Flask zuvor in Redis gespeichert hat.

**Wann benutzen:** Immer — das ist der einzige Real-Time-Einstiegspunkt.

```python
# Source: fastapi.tiangolo.com/advanced/websockets/ + eigene Anpassung
@router.websocket("/ws/{session_token}")
async def websocket_endpoint(
    websocket: WebSocket,
    session_token: str,
    manager: SessionManager = Depends(get_session_manager),
):
    # 1. Token in Redis validieren → user_id, session_mode, profil_id
    session_data = await redis_bridge.get_session(session_token)
    if not session_data:
        await websocket.close(code=4401)
        return
    await websocket.accept()
    await manager.handle_session(websocket, session_data)
```

### Pattern 2: Deepgram asyncwebsocket Adapter

**Was:** `AsyncListenWebSocketClient` ist der asyncio-native Weg in deepgram-sdk v3.10.0. `start()` ist eine Coroutine, `send()` ist sync (schickt Bytes direkt in den internen Queue).

**Wichtig:** `start()` returned wenn die Verbindung steht. Die Verbindung läuft dann im Hintergrund via asyncio. Es ist NICHT `await asyncwebsocket.start()` in einem blocking loop — es ist `await ws_conn.start(options)` einmal, dann `ws_conn.send(audio_bytes)` in einem async loop.

```python
# Source: Live-Inspektion deepgram-sdk 3.10.0 + Deepgram Blog FastAPI-Artikel
from deepgram import DeepgramClient, LiveTranscriptionEvents, LiveOptions

class DeepgramAdapter:
    def __init__(self, api_key: str):
        self._client = DeepgramClient(api_key)

    async def connect(self, mode: str, on_transcript_cb) -> "connection":
        conn = self._client.listen.asyncwebsocket.v("1")
        conn.on(LiveTranscriptionEvents.Transcript, on_transcript_cb)
        options = LiveOptions(
            model="nova-2", language="de",
            smart_format=(mode != 'meeting'),
            interim_results=True, endpointing=900, punctuate=True,
            diarize=(mode == 'meeting'), encoding="linear16",
            sample_rate=16000,
            **({"utterance_end_ms": "1000"} if mode == 'meeting' else {}),
        )
        success = await conn.start(options)  # Coroutine — awaitable
        if not success:
            raise RuntimeError("Deepgram connection failed")
        return conn

    async def send_audio(self, conn, data: bytes):
        conn.send(data)  # sync send — thread-safe intern

    async def close(self, conn):
        await conn.finish()
```

**Kritischer Unterschied zu bestehendem Code:** Aktuell wird `client.listen.websocket.v("1")` (sync, threading) verwendet. Die neue Engine verwendet `client.listen.asyncwebsocket.v("1")` (async, asyncio). API-Oberfläche ist identisch, aber Lifecycle-Methoden sind Coroutines.

### Pattern 3: Redis Bridge (Flask → Engine)

**Was:** Flask schreibt Session-Config nach Redis, Engine liest sie. Engine publiziert Analyse-Ergebnisse, Flask kann sie pollen oder subscriben.

```python
# Source: redis.io docs + redis-py asyncio examples
from redis import asyncio as aioredis

# Engine-Seite
redis = await aioredis.from_url("redis://127.0.0.1:6379")

# Session starten (von Flask veröffentlicht):
# HSET nerve:session:{token} user_id 42 mode cold_call profile_id 7
# EXPIRE nerve:session:{token} 60  (60s Token-TTL)

async def get_session(token: str) -> dict | None:
    data = await redis.hgetall(f"nerve:session:{token}")
    return {k.decode(): v.decode() for k, v in data.items()} if data else None

# Ergebnis veröffentlichen (Engine → alle Subscriber):
await redis.publish(f"nerve:results:{user_id}", json.dumps(result_payload))
```

### Pattern 4: WebSocket Push (Engine → Browser)

**Was:** Engine pushed Analyse-Ergebnis direkt über die bestehende WebSocket-Verbindung. Kein Polling mehr.

```python
# Source: fastapi.tiangolo.com/advanced/websockets/
import json

async def push_analysis_result(websocket: WebSocket, result: dict):
    await websocket.send_json({
        "type": "analysis_update",
        "version": result["version"],
        "ergebnis": result["ergebnis"],
        "line_id": result["line_id"],
        "kaufbereitschaft": result["kaufbereitschaft"],
        "readiness_score": result["readiness_score"],
        "readiness_bucket": result["readiness_bucket"],
        "active_hint": result["active_hint"],
        "ewb_buttons": result["ewb_buttons"],
        "current_phase": result["current_phase"],
        "current_phase_name": result["current_phase_name"],
    })
```

**Frontend-Seite:** `pollErgebnis` (500ms setInterval chain) durch einen WebSocket-Message-Handler ersetzen:
```javascript
// Bestehend: const socket = io({ transports: ['polling'] })
// Neu: Native Browser WebSocket zur Engine
const rtSocket = new WebSocket(`wss://getnerve.app/ws/${sessionToken}`);
rtSocket.onmessage = (event) => {
    const data = JSON.parse(event.data);
    if (data.type === 'analysis_update') {
        // Gleiche Logik wie bisheriger pollErgebnis-Handler
        if (data.version > letzteVersion && data.ergebnis !== null) {
            zeigeKarte(data.ergebnis, data.line_id);
        }
        // ... rest unverändert
    }
};
```

**Wichtig:** Socket.IO (für Audio-Chunks, Transcript-Events) läuft weiter über Flask. Die neue Engine bekommt Audio-Chunks über einen separaten WebSocket-Kanal (nicht Socket.IO). Das bedeutet das Frontend braucht **zwei** Verbindungen während der Cutover-Phase.

### Anti-Patterns vermeiden

- **Audio-Chunks durch Redis routen:** VERBOTEN. Redis ist für Steuerungs-Signale, nicht für Audio-Stream-Daten. Latenz wäre inakzeptabel, Speicherverbrauch würde explodieren.
- **Flask-Session-Cookie in FastAPI lesen:** Nicht sicher, da verschiedene SECRET_KEYs möglich. Stattdessen: kurzlebiger UUID-Token in Redis.
- **`blocking_redis.listen()` im asyncio Event Loop:** Blockiert den Loop. Stattdessen `asyncio.create_task(redis.subscribe(...))` mit `async for message in pubsub.listen()`.
- **analyse_loop und coaching_loop 1:1 portieren:** Diese sync `threading.Event.wait()` Loops MÜSSEN als asyncio Tasks neu geschrieben werden. Kein `time.sleep()` im async Code.

## Don't Hand-Roll

| Problem | Nicht selbst bauen | Stattdessen nutzen | Warum |
|---------|--------------------|--------------------|-------|
| WebSocket-Protocol-Handling | Eigene Frame-Parser | `fastapi.WebSocket` (via Starlette/uvicorn) | Masking, fragmentation, ping/pong, close-handshake sind komplex |
| Async Redis-Verbindungspool | Eigener Pool | `redis.asyncio.from_url()` mit Pool-Default | redis-py verwaltet Connection-Pool automatisch |
| Session-Token-Sicherheit | Eigener Zufallsgenerator | `secrets.token_urlsafe(32)` | Kryptografisch sicher, stdlib |
| Pydantic-Validierung | Manuelle dict-Checks | FastAPI Pydantic-Modelle für WS-Messages | Type-Safety, Fehler-Meldungen, Dokumentation gratis |
| Deepgram async Reconnect | Eigene Reconnect-Logik | SDK-internes Reconnect (bereits implementiert) | Deepgram SDK hat eigenes Retry-Handling |

## Common Pitfalls

### Pitfall 1: Audio-Bytes über asyncio.Queue ohne Backpressure
**Was schiefläuft:** Browser schickt Audio-Chunks schneller als Deepgram sie verarbeiten kann. Queue wächst unbegrenzt.
**Warum:** asyncio.Queue hat kein maxsize per default.
**Vermeidung:** `asyncio.Queue(maxsize=100)` — wenn voll, älteste Chunks droppen (nicht blockieren!). Audio-Latenz ist besser als Memory-Exhaustion.
**Warnsignal:** Memory-Verbrauch der Engine wächst während Session.

### Pitfall 2: Blocking Code im asyncio Event Loop
**Was schiefläuft:** `anthropic.Anthropic().messages.create(...)` ist synchron und blockiert den gesamten Event Loop. Alle anderen WebSocket-Verbindungen frieren ein.
**Warum:** FastAPI/uvicorn nutzt einen einzigen Thread für alle async Tasks.
**Vermeidung:** `loop.run_in_executor(None, sync_claude_call, ...)` oder async anthropic client (`anthropic.AsyncAnthropic`). Die anthropic-Library hat seit v0.7.0 einen async Client. [ASSUMED: async Client ist in 0.86.0 verfügbar — basierend auf Changelog-Pattern]
**Warnsignal:** Alle Sessions "frieren" gleichzeitig ein wenn Claude antwortet.

### Pitfall 3: Redis pub/sub listener blockiert asyncio
**Was schiefläuft:** `pubsub.get_message()` in einem `while True` Loop ohne `await` blockiert.
**Warum:** Synchrones Redis-Polling im async Context.
**Vermeidung:** `async for message in pubsub.listen():` — das ist der korrekte async-Weg in redis-py asyncio. [VERIFIED: redis.readthedocs.io asyncio examples]
**Warnsignal:** Engine hängt, nur Redis-Subscriber bekommt Updates.

### Pitfall 4: Flask-Session-Cookie kann nicht in FastAPI gelesen werden
**Was schiefläuft:** `request.cookies.get('session')` in FastAPI gibt den Flask-itsdangerous-signierten Cookie zurück, aber FastAPI kennt den Flask SECRET_KEY nicht und kann ihn nicht dekodieren.
**Warum:** Flask-Session ist verschlüsselt mit HMAC(SECRET_KEY, payload).
**Vermeidung:** Kurzlebiger UUID-Token in Redis (Pattern 3). Flask stellt Token aus, Engine validiert Token gegen Redis. Keine gemeinsame Session-Mechanik nötig.
**Warnsignal:** Auth-Fehler auf WebSocket-Connect trotz eingeloggtem User.

### Pitfall 5: Deepgram asyncwebsocket start() return type
**Was schiefläuft:** `conn.start(options)` gibt `bool` zurück, KEIN Coroutine-Objekt. Aufruf als `await conn.start(options)` kann je nach SDK-Version Fehler werfen.
**Warum:** Introspection bestätigt `start signature: (...) -> bool`. Aber als `await conn.start()` funktioniert es trotzdem — Python erlaubt await auf non-coroutines wenn __await__ implementiert ist.
**Vermeidung:** Prüfen ob start() awaitable ist: `asyncio.iscoroutinefunction(conn.start)`. Im Zweifelsfall in `run_in_executor` wrappen.
**Warnsignal:** `TypeError: object bool can't be used in 'await' expression`

### Pitfall 6: Zwei WebSocket-Verbindungen im Frontend während Cutover
**Was schiefläuft:** Frontend öffnet Socket.IO (Flask, für Audio) UND native WebSocket (Engine, für Analyse). Wenn eine der beiden disconnectet, ist das Session-State inkonsistent.
**Warum:** Split-Phase — Audio läuft noch über Flask, Analyse kommt von Engine.
**Vermeidung:** Definierter State-Machine im Frontend: wenn RT-WebSocket disconnectet → fallback auf pollErgebnis (window._pollingActive = true). Beide Verbindungen haben unabhängige Reconnect-Logik.
**Warnsignal:** Nutzer sieht Transkript aber keine Analyse-Karten.

### Pitfall 7: live_session.py Globals sind Single-Process
**Was schiefläuft:** Die neue Engine importiert `services/live_session.py` aus dem Flask-Code. Die threading.Lock Globals in live_session.py haben keinen State-Sync zwischen Prozessen.
**Warum:** Module-Globals existieren pro Process. Engine-Prozess und Flask-Prozess haben separate Kopien.
**Vermeidung:** Die Engine bekommt EIGENE Session-State-Klasse (asyncio-kompatibel, pro-Session Instanz). live_session.py wird von der Engine NICHT importiert. Geteilter State nur über Redis.
**Warnsignal:** State-Updates in Engine erscheinen nicht in Flask-Dashboard.

## Code Examples

### STTProvider Interface (ABC)

```python
# Source: Python abc-Modul + D-03 Anforderungen aus CONTEXT.md
from abc import ABC, abstractmethod
from typing import Callable, Awaitable

class STTProvider(ABC):
    """Abstraktes Interface für Speech-to-Text Providers."""

    @abstractmethod
    async def connect(
        self,
        mode: str,                              # 'cold_call' | 'meeting'
        on_transcript: Callable[..., Awaitable] # async callback: (text, is_final, speaker) -> None
    ) -> object:
        """Öffnet STT-Verbindung. Returns connection handle."""

    @abstractmethod
    async def send_audio(self, conn: object, data: bytes) -> None:
        """Schickt Audio-Chunk an STT-Provider."""

    @abstractmethod
    async def close(self, conn: object) -> None:
        """Schließt STT-Verbindung sauber."""
```

### LLMProvider Interface (ABC)

```python
# Source: D-04 + D-05 aus CONTEXT.md + bestehender analysiere_mit_claude() Signatur
from abc import ABC, abstractmethod

class LLMProvider(ABC):
    """Abstraktes Interface für LLM-Provider."""

    @abstractmethod
    async def analyse(
        self,
        text: str,
        context: str,
        profile_data: dict,
        session_mode: str,
    ) -> dict:
        """Analysiert Transkript-Segment. Returns strukturiertes JSON-Ergebnis."""

    @property
    @abstractmethod
    def model_id(self) -> str:
        """Provider-Identifier für Logging (z.B. 'claude-haiku-4-5-20251001')."""
```

## State of the Art

| Alter Ansatz | Aktueller Ansatz | Geändert | Impact |
|--------------|------------------|----------|--------|
| aioredis (separates Paket) | redis-py asyncio (`from redis import asyncio as aioredis`) | redis-py 4.2.0 (2022) | Kein separates Paket mehr nötig |
| Deepgram `listen.websocket` (threading) | `listen.asyncwebsocket` (asyncio) | SDK v3.x | Native asyncio, kein eventlet-Konflikt |
| Flask-SocketIO mit eventlet | FastAPI native WebSocket | Dieser Phase | Löst das Root-Problem |
| HTTP Polling (500ms) | WebSocket Push | Dieser Phase | -100% Polling-Load, echtes Push |
| Sync `threading.Event.wait()` Loops | `asyncio.Task` mit `await asyncio.sleep()` | Dieser Phase | Kein Thread-Overhead, bessere Cancellation |

**Deprecated/Outdated:**
- `eventlet.monkey_patch()`: Funktioniert nicht mit Deepgram SDK — dies ist der Root Cause dieses Rebuilds
- `aioredis` (separates PyPI Package): Offiziell archived, in redis-py aufgegangen. Letzte Version 2.0.1, Dezember 2021. [CITED: github.com/aio-libs-abandoned/aioredis-py]

## Open Questions

1. **Anthropic async Client Verfügbarkeit**
   - Was wir wissen: `anthropic` 0.86.0 ist installiert. Training-Wissen sagt AsyncAnthropic ist seit ~v0.7 verfügbar.
   - Was unklar ist: Genaue API-Signatur von `AsyncAnthropic.messages.create()` in 0.86.0
   - Empfehlung: Planner soll `python -c "from anthropic import AsyncAnthropic; print('ok')"` als Wave-0-Check einbauen. Bei Fehler Fallback auf `run_in_executor`.

2. **Redis bereits auf VPS?**
   - Was wir wissen: Redis ist lokal nicht installiert. VPS-Status unbekannt (kein SSH-Zugriff während Research).
   - Was unklar ist: Ist Redis-Server bereits auf dem Hetzner CX22 installiert?
   - Empfehlung: Deploy-Wave prüft per `redis-cli ping` bevor Installation. `redis-cli ping` → `PONG` = bereits installiert.

3. **nginx /ws/ vs /socket.io/ Routing-Konflikt**
   - Was wir wissen: Bestehend läuft `location /` alles auf :8000. Socket.IO nutzt `/socket.io/` Prefix.
   - Was unklar ist: Ob `location /ws/` VOR `location /` platziert werden muss (nginx longest-match precedence).
   - Empfehlung: `/ws/` als named location VOR `/` in nginx.conf platzieren. nginx matcht specific locations vor general.
