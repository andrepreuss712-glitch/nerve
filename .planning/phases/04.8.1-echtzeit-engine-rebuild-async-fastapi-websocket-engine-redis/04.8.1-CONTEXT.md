# Phase 04.8.1: Echtzeit-Engine Rebuild - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning
**Source:** User discussion (2026-04-09/10 session)

<domain>
## Phase Boundary

Replace the polling-based real-time pipeline with an async FastAPI+uvicorn WebSocket engine running as a separate service. Flask keeps web UI (auth, payments, dashboard). Redis bridges communication between Flask and the engine. Abstract STT and LLM interfaces so providers can be swapped without touching business logic.

**What this phase delivers:**
- A standalone async real-time engine (FastAPI + uvicorn) handling WebSocket connections, STT streaming, LLM analysis, and push notifications
- Redis pub/sub bridge between Flask (web app) and the real-time engine
- STTProvider abstraction layer with Deepgram adapter (faster-whisper adapter later)
- LLMProvider abstraction layer with Claude adapter and Shadow Mode infrastructure (fine-tuned model later)
- WebSocket-based push replacing HTTP polling for all live-session data
- Zero-downtime cutover: build alongside existing system, switch when ready

**What this phase does NOT deliver:**
- faster-whisper adapter (only the interface — actual adapter is a later phase)
- Fine-tuned model (only the shadow infrastructure — actual model training is a later phase)
- Frontend redesign (frontend gets minimal changes: switch from polling to WebSocket listener)
- PostgreSQL migration (stays SQLite for now)

</domain>

<decisions>
## Implementation Decisions

### D-01: Split Architecture (LOCKED)
Flask stays for web UI (auth, payments, dashboard, static pages). A new FastAPI+uvicorn service handles all real-time work (WebSocket connections, STT streaming, LLM analysis, push notifications). Two separate processes, same VPS.

**Rationale:** eventlet monkey_patch() breaks Deepgram SDK WebSocket threading. Flask+eventlet cannot scale real-time AND web simultaneously. Separation lets each use the right async model.

### D-02: Redis Bridge (LOCKED)
Redis pub/sub for communication between Flask and the real-time engine. Flask publishes session lifecycle events (start, stop, config changes). Engine publishes analysis results that Flask can query for dashboard/API.

**Rationale:** Loose coupling. Either service can restart independently. Redis is lightweight, already proven at scale, and available on Hetzner.

### D-03: STT Abstraction Layer (LOCKED)
Abstract `STTProvider` interface. Deepgram adapter implements it now. Interface must support:
- Streaming audio input (chunked WebSocket frames)
- Real-time transcript callbacks
- Speaker diarization (meeting mode)
- Language configuration
- Connection lifecycle (open, close, reconnect)

faster-whisper adapter comes later (break-even at ~25k-50k streaming minutes/month). The interface must NOT assume Deepgram-specific features (e.g., smart_format, utterance_end_ms are adapter internals, not interface methods).

### D-04: LLM Abstraction Layer with Shadow Mode (LOCKED)
Abstract `LLMProvider` interface. Claude (Haiku) adapter implements it now. Interface must support:
- Synchronous analysis call (transcript + context in, structured result out)
- Model configuration (model name, temperature, max tokens)
- Response parsing (each adapter handles its own output format)

**Shadow Mode:** When enabled, both primary and shadow providers get identical input. Only primary result is shown to user. Both results are logged with timestamps for comparison. Shadow provider failures are silent (never affect user experience).

### D-05: Shadow Mode Data Collection (LOCKED)
Fine-tuned model will be trained on REAL user data, NOT Claude outputs. Training data comes from:
- Which EWB button the user actually clicked (ground truth for objection detection)
- Whether readiness score went up or down after an intervention
- Which coaching hints the user found useful (implicit: did they act on it?)
- Session outcomes (successful close, abandoned, etc.)

Claude outputs are logged for comparison only, never as training targets. This prevents the fine-tuned model from just imitating Claude — it learns from actual user behavior.

### D-06: WebSocket Push (LOCKED)
Replace 500ms HTTP polling with WebSocket push for all live-session data:
- Transcript updates
- Analysis results (score, phase, hints, EWB buttons)
- Status changes (mic state, connection state)
- Error notifications

Frontend changes: replace `setInterval(pollErgebnis, 500)` with `socket.on('analysis_update', ...)` listener. Minimal frontend change — data format stays the same, only transport changes.

### D-07: Build Alongside, Then Cutover (LOCKED)
The new engine runs in parallel with the existing Flask real-time code. Both work simultaneously during development. Cutover happens when the new engine passes all verification. No big-bang rewrite.

**Rationale:** Zero risk of breaking production. Can A/B test between old and new. Rollback = just point nginx back to Flask.

### D-08: VPS Deployment (LOCKED)
Both services run on the same Hetzner VPS (178.104.82.166). nginx routes:
- `/socket.io/` and `/ws/` → FastAPI engine (uvicorn)
- Everything else → Flask (gunicorn)

systemd manages both services independently. Redis runs locally on the VPS.

### D-09: DSGVO Compliance (LOCKED - from CLAUDE.md)
- No audio storage. Ever. Ephemeral processing only.
- Cold call mode: only seller audio, EWB buttons for customer objections
- Meeting mode: consent popup before recording
- All services on EU servers (Deepgram EU endpoint, Claude Bedrock Frankfurt, Hetzner Nuernberg)

<specifics>
## Specific Ideas

- **Eventlet problem:** Current Flask uses eventlet monkey_patch() which breaks Deepgram SDK's native WebSocket threading. This is the primary technical driver for the split architecture.
- **Polling scalability:** 500ms polling = 2 req/s per user. At 100 concurrent users = 200 req/s just for status polling. WebSocket push eliminates this entirely.
- **faster-whisper context:** No native streaming support (needs custom VAD pipeline), no built-in diarization, but better dialect handling and cheaper at scale. The STT interface must accommodate both streaming (Deepgram) and batch-with-VAD (faster-whisper) patterns.
- **Shadow Mode logging:** Needs structured storage for A/B comparison. Both model outputs + user actions logged per analysis cycle. Schema: timestamp, session_id, input_hash, primary_output, shadow_output, user_action (which EWB clicked, score delta).

</specifics>
