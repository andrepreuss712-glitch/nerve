# Phase 04.6.1: Auth-Upgrade Google + Microsoft OAuth Login - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning
**Source:** PRD Express Path (NERVE Phase 4.6.2 Auth-Upgrade.md)

<domain>
## Phase Boundary

Adds Google and Microsoft OAuth login to NERVE alongside the existing email/password auth. Users can sign in with either provider. New users get an Org + User created automatically (without password). Existing email/password users logging in via OAuth get their accounts linked (no duplicates). Login UI on the landing page gets Google and Microsoft buttons. This phase does NOT include 2FA, password reset, or social login on the invite-registration page.

</domain>

- All four must be added to `.env.example` (without real values) and to the production environment

### New Routes — separate `routes/oauth.py` (keep auth.py clean)
- `GET /auth/google` — redirect to Google OAuth authorization URL
- `GET /auth/google/callback` — exchange code for token, fetch userinfo, login or register
- `GET /auth/microsoft` — redirect to Microsoft OAuth authorization URL
- `GET /auth/microsoft/callback` — exchange code for token, fetch userinfo, login or register

### Callback Logic (identical for both providers)
1. Provider returns `email`, `name`, `picture`
2. Look up User by email in DB:
   - **Found:** call `_do_login`-equivalent (set `session['user_id']`), set `oauth_provider` + `oauth_id` if not already set
   - **Not found:** create new Org (name from `name` field, updateable in onboarding) + new User without password
3. Check `onboarding_done` → redirect to wizard if False, else dashboard

### User Model Changes (`database/models.py`)
```python
passwort_hash = Column(String(256), nullable=True)   # was NOT NULL — must migrate
oauth_provider = Column(String(50), nullable=True)    # 'google' | 'microsoft' | None
oauth_id       = Column(String(200), nullable=True)   # provider's sub/id field
avatar_url     = Column(String(500), nullable=True)   # profile picture URL
```

### Database Migration
- SQLite: existing `passwort_hash` column needs to be migrated to nullable (SQLite ALTER TABLE cannot change nullability directly — use table rebuild or a migration script)
- New columns (`oauth_provider`, `oauth_id`, `avatar_url`) added via `ALTER TABLE` — they are nullable by default in SQLite so no data loss

### Login UI (`templates/landing.html`)
- In the login modal: add "Mit Google anmelden" button (Google logo + text, full width) pointing to `/auth/google`
- Below that: "Mit Microsoft anmelden" button (Microsoft logo + text, full width) pointing to `/auth/microsoft`
- Add "— oder —" separator between OAuth buttons and the existing email/password form
- In the register modal (if exists): same two OAuth buttons at top
- When registering via OAuth: org name is NOT asked in the form — it's collected in the onboarding wizard instead

### Account Linking
- Email match on OAuth callback → use existing account, update `oauth_provider` + `oauth_id` on user row
- NEVER create duplicate accounts for the same email
- OAuth-registered users (no password): "Passwort setzen" option is nice-to-have, out of scope for v1

### Security
- Use authlib's built-in state parameter (CSRF protection) — do NOT implement manually
- Use nonce for Google ID-token validation
- HTTPS-only enforcement is already in place (no action needed)
- Registered callback URLs (do NOT change):
  - Google prod: `https://app.getnerve.app/auth/google/callback`
  - Google local: `http://localhost:5000/auth/google/callback`
  - Microsoft prod: `https://app.getnerve.app/auth/microsoft/callback`

### What NOT to build
- ❌ 2FA / TOTP
- ❌ Password reset via email
- ❌ Social login on the invite-registration page (`/register?token=...`)

### Claude's Discretion
- How to store authlib OAuth client instances (module-level or app factory)
- Error handling for OAuth failures / user cancellation → show flash message, redirect to login
- Whether to use authlib's Flask integration (`authlib.integrations.flask_client`) vs. raw requests
- Org name fallback when provider doesn't return a name

</decisions>
