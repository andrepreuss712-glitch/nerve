# Phase 04.6.1: Auth-Upgrade (Google + Microsoft OAuth) - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning
**Source:** PRD Express Path (NERVE Phase 4.6.2 Auth-Upgrade.md)

<domain>
## Phase Boundary

User können sich mit Google- oder Microsoft-Account einloggen. Bestehende Email/Passwort-Registrierung bleibt erhalten. Neue User werden automatisch via OAuth angelegt (inkl. Org), bestehende Email-User werden per Email-Match mit OAuth verknüpft (kein Duplikat). Login-Modal auf Landing Page bekommt Google-/Microsoft-Buttons. Keine Änderung am bestehenden Session-Modell, keine Änderung am Invitation-Flow.

</domain>

<decisions>
## Implementation Decisions

### OAuth-Library & Dependencies
- Neues Package: `authlib` in `requirements.txt` (nutzt State-Parameter für CSRF-Schutz + Nonce für ID-Token automatisch)
- Keine weiteren Auth-Frameworks einführen — Session-basierte Auth bleibt bestehen
- Kein Framework-Wechsel (Flask + Vanilla JS Constraint, CLAUDE.md)

### Env-Vars (bereits eingerichtet)
- `GOOGLE_CLIENT_ID=1007634969043-0o11o76lshkhgki6p938pjai8tl7o9n4.apps.googleusercontent.com`
- `GOOGLE_CLIENT_SECRET=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`
- `MICROSOFT_CLIENT_ID=5b198801-70ca-4dfb-b7bf-11fe3c4a789f`
- `MICROSOFT_CLIENT_SECRET=VY48Q~8x7InEIOM22a66i5eFPiV2SPjOXK1Jgae_`
- Müssen in `.env` / `config.py` geladen werden (wie bestehende Secrets via `python-dotenv`)
- `.env.example` entsprechend erweitern

### Routes (neues Blueprint `oauth_bp` in `routes/oauth.py`)
- `GET /auth/google` → Authorization-Redirect zu Google
- `GET /auth/google/callback` → Code → Token → Userinfo → Login/Register
- `GET /auth/microsoft` → Authorization-Redirect zu Microsoft
- `GET /auth/microsoft/callback` → Code → Token → Userinfo → Login/Register
- Registrierung: `app.py` registriert `oauth_bp` analog zu `auth_bp`

### Callback-Logik (identisch für Google & Microsoft)
1. Userinfo holen (`email`, `name`, `picture`)
2. User per Email in DB suchen:
   - Existiert → Session setzen (wie `_do_login` aus `routes/auth.py`), ggf. `oauth_provider`/`oauth_id`/`avatar_url` nachziehen
   - Existiert nicht → neuen User + neue Org anlegen (analog `api_register`, ohne Passwort)
3. `onboarding_done` prüfen → ggf. Redirect auf Wizard
4. Sonst Redirect auf Dashboard
5. `_do_login` & Register-Helper wiederverwenden, NICHT duplizieren

### User-Model Änderungen (`database/models.py`)
- `passwort_hash` → `nullable=True` (Migration nötig)
- `oauth_provider` → `Column(String(50), nullable=True)` ('google' | 'microsoft' | None)
- `oauth_id` → `Column(String(200), nullable=True)` (Provider Sub-ID)
- `avatar_url` → `Column(String(500), nullable=True)`
- Migration: in `app.py` Migration-Sektion (siehe bestehende `[DB] Migration: added users.{col}` Pattern) — neue Spalten per ALTER TABLE ADD, `passwort_hash` Nullability-Migration SQLite-kompatibel (ggf. Rebuild-Tabelle oder pragmatisch belassen, falls SQLite in Praxis NULL toleriert)

### Account-Verknüpfung (Rule)
- Email-Match → gleichen Account wiederverwenden, `oauth_provider` + `oauth_id` + `avatar_url` setzen
- Keine Duplikat-Accounts
- OAuth-only User können sich nicht via Email/Passwort einloggen — „Passwort setzen" im Profil ist NICHT Teil dieser Phase

### Login-UI (`templates/landing.html`)
- Login-Modal: „Mit Google anmelden" Button (volle Breite, Google-Logo + Text) über dem Email/Passwort-Formular → `/auth/google`
- Darunter: „Mit Microsoft anmelden" Button (volle Breite, MS-Logo + Text) → `/auth/microsoft`
- Trennlinie „— oder —" zwischen OAuth-Buttons und Email-Formular
- Register-Modal (falls vorhanden): gleiche zwei OAuth-Buttons
- OAuth-Registrierung: Firmenname wird über Onboarding-Wizard abgefragt (nicht im OAuth-Flow)
- CSS folgt `nerve.css` / bestehendem Button-Stil

### Sicherheit
- `state`-Parameter (CSRF) — authlib automatisch
- Nonce für ID-Token Validierung — authlib automatisch
- HTTPS only in Produktion (bereits gegeben)
- Google/Microsoft Redirect URIs sind eingerichtet:
  - Google: `https://app.getnerve.app/auth/google/callback`, `http://localhost:5000/auth/google/callback`
  - Microsoft: `https://app.getnerve.app/auth/microsoft/callback`
- DSGVO: OAuth-Provider sind Third-Parties, berühren keine Audio-/Call-Daten (ephemeral processing bleibt unberührt)

### Out of Scope (explizit NICHT bauen)
- 2FA / TOTP
- Passwort-Reset per Email (kein Email-Service)
- OAuth-Buttons auf Register-via-Invitation Seite (nur Landing Page)
- „Passwort setzen" Flow für OAuth-Only User

### Claude's Discretion
- Exakte Template-Struktur & CSS-Klassen der OAuth-Buttons (an `nerve.css` anlehnen)
- Konkrete SQLite-Migrationsstrategie für `passwort_hash` Nullability (in Linie mit `[DB] Migration: ...` Pattern in `app.py`)
- Fehlermeldungs-Texte bei OAuth-Abbruch / Provider-Fehler (kurz, Deutsch, „Anti-Berater"-Stil)
- Genaue Code-Aufteilung zwischen `routes/oauth.py` und Helper-Funktionen in `routes/auth.py`
- Test-Ansatz (pytest + Mock-Provider) falls Test-Suite vorhanden

</decisions>

<specifics>
## Specific Ideas

- `authlib` ist Referenz-Library (Quelle: PRD)
- Callback-Logik soll `_do_login` aus `routes/auth.py` wiederverwenden, nicht duplizieren
- Register-Logik (Org + User anlegen) soll aus `api_register` in einen Helper extrahiert werden, sodass OAuth-Pfad und Email-Pfad dieselbe Funktion nutzen
- Redirect nach erfolgreichem OAuth: gleiche Logik wie Email-Login (Onboarding-Wizard wenn `onboarding_done=False`, sonst Dashboard)
- Naming: deutsche Domain-Begriffe wo passend (`passwort_hash`), aber Auth-Standards (`oauth_provider`, `oauth_id`) englisch

</specifics>
