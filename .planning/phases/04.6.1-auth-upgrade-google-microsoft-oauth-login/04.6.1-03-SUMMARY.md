3. **Register-Modal Step 1** (direkt nach `<div class="step-panel active" id="step1">`, vor `.form-row`):
   - Identische zwei `<a>`-Buttons mit IDs `r-google` / `r-microsoft` und Text "Mit Google/Microsoft registrieren"
   - Divider-Text "oder mit E-Mail registrieren"

4. **JavaScript IIFE** (~Zeile 723) erweitert:
   ```javascript
   const oauthErr=p.get('oauth_error');
   if(oauthErr){
     const errMap={ google_failed:..., microsoft_failed:..., unverified:..., not_configured:..., no_email:..., inactive:... };
     openLogin();
     const el=document.getElementById('l-error');
     if(el){ el.textContent=errMap[oauthErr]||'OAuth-Login fehlgeschlagen.'; el.classList.add('show'); }
     return;
   }
   ```
   - `textContent` (nicht `innerHTML`) → kein XSS
   - errMap-Lookup → URL-Wert wird nie direkt gerendert

## Threat Mitigations Applied

| Threat ID | Mitigation |
|-----------|------------|
| T-04.6.1-16 (Tampering oauth_error Rendering) | errMap clientseitig + `textContent` statt `innerHTML` → kein HTML-Injection / XSS via URL-Param |
| T-04.6.1-17 (Info Disclosure OAuth-href) | href ist statisches `/auth/google` bzw. `/auth/microsoft` — kein User-Input, kein Open-Redirect-Vector |
| T-04.6.1-18 (Spoofing externe SVG-Logos) | Inline-SVG, kein CDN-Load, kein 3rd-Party-Tracking |
| T-04.6.1-19 (Repudiation OAuth-Klick) | Akzeptiert — Server-side `[OAuth] ... login` Print-Audit aus Plan 02 ausreichend |
