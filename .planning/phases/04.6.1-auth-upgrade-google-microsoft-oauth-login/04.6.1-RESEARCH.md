# Phase 04.6.1: Auth-Upgrade (Google + Microsoft OAuth) – Research

**Researched:** 2026-04-06
**Domain:** Flask Session-Auth + OAuth 2.0 / OIDC (Google, Microsoft)
**Confidence:** HIGH (Stack, Patterns), MEDIUM (SQLite-Nullability-Migration)

## Summary

Die bestehende NERVE-Auth ist ein klassisches Flask-Session-Setup mit `werkzeug.security` Password-Hashing, `_do_login()` als zentraler Einlogg-Helper und `api_register` als Account+Org-Anlegehelfer. Für OAuth wird `authlib` als Flask-Client eingehängt — authlib übernimmt State/CSRF, Nonce, PKCE und ID-Token-Validierung automatisch, sodass wir nur `authorize_redirect()` und `authorize_access_token()` aufrufen. Google liefert Userinfo direkt im ID-Token (OIDC), Microsoft ebenso via `common`-Endpoint (Multi-Tenant für Work + Personal).

Zwei Fallstricke dominieren die Planung: (1) `passwort_hash` ist `NOT NULL` und SQLite kann `ALTER COLUMN` nicht — wir brauchen entweder einen Tabellen-Rebuild oder den pragmatischen Weg, das `NOT NULL` im ORM aufzuheben und neue OAuth-User mit einem Dummy-Hash `''` anzulegen. (2) `_do_login` nimmt aktuell `(email, passwort)` — für OAuth muss es refactored werden in einen „Login-by-User-Object"-Pfad.

**Primary recommendation:** `authlib` 1.3+ einhängen, `_do_login` in zwei Funktionen splitten (`_login_user(db, user)` und `_do_login(email, passwort)`), Register-Logik aus `api_register` in `_create_org_and_user(email, vorname, nachname, firmenname, passwort_hash=None)` extrahieren, SQLite-Rebuild vermeiden → neue OAuth-User bekommen `passwort_hash=''` (leerer String, `check_password_hash('', x)` gibt False, Email-Login bleibt sicher blockiert).

## Architecture Patterns

### Blueprint-Struktur (entspricht NERVE-Konventionen)

```
routes/
├── auth.py       # bestehend — _do_login REFACTORED, Register-Helper EXTRAHIERT
└── oauth.py      # NEU — oauth_bp mit Google + Microsoft Flows
```

### Pattern 1: authlib OAuth Client Registration (auf App-Ebene)

**Was:** `OAuth` Instanz als Modul-Global in `routes/oauth.py`, Clients per `register()` für Google + Microsoft. authlib nutzt `server_metadata_url` (Discovery-Endpoint), um Endpoints, JWKS und Scopes automatisch zu laden.

**Quelle:** [CITED: docs.authlib.org/en/latest/client/flask.html — "Flask Client"]

```python
# routes/oauth.py (Skizze — vollständige Implementierung im PLAN)
from authlib.integrations.flask_client import OAuth
from flask import Blueprint, redirect, url_for, session, flash, current_app
import os

oauth_bp = Blueprint('oauth', __name__)
oauth = OAuth()

def init_oauth(app):
    """Wird in app.py nach Blueprint-Registrierung aufgerufen."""
    oauth.init_app(app)
    oauth.register(
        name='google',
        client_id=os.environ['GOOGLE_CLIENT_ID'],
        client_secret=os.environ['GOOGLE_CLIENT_SECRET'],
        server_metadata_url='https://accounts.google.com/.well-known/openid-configuration',
        client_kwargs={'scope': 'openid email profile'},
    )
    oauth.register(
        name='microsoft',
        client_id=os.environ['MICROSOFT_CLIENT_ID'],
        client_secret=os.environ['MICROSOFT_CLIENT_SECRET'],
        server_metadata_url='https://login.microsoftonline.com/common/v2.0/.well-known/openid-configuration',
        client_kwargs={'scope': 'openid email profile'},
    )

@oauth_bp.route('/auth/google')
def google_login():
    redirect_uri = url_for('oauth.google_callback', _external=True)
    return oauth.google.authorize_redirect(redirect_uri)

@oauth_bp.route('/auth/google/callback')
def google_callback():
    try:
        token = oauth.google.authorize_access_token()  # validiert State + Nonce automatisch
    except Exception as e:
        flash('Google-Login abgebrochen oder fehlgeschlagen.', 'error')
        return redirect('/?modal=login')
    userinfo = token.get('userinfo')  # authlib parst ID-Token automatisch
    email = (userinfo.get('email') or '').strip().lower()
    if not email or not userinfo.get('email_verified', False):
        flash('Email vom Provider nicht bestätigt.', 'error')
        return redirect('/?modal=login')
    return _oauth_login_or_create(
        provider='google',
        oauth_id=userinfo['sub'],
        email=email,
        vorname=userinfo.get('given_name', ''),
        nachname=userinfo.get('family_name', ''),
        avatar_url=userinfo.get('picture'),
    )
```

**Wichtig für Microsoft:** Das `userinfo`-Dict kommt aus dem ID-Token (OIDC). Microsoft hat `email_verified` nicht immer gesetzt — Workaround unten in „Pitfalls".

### Pattern 2: `_do_login` Refactoring (Split in zwei Funktionen)

**Was:** `_do_login(email, passwort)` macht heute zu viel — User finden + Passwort prüfen + Session setzen + DbSession anlegen. Split:

```python
# routes/auth.py — NEUE Struktur
def _login_user(db, user):
    """Session + DbSession für einen bereits validierten User setzen.
    Liest alle User-Attribute VOR db.close() (siehe CLAUDE.md Konvention).
    Returns user_info dict.
    """
    user_id   = user.id
    user_org  = user.org_id
    user_rol  = user.rolle
    user_coa  = bool(getattr(user, 'is_coach', False))
    user_onb  = bool(getattr(user, 'onboarding_done', True))

    session.permanent   = True
    session['user_id']  = user_id
    session['org_id']   = user_org
    session['rolle']    = user_rol
    session['is_coach'] = user_coa

    tok    = secrets.token_urlsafe(32)
    ablauf = datetime.now(timezone.utc).replace(tzinfo=None) + timedelta(hours=MAX_SESSION_HOURS)
    db.add(DbSession(user_id=user_id, token=tok, ablauf_am=ablauf))
    db.commit()

    return {
        'id': user_id, 'org_id': user_org, 'rolle': user_rol,
        'is_coach': user_coa, 'onboarding_done': user_onb,
    }


def _do_login(email, passwort):
    """Unverändert nach außen — ruft intern _login_user."""
    db = get_session()
    try:
        user = db.query(User).filter_by(email=email, aktiv=True).first()
        if user and user.passwort_hash and check_password_hash(user.passwort_hash, passwort):
            return _login_user(db, user), None
        return None, 'E-Mail oder Passwort falsch.'
    finally:
        db.close()
```

**Kritisch:** Die `and user.passwort_hash` Prüfung blockiert Email-Login für OAuth-only User automatisch (leerer String → falsy).

### Pattern 3: Register-Helper Extraktion

```python
# routes/auth.py
def _create_org_and_user(email, vorname, nachname, firmenname,
                         passwort_hash=None, oauth_provider=None,
                         oauth_id=None, avatar_url=None,
                         teamgroesse='1-5'):
    """Erstellt Org + User. Wird von api_register UND OAuth-Callback genutzt.
    Returns: (user_object, org_object) — Caller muss commit + login auslösen.
    Caller behält db-Session-Management.
    """
    from config import PLANS
    size_to_plan = {'1-5': 'starter', '6-15': 'starter',
                    '16-30': 'business', '30+': 'business'}
    plan      = size_to_plan.get(teamgroesse, 'starter')
    max_users = PLANS[plan]['max_users']

    now       = datetime.now(timezone.utc).replace(tzinfo=None)
    trial_end = now + timedelta(days=14)

    org = Organisation(
        name=firmenname or f"{vorname or 'Neue'} Organisation",
        plan=plan, max_users=max_users,
        billing_email=email, trial_starts_at=now,
    )
    # DB-Flush + User-Create im Caller (Transaction-Kontrolle)
    return org, dict(
        email=email, passwort_hash=(passwort_hash or ''),
        rolle='owner', is_trial=True, trial_ends_at=trial_end,
        vorname=vorname, nachname=nachname,
        onboarding_done=False,  # OAuth-User MUSS durch Wizard → Firmenname
        oauth_provider=oauth_provider,
        oauth_id=oauth_id,
        avatar_url=avatar_url,
    )
```

`api_register` und OAuth-Callback rufen diesen Helper auf und managen `db.flush/commit/close` selbst.

### Pattern 4: OAuth-Callback Dispatch (geteilt für Google + MS)

```python
def _oauth_login_or_create(provider, oauth_id, email, vorname, nachname, avatar_url):
    db = get_session()
    try:
        user = db.query(User).filter_by(email=email).first()
        if user:
            # Bestehender Account — OAuth-Felder nachziehen (idempotent)
            changed = False
            if not user.oauth_provider:
                user.oauth_provider = provider
                user.oauth_id       = oauth_id
                changed = True
            if avatar_url and user.avatar_url != avatar_url:
                user.avatar_url = avatar_url
                changed = True
            if changed:
                db.commit()
            user_info = _login_user(db, user)
        else:
            # Neuer User — Org mit Firmenname-Platzhalter anlegen,
            # onboarding_done=False → Wizard fragt Firmenname nach
            org, user_kwargs = _create_org_and_user(
                email=email, vorname=vorname, nachname=nachname,
                firmenname=f"{vorname} {nachname}".strip() or email.split('@')[0],
                passwort_hash='', oauth_provider=provider,
                oauth_id=oauth_id, avatar_url=avatar_url,
            )
            db.add(org)
            db.flush()
            user = User(org_id=org.id, **user_kwargs)
            db.add(user)
            db.flush()
            db.commit()
            user_info = _login_user(db, user)

        # Onboarding-Check wie login_required
        if not user_info['onboarding_done']:
            return redirect(url_for('onboarding.wizard'))
        return redirect(url_for('dashboard.index'))
    finally:
        db.close()
```

### Anti-Patterns to Avoid
- **OAuth-Client in `__init__.py` oder App-Factory-Global registrieren** ohne `init_app(app)` → `SECRET_KEY` noch nicht gesetzt → State-Speicherung bricht. Immer `oauth.init_app(app)` **nach** App-Konfig aufrufen.
- **`token['userinfo']` ohne Fallback** — bei älteren authlib-Versionen muss `oauth.google.parse_id_token(token, nonce=...)` oder `oauth.google.userinfo(token=token)` als Fallback dienen. Version ≥1.3 liefert `userinfo` automatisch wenn Scope `openid` gesetzt. [CITED: authlib changelog 1.2+]
- **Email-Case-Sensitivity ignorieren** — immer `.strip().lower()` vor DB-Query (wie `api_register` es macht).
- **Session-Fixation:** Bei Provider-Wechsel alte Session nicht clearen — `session.clear()` vor `_login_user` wäre defensiv, aber: der bestehende Code cleart auch nicht, also konsistent bleiben.

## Don't Hand-Roll

| Problem | Nicht bauen | Stattdessen | Warum |
|---------|-------------|-------------|-------|
| OAuth 2.0 Authorization Code Flow | Eigener `requests.post` auf Token-Endpoint | `authlib.integrations.flask_client` | PKCE, State, Nonce, ID-Token-Signaturprüfung, JWKS-Rotation |
| ID-Token JWT-Validierung | `jwt.decode` mit hardcoded Key | `authlib` parst ID-Token aus `userinfo` im Token-Dict automatisch | JWKS wechseln, Signatur-Algo, Claims-Validation |
| Google/MS Endpoint-URLs pflegen | Hardcoded URLs | `server_metadata_url` Discovery | Provider ändern Endpoints, Discovery ist robust |
| Userinfo Fetch (2. HTTP Call) | `requests.get(userinfo_endpoint, token)` | ID-Token liefert alle Claims (`email`, `name`, `picture`, `sub`) | Ein Roundtrip weniger, ohnehin signiert |
| State/CSRF-Token | `secrets.token_urlsafe` + Session | authlib macht das | Schon fehleranfällig, Standard-Lösung nutzen |

**Key insight:** OAuth ist Security-Critical. Jede Zeile eigener Token-Handling-Code ist eine potenzielle CVE. authlib ist maintained, audited und wird breit verwendet. Nur Callback-Dispatch + DB-Logik selber schreiben.

## Common Pitfalls

### Pitfall 1: Microsoft `email_verified` fehlt im ID-Token
**Was passiert:** Google setzt `email_verified: true` im ID-Token zuverlässig. Microsoft tut das bei Work-Accounts oft **nicht** (Feld fehlt komplett). Wenn Callback `if not userinfo.get('email_verified')` prüft, würde jeder MS-Login abgelehnt.
**Warum:** Microsoft geht davon aus, dass Work-Emails durch Azure AD verifiziert sind; Personal-Accounts (@outlook.com) liefern `xms_edov` oder gar nichts.
**Vermeidung:** Für Google `email_verified == True` verlangen. Für Microsoft Prüfung überspringen — Azure AD Auth selbst ist die Verifikation. Im Callback getrennte Logik:
```python
if provider == 'google' and not userinfo.get('email_verified'):
    flash('Google-Email nicht verifiziert.', 'error')
    return redirect('/?modal=login')
# Microsoft: keine Prüfung — Azure AD Auth ist die Verifikation
```
[CITED: learn.microsoft.com/en-us/azure/active-directory/develop/id-token-claims-reference]

### Pitfall 2: Microsoft `common` vs. `consumers` vs. `organizations`
**Was passiert:** Der Tenant-Segment in der URL entscheidet, wer sich einloggen darf.
- `common` → Work + Personal (Azure AD + Microsoft Account) ← **hier richtig**
- `organizations` → nur Work/School (Azure AD)
- `consumers` → nur Personal
- `{tenant-id}` → nur spezifischer Tenant
**Empfehlung:** `common` — maximale Reichweite (User mit Firmen-Office365 und User mit persönlichem outlook.com können sich einloggen). App muss in Azure Portal als „Accounts in any organizational directory and personal Microsoft accounts" registriert sein.
[CITED: learn.microsoft.com/en-us/azure/active-directory/develop/v2-protocols — "The /common endpoint"]

### Pitfall 3: SQLite kann `passwort_hash NOT NULL` nicht in-place ändern
**Was passiert:** `ALTER TABLE users ALTER COLUMN passwort_hash DROP NOT NULL` ist **kein gültiges SQLite SQL**. SQLite kennt nur `ALTER TABLE ADD COLUMN` und `ALTER TABLE RENAME`. Der saubere Weg wäre:
1. `CREATE TABLE users_new (...)` mit `passwort_hash` nullable
2. `INSERT INTO users_new SELECT * FROM users`
3. `DROP TABLE users`
4. `ALTER TABLE users_new RENAME TO users`
5. Foreign-Keys + Indizes neu anlegen

Das ist **riskant** in einem Prod-System mit Live-Daten und bricht den bestehenden `[DB] Migration:` Pattern in `app.py`.

**Pragmatische Lösung (empfohlen):** SQLite **ignoriert `NOT NULL` praktisch nicht** — aber man kann den Constraint im **ORM** (`Column(String(256), nullable=True)`) ändern, und SQLite akzeptiert weiterhin Inserts mit leerem String `''` ohne Fehler (empty string ≠ NULL). Für neue OAuth-User einfach `passwort_hash=''` setzen.
- Das ORM-Model wird geändert: `passwort_hash = Column(String(256), nullable=True)` — das betrifft nur SQLAlchemy's Validation, nicht das physische Schema.
- Neue User via OAuth bekommen `passwort_hash=''`
- `_do_login` prüft `if user.passwort_hash and check_password_hash(...)` — leerer String → False → korrektes Verhalten

**Wenn der Planner echtes `NULL` will:** Tabellen-Rebuild in eigener Migration (separate Funktion in `app.py`, geschützt via Idempotenz-Check `PRAGMA table_info(users)` → prüft, ob `passwort_hash` notnull ist). Zusatzrisiko rechtfertigt das m.E. nicht.

**Entscheidung:** Der Planner soll die pragmatische Lösung wählen — leerer String als Sentinel. Es funktioniert, es ist rückwärtskompatibel, es passt ins bestehende Migration-Pattern (nur `ADD COLUMN` für die 3 neuen OAuth-Felder).

[VERIFIED: sqlite.org/lang_altertable.html — ALTER TABLE kann weder DROP COLUMN noch ALTER COLUMN, nur ADD COLUMN + RENAME]

### Pitfall 4: `avatar_url` String(500) zu kurz für Google Photo URLs
**Was passiert:** Google liefert `picture` als `https://lh3.googleusercontent.com/a/ACg8oc...=s96-c` — meist < 200 Zeichen, aber bei gewissen Tokens >500.
**Vermeidung:** `String(500)` ist ausreichend laut Google-Docs, aber defensiv könnte man `String(1000)` nehmen. Für NERVE: `500` reicht, bei Überlauf Exception → im Callback `try/except` und Avatar-URL einfach auf `None` setzen statt abzubrechen.

### Pitfall 5: `/auth/google` kollidiert mit `auth_bp`?
**Was passiert:** `auth_bp` hat `/login`, `/logout`, `/api/login`, `/api/register`, `/register`. `oauth_bp` bringt `/auth/google`, `/auth/microsoft`. **Keine Kollision** — `auth_bp` hat kein Prefix und keine `/auth/*` Routes. Safe.

### Pitfall 6: OAuth-Callback vor `init_oauth(app)` aufgerufen
**Was passiert:** Wenn `oauth.register(...)` in `routes/oauth.py` Modul-Import-Zeit passiert, ist `app` noch nicht da. authlib braucht `init_app(app)` für State-Storage via Flask-Session.
**Vermeidung:** `init_oauth(app)` als explizite Funktion, in `app.py` **nach** `app.register_blueprint(oauth_bp)` aufrufen. Import-Zeit nur `oauth = OAuth()` und `oauth_bp = Blueprint(...)`.

### Pitfall 7: Redirect URI Mismatch (localhost vs. prod)
**Was passiert:** Google/MS validieren die Redirect URI byte-exact. `url_for('oauth.google_callback', _external=True)` baut sie aus `request.host_url` — in Prod hinter nginx muss `X-Forwarded-Proto: https` gesetzt sein, sonst erzeugt Flask `http://app.getnerve.app/...` statt `https://...`.
**Vermeidung:** nginx setzt bereits `X-Forwarded-Proto` (siehe Phase 03). Zusätzlich `ProxyFix` in `app.py` aktivieren falls noch nicht:
```python
from werkzeug.middleware.proxy_fix import ProxyFix
app.wsgi_app = ProxyFix(app.wsgi_app, x_proto=1, x_host=1)
```
Planner soll prüfen, ob `ProxyFix` bereits da ist (Grep in `app.py`). [CITED: werkzeug.palletsprojects.com/en/3.0.x/middleware/proxy_fix/]

### Pitfall 8: `authlib` State in Flask-Session → Session-Cookie Size
**Was passiert:** authlib speichert State + Nonce in `flask.session`. Bei `SESSION_COOKIE` (default) wird das in einen signierten Cookie geschrieben. Normalerweise kein Problem, aber falls Cookie-Size eng wird, kann State verloren gehen.
**Vermeidung:** Standard-Flask-Session reicht — kein Server-Side-Session-Store nötig für den OAuth-Flow.

## State of the Art

| Alte Approach | Aktuelle Approach | Geändert | Impact |
|---------------|-------------------|----------|--------|
| `requests-oauthlib` + manuelles ID-Token | `authlib` mit `server_metadata_url` | ~2022 | Weniger Code, sicherer |
| 2 separate HTTP-Calls (Token + Userinfo) | ID-Token enthält Userinfo via `openid email profile` Scope | seit OIDC Core | 1 Call weniger |
| Hardcoded Endpoints | Discovery (`.well-known/openid-configuration`) | seit OIDC | Provider kann Endpoints rotieren |
| Microsoft ADAL / v1 Endpoint | Microsoft Identity Platform v2 (`/common/v2.0/`) | seit 2019, ADAL deprecated 2022 | v1 ist deprecated — **nicht verwenden** |

**Deprecated:**
- Microsoft ADAL-Library → ersetzt durch `msal` oder (besser für Flask) `authlib` + v2 Endpoints
- `oauth.google.parse_id_token(token)` ohne `nonce` → authlib 1.0+ erfordert Nonce; `token['userinfo']` ist der neue Weg

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Gilt | Standard Control |
|---------------|------|-----------------|
| V2 Authentication | **ja** | authlib (OIDC), werkzeug `check_password_hash` bestehend |
| V3 Session Management | ja | Flask session (signed cookie), `session.permanent`, `DbSession` Model |
| V4 Access Control | ja (bestehend) | `login_required` Decorator, `rolle`-Check |
| V5 Input Validation | ja | Email `.strip().lower()`, authlib validiert ID-Token Claims |
| V6 Cryptography | ja | authlib JWKS + RS256 ID-Token-Signatur; `SECRET_KEY` für Flask-Session |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| CSRF auf OAuth-Callback | Tampering | authlib `state` Parameter (automatisch) |
| Replay-Attack auf ID-Token | Spoofing | authlib Nonce-Check (automatisch) |
| Account-Takeover via unverified Email (Google) | Spoofing | `email_verified` Claim prüfen (Google) |
| Session-Fixation | Elevation | Flask session signed + neuer `DbSession` Token bei Login |
| Open Redirect nach Login | Tampering | Hardcoded Redirects (`url_for('dashboard.index')`), kein `next`-Param |
| Redirect URI Injection | Tampering | Google/MS validieren Redirect URI gegen registrierte Whitelist |
| Dummy-Passwort-Hash `''` ermöglicht Login | Spoofing | `if user.passwort_hash and check_password_hash(...)` — leerer String ist falsy |

## Open Questions

1. **SQLite `passwort_hash=''` Insert-Verhalten in der Prod-DB**
   - Was wir wissen: SQLite setzt `NOT NULL` auf physisches Schema durch; leerer String ≠ NULL, also sollte Insert klappen
   - Was unklar: Verhält sich die konkrete `nerve.db` so wie dokumentiert?
   - Empfehlung: Planner fügt in Wave 0 einen 5-Zeilen-Migration-Check hinzu, der vor allem anderen einen Dummy-Insert testet und bei Fehler ein `PRAGMA table_info(users)` + Warning loggt.

2. **Azure App Registration: ist `supportedAccountTypes` bereits auf `AzureADandPersonalMicrosoftAccount` gesetzt?**
   - Was wir wissen: Redirect URI ist registriert
   - Was unklar: Tenant-Konfiguration
   - Empfehlung: Planner schreibt in den Plan einen manuellen Prereq-Check — User muss im Azure Portal bestätigen, sonst `common` Endpoint → Fehler bei Personal-Accounts.

3. **Fehler-UX bei OAuth-Abbruch**
   - Was wir wissen: `flash()` + `redirect('/?modal=login')` existiert im Codebase
   - Was unklar: Zeigt das Login-Modal Flash-Messages sichtbar? (Modal wird client-side geöffnet, Flash kommt aus Server-Template)
   - Empfehlung: Planner verifiziert, ob `landing.html` `{% with messages = get_flashed_messages(...) %}` rendert. Falls nicht → Query-Param `?oauth_error=google_cancelled` als Fallback und JS liest ihn aus.
