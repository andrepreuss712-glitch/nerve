# Phase 04.6.1 Plan 01: OAuth DB-Schema + Auth-Helpers Summary

**One-liner:** User-Tabelle bekommt OAuth-Felder (oauth_provider/oauth_id/avatar_url), `_do_login` wird gesplittet in `_login_user` + Email-Pfad, Register-Logik wandert in `_create_org_and_user` Helper für Wiederverwendung im OAuth-Flow.

## What Was Built

### Task 1: User-Model + DB-Migration
- `database/models.py`: `passwort_hash` ist jetzt `nullable=True` (mit Inline-Kommentar zur SQLite-Sentinel-Strategie). Block 13 hinzugefügt mit `oauth_provider VARCHAR(50)`, `oauth_id VARCHAR(200)`, `avatar_url VARCHAR(500)`.
- `app.py` `_migrate()`: Drei neue ALTER TABLE Statements im users-Block (Block 13). Migration läuft idempotent (try/except/pass Pattern).
- Verifiziert: `[DB] Migration: added users.oauth_provider/oauth_id/avatar_url` einmalig im stdout, Spalten via `inspect(engine).get_columns('users')` bestätigt.
- **Commit:** `4e825dc`

### Task 2: routes/auth.py Refactor
- **`_login_user(db, user)`** — neuer Helper. Setzt Flask-Session + DbSession für ein bereits validiertes User-Objekt. Liest alle User-Attribute VOR `db.commit()` (CLAUDE.md Konvention). Returns user_info dict.
- **`_do_login(email, passwort)`** — auf Email/Passwort-Pfad reduziert. Lehnt User mit leerem/None `passwort_hash` mit Standard-Errortext ab (T-04.6.1-01 Mitigation). Ruft `_login_user` nach erfolgreichem `check_password_hash`.
- **`_create_org_and_user(db, *, ...)`** — neuer Helper. Legt Org + User in derselben DB-Session an, ohne `db.commit()` (Caller-Verantwortung). Akzeptiert optional `oauth_provider`, `oauth_id`, `avatar_url`. `passwort_hash=None` → leerer String als Sentinel.
- **`api_register`** — refactored auf Helper-Aufrufe: Validierung → `_create_org_and_user` → `db.commit()` → `_login_user` → `session['welcome_trial']=True`. Verhalten 1:1 erhalten.
- Verifiziert: `inspect.signature(_create_org_and_user)` enthält erwartete Parameter inkl. `oauth_provider`.
- **Commit:** `fd7f1f9`

## Threat Mitigations Applied

| Threat ID | Mitigation |
|-----------|------------|
| T-04.6.1-01 (Spoofing) | `_do_login` lehnt leeren `passwort_hash` sofort ab — OAuth-Only-User können kein Email-Login bypassen |
| T-04.6.1-02 (Tampering) | ALTER TABLE in try/except/pass — idempotent |
| T-04.6.1-03 (Information Disclosure) | Identischer Errortext bei "User existiert nicht", "kein Hash", "Passwort falsch" |
| T-04.6.1-04 (Elevation of Privilege) | `_create_org_and_user` hardcodet `rolle='owner'` nur für die NEU erstellte Org |
