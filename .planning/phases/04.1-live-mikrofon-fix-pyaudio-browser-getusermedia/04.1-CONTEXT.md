# Phase 4.1: Live-Mikrofon Fix: PyAudio → Browser getUserMedia — Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the server-side PyAudio mic capture in the live session with browser-side `getUserMedia` + Socket.IO streaming, so the live transcription feature works on the VPS (no audio hardware).

**What this phase delivers:** Transcripts flow on `getnerve.app` for real calls.
**What this phase does NOT deliver:** Cold-Call vs Meeting Mode UX, DSGVO consent flow for meeting mode (Phase 4 LEGAL-01), mode selection UI, fair-use metering, legal pages — all stay in Phase 4.

</domain>

<decisions>
## Implementation Decisions

### Audio Pipeline

- **D-01:** Use **AudioWorkletNode → PCM16 → Socket.IO** for browser-to-server audio streaming.
  - Browser: `getUserMedia` → `AudioContext` (sampleRate: 16000) → `MediaStreamSource` → `AudioWorkletNode` → emit `audio_chunk` (Int16Array buffer) via Socket.IO
  - New static file required: `static/audio-processor.js` (AudioWorklet module, ~15 lines)
  - Keeps existing Deepgram config (`encoding=linear16`, `sample_rate=16000`) — no server-side encoding change
  - MediaRecorder rejected: variable chunk timing, unreliable for Deepgram live streaming
  - ScriptProcessorNode rejected: deprecated
  - Emit every ~100ms (1600 samples at 16kHz = 3200 bytes)

### Session Architecture

- **D-02:** **Per-Socket.IO session** Deepgram connections. Each `sid` gets its own `DeepgramClient` + live connection.
  - Connection lifecycle: open on `start_live_session` socket event (when user clicks "Start"), close on `stop_live_session` or socket disconnect
  - Server maintains a dict: `_deepgram_sessions: dict[sid, connection]`
  - Remove the global `deepgram_starten()` call at server startup (app.py:683-686) — replaced by per-session handlers
  - Transcript/coaching events emit to the specific `sid` room (not broadcast to all)
  - `services/deepgram_service.py` needs refactoring: `on_message` must accept `sid` parameter to emit to correct client

### Diarization

- **D-03:** **Keep Deepgram diarization** (`diarize=True`). Same options as current (`nova-2`, `language=de`, diarize, smart_format, interim_results, endpointing=900, punctuate).
  - v1 use case: classic phone call — user has headset for phone, browser open next to phone (phone on speakerphone). Computer mic captures both voices.
  - Speaker 0 = Berater, Speaker 1 = Kunde — existing role mapping in `live_session.py` stays unchanged.

### Scope Boundary with Phase 4

- **D-04:** Phase 4.1 is the **minimal technical fix only**. DSGVO banner for Meeting Mode, Cold-Call vs Meeting Mode selection, and LEGAL-01 stay in Phase 4.
  - The existing DSGVO banner (triggered on socket connect, from Phase 2) remains as-is for now.
  - No new UI beyond what's needed to start/stop the browser mic stream.

### Claude's Discretion

- Socket event names: `audio_chunk` (browser → server), `start_live_session` / `stop_live_session` (lifecycle), existing `transcript` / `coaching` (server → browser, unchanged)
- Exact chunk size / emit interval (recommend 100ms / 3200 bytes)
- How to store per-session Deepgram connections on the server (module-level dict keyed by sid)
- Error handling for `getUserMedia` permission denial (show existing DSGVO/mic permission UI)
- Whether to register Socket.IO event handlers in `services/deepgram_service.py` or in a new `routes/live_audio.py`

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `services/live_session.py`: All session state (`conversation_log`, `pause_lock`, `is_paused`, speaker tracking, merge logic) is reusable unchanged — only the audio input source changes
- `services/deepgram_service.py:on_message`: The transcript processing logic is correct and reusable — only needs `sid` parameter added to target the right Socket.IO room
- `templates/training.html:808-816`: `getUserMedia` permission flow pattern (can copy the permission error handling)

### Established Patterns
- Socket.IO `async_mode='threading'` (app.py:29) — threading mode, not eventlet/gevent. Per-session dict must be thread-safe (use threading.Lock or the GIL is sufficient for dict operations)
- Services emit via `extensions.socketio.emit()` with a room/sid target
- Pause state lives in `services/live_session.py` — browser mic must respect `ls.is_paused` (stop sending `audio_chunk` events when paused, or server ignores chunks when paused)

### Integration Points
- `app.py:683-686`: Remove `deepgram_starten()` startup call
- `static/app.js`: Add AudioWorklet setup + Socket.IO `audio_chunk` emission on session start
- New `static/audio-processor.js`: AudioWorklet module (convert Float32 → Int16, post to main thread)
- Socket.IO event handlers: `start_live_session`, `stop_live_session`, `audio_chunk` — register in deepgram_service or a new live_audio route
