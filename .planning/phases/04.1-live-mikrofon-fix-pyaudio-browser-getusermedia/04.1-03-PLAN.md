<investigation_notes>
## Root Cause Analysis

**Gap 1 — No transcript (CRITICAL):**

The primary root cause is **AudioContext suspension due to Chrome's autoplay policy**.

`startMicStream()` is called from `socket.on('connect')`, which fires automatically when the page loads — no user gesture involved. Chrome's autoplay policy suspends any `AudioContext` created without a user interaction. The AudioContext is created in "suspended" state, the AudioWorklet node is set up, but no audio actually flows through the pipeline. `getUserMedia` succeeds (mic permission prompt IS a user gesture), but by the time the permission is granted and the async function continues, the AudioContext was already created in the catch-up flow and remains suspended.

**Fix:** Add `await audioCtx.resume()` immediately after `new AudioContext({ sampleRate: 16000 })`. The `getUserMedia` permission grant counts as a user gesture, so `resume()` will succeed. Also log `audioCtx.state` for diagnostics.

**Gap 2 — Training mic prompt (SIDE-EFFECT — NOT REPRODUCIBLE):**

Investigation shows `static/app.js` is ONLY loaded in `templates/app.html` (the /live page). Training.html does NOT load Socket.IO or app.js. Training has its own `getUserMedia` call triggered by the push-to-talk button (user gesture). The reported "training mic prompt side-effect" is likely the normal PTT mic permission request, not a leak from app.js. No code change needed for Gap 2.

However, as a defensive measure, the plan adds a `window.location.pathname` guard in the connect handler. If app.js ever gets loaded on a non-/live page in the future, this prevents unwanted mic access.
</investigation_notes>
