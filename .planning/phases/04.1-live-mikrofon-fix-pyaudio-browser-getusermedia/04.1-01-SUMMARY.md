# Phase 4.1 Plan 01: PyAudio Removal — Per-Session Deepgram via Browser getUserMedia Summary

**One-liner:** Replaced global PyAudio-based Deepgram stream with per-Socket.IO-session Deepgram WebSocket connections forwarding browser audio chunks, enabling VPS deployment without audio hardware.

## What Was Built

`services/deepgram_service.py` was fully rewritten from a single global PyAudio capture loop to a per-session architecture:

- `_deepgram_sessions = {}` — dict mapping sid to active Deepgram connection
- `_make_on_message(sid)` — factory closure preserving all speaker tracking, merge window, conversation_log, and emit logic with `room=sid` targeting
- `_make_on_open(sid)`, `_make_on_error(sid)` — per-session lifecycle logging and error forwarding
- `_open_deepgram_connection(sid)` — creates DeepgramClient, registers handlers, starts with nova-2/de/diarize/linear16 config
- `_close_deepgram_connection(sid)` — calls `connection.finish()`, removes from dict
- `register_audio_handlers(sio)` — registers 4 Socket.IO event handlers: `start_live_session`, `stop_live_session`, `audio_chunk`, `disconnect`

`app.py` was updated to call `register_audio_handlers(socketio)` instead of spawning a `deepgram_starten` background thread.
