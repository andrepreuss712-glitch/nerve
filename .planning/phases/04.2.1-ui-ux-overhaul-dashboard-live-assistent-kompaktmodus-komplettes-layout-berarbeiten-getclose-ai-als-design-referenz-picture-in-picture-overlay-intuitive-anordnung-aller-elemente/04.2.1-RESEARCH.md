## Summary

Phase 04.2.1 is a pure layout restructure across three surfaces: the `/live` Kompaktmodus (replacing the existing floating `.kompakt-panel` overlay with a Document Picture-in-Picture window), the `/live` Vollbild (reducing 3 chrome bars to 1 single 52px header), and `/dashboard` (replacing SVG stat circles, heatmap, and quote-of-day with a North Star metric, usage progress bars, recent calls list, and Next Best Action card).

The existing codebase is well-prepared. The kompakt toggle logic, EWB rendering, timer sync, and usage data are already in place — they need restructuring and extension, not replacement. The Document PiP API is a native browser API (Chrome 116+) requiring no library. The key technical challenge is PiP stylesheet injection (manual copy required, `copyStyleSheets` option was removed from the spec) and Socket.IO event listener re-attachment after DOM node movement into the PiP window.

The dashboard route (`routes/dashboard.py`) already exposes all data needed: `usage` dict (minutes/sessions used/limit), `stats` dict (avg_kb, einwand_erfolg, trend_score), `recent_logs`, `streak`, and `level_info`. North Star score can be derived from `stats.avg_kb` or `stats.einwand_erfolg`. Next Best Action logic can reuse `_generate_improvement_tip()` logic. The only missing data point is a per-session `session_mode` label in `recent_logs` — that field exists in `ConversationLog` from Phase 4.2 but is not currently passed to the dashboard template.

**Primary recommendation:** Implement in three sequential waves: (1) sidebar collapse, (2) dashboard restructure, (3) PiP Kompaktmodus + Vollbild chrome reduction.

### Pattern 1: Document PiP Initialization
**What:** Open a real browser window via Document PiP API, inject styles, move DOM node.
**When to use:** Replacing the existing `toggleKompakt()` + `body.kompakt` pattern.

```javascript
// Source: https://developer.chrome.com/docs/web-platform/document-picture-in-picture
async function openPipWindow() {
  if (!window.documentPictureInPicture) {
    // Fallback: existing .kompakt-panel overlay
    document.body.classList.add('kompakt');
    showPipFallbackWarning();
    return;
  }

  const pipWindow = await window.documentPictureInPicture.requestWindow({
    width: 380,
    height: 280,
  });

  // Copy stylesheets (copyStyleSheets option was removed from spec)
  [...document.styleSheets].forEach((styleSheet) => {
    try {
      const cssRules = [...styleSheet.cssRules].map((rule) => rule.cssText).join('');
      const style = document.createElement('style');
      style.textContent = cssRules;
      pipWindow.document.head.appendChild(style);
    } catch (e) {
      const link = document.createElement('link');
      link.rel = 'stylesheet';
      link.href = styleSheet.href;
      pipWindow.document.head.appendChild(link);
    }
  });

  // Move PiP content element into PiP window
  const pipContent = document.getElementById('pip-window');
  pipWindow.document.body.appendChild(pipContent);

  // Detect PiP closure — restore Vollbild
  pipWindow.addEventListener('pagehide', () => {
    document.getElementById('main-content').appendChild(pipContent);
    try { localStorage.setItem('sn_kompakt', '0'); } catch(e) {}
  });

  try { localStorage.setItem('sn_kompakt', '1'); } catch(e) {}
  window._pipWindow = pipWindow;
}
```

**Key insight:** Socket.IO in the PiP window does NOT need re-initialization. The PiP window shares the same JS execution context as the opener window. Socket events registered in the main window fire in both contexts — the DOM update functions just need to target elements by ID, which now live in the PiP document. The approach: event handlers use `document.getElementById(id) || (window._pipWindow && window._pipWindow.document.getElementById(id))` to find elements in either document.

### Pattern 2: PiP Tab State Machine
**What:** Auto-switching tabs driven by KI, with manual lock override.
**When to use:** All four tabs (Opener / Einwand / Coaching / EWB) in the PiP window.

```javascript
// State variable — null = auto-mode, 'opener'|'einwand'|'coaching'|'ewb' = locked
window._pipTabLocked = null;
let _lastKiResultTime = 0;

function setPipTab(tabName, fromKI = false) {
  if (fromKI && window._pipTabLocked) {
    // KI wants to switch — check if lock should expire (30s timeout)
    if (Date.now() - _lastKiResultTime > 30000) {
      window._pipTabLocked = null;
    } else {
      // Update content but don't switch tab
      updatePipContent(tabName);
      return;
    }
  }
  _lastKiResultTime = Date.now();
  // Set active tab, update lock indicator
  activatePipTab(tabName, !!window._pipTabLocked);
}

function handlePipTabClick(tabName) {
  if (window._pipTabLocked === tabName) {
    // Click on locked tab = unlock
    window._pipTabLocked = null;
  } else {
    window._pipTabLocked = tabName;
  }
  activatePipTab(tabName, !!window._pipTabLocked);
}
```

### Anti-Patterns to Avoid

- **Do NOT re-initialize Socket.IO in the PiP window.** The PiP window shares the same JS execution context. Re-initializing creates duplicate event handlers and duplicate Deepgram connections. Instead, use a helper function that searches both `document` and `window._pipWindow.document` for elements.
- **Do NOT use `copyStyleSheets: true` option in `requestWindow()`.** This option was removed from the spec. Use manual stylesheet copying via the loop pattern above.
- **Do NOT move the `<body>` or `<html>` elements into PiP.** Move only the `#pip-window` div. The PiP window gets its own empty `document`.
- **Do NOT hide main content when PiP is open.** Vollbild remains fully functional as a parallel view when PiP is open. Only the `body.kompakt` CSS hide-rules pattern (from Phase 3.2) is replaced.
- **Do NOT delete hidden nav items.** Per D-31 decision, hide via `display:none`, keep in DOM.

## Common Pitfalls

### Pitfall 1: PiP Stylesheet CORS Block
**What goes wrong:** The stylesheet copy loop throws a CORS SecurityError when trying to read `.cssRules` from a cross-origin `<link rel="stylesheet">` (e.g., Google Fonts CDN).
**Why it happens:** Chrome blocks cssRules access on cross-origin sheets. Google Fonts and Lucide CDN links are cross-origin.
**How to avoid:** Wrap each styleSheet access in `try/catch`. On CORS error, fall back to `<link>` element copy (the `link.href = styleSheet.href` branch in the pattern above). Google Fonts and Lucide load fine as `<link>` in the PiP window — the PiP window can fetch external resources.
**Warning signs:** PiP window appears unstyled (no CSS tokens applied). Check console for SecurityError.

### Pitfall 2: Socket.IO Double-Attach
**What goes wrong:** If `openPipWindow()` is called multiple times (user opens/closes PiP repeatedly), event handlers accumulate.
**Why it happens:** Socket.IO `.on()` calls stack without automatic deduplication.
**How to avoid:** Guard `openPipWindow()` with a check: `if (window._pipWindow && !window._pipWindow.closed) return;`. On `pagehide`, set `window._pipWindow = null`.

### Pitfall 3: DOM Node Gone from Main Document
**What goes wrong:** After moving `#pip-window` div into the PiP document, JS in the main page fails to find it via `document.getElementById('pip-window')`.
**Why it happens:** `appendChild` to PiP document body moves (not copies) the node — it leaves the main document.
**How to avoid:** All functions that update PiP content must use a helper: `function getPipElement(id) { return window._pipWindow?.document.getElementById(id) || document.getElementById(id); }`. On `pagehide`, move node back to main document before `_pipWindow = null`.

### Pitfall 4: Dashboard North Star — No "Weekly Score" Field
**What goes wrong:** There is no pre-computed "weekly score" field in the existing `stats` dict.
**Why it happens:** `dashboard.py` computes `avg_kb` (Kaufbereitschaft average over 30 days), not a composite weekly score.
**How to avoid:** Two options: (A) Use `stats.avg_kb` directly as the North Star number (simple, accurate for existing data), or (B) compute a composite (avg_kb 50% + einwand_erfolg 50%) in the route. Option A recommended — simpler, dashboard.py already has trend_score for the trend indicator.

### Pitfall 5: `session_mode` Missing from Recent Logs
**What goes wrong:** The recent calls list should show "Cold Call" or "Meeting" type per D-23, but `get_recent_logs()` in `dashboard.py` parses log files — not the DB — and does not include `session_mode`.
**Why it happens:** `get_recent_logs()` reads from `.txt` files in `LOG_DIR`, not from `ConversationLog` DB records. `session_mode` is stored in `ConversationLog.session_mode` (added in Phase 4.2) but not exposed here.
**How to avoid:** Two options: (A) Switch `get_recent_logs()` from file-based to DB-based using `ConversationLog` (preferred — gives direct access to `session_mode`, score, and structured data), or (B) keep file-based but add a DB lookup to enrich results with `session_mode`. Option A is cleaner.

### Pitfall 6: PiP Window Width Too Narrow at 380px
**What goes wrong:** Four pill tabs at 380px may be very tight — each tab label ("Einwand", "Coaching") needs ~70-80px minimum.
**Why it happens:** 380px minus 2×16px padding = 348px for 4 tabs = ~87px per tab. Tight but feasible.
**How to avoid:** Use `font-size: 12px` for tab labels, `padding: 6px 8px` (not the standard pill padding of 8px 20px). Verify visually. If too tight, increase initial PiP width to 420px (Claude's Discretion on start size).

### Pitfall 7: `g.org.plan` vs `g.org.plan_typ`
**What goes wrong:** Plan badge reads wrong field — shows `bundle` or `null` instead of `Starter`/`Pro`.
**Why it happens:** `Organisation` model has two plan fields: `plan` (starter/team/business/enterprise) and `plan_typ` (training/live/bundle/coach). The `PLANS` dict in `app.py` uses the `plan` field for display names.
**How to avoid:** Read `g.org.plan` for the plan badge. Map to display name via `PLANS[g.org.plan].name` in the Jinja template: `{{ plans[g.org.plan].name if g.org.plan else 'Starter' }}`. The `PLANS` dict needs to be passed to `base.html` render context or accessed via a template global.

## Open Questions

1. **North Star Metric — composite vs. avg_kb only**
   - What we know: `stats.avg_kb` (30-day Kaufbereitschaft average) and `stats.einwand_erfolg` both available.
   - What's unclear: Is a single number or composite more useful as the "North Star"?
   - Recommendation: Use `stats.avg_kb` alone in this phase (layout only). Score composition is a logic decision for a later phase.

2. **`opener_complete` socket event — does it exist yet?**
   - What we know: D-10 specifies a `opener_complete` socket event from Claude Haiku analysis. This event does NOT currently exist in the codebase — `services/claude_service.py` does not emit it.
   - What's unclear: Is this event implemented as part of this phase, or does the Opener tab simply stay active until manual tab click?
   - Recommendation: Implement Opener tab as "active by default, user switches manually" for this phase. Leave `opener_complete` socket integration as a follow-up. The tab is functional; auto-switching from Opener to Einwand based on KI detection is Phase 04.2.2 territory.

3. **`session_mode` in recent calls list**
   - What we know: `get_recent_logs()` reads `.txt` files; `ConversationLog.session_mode` exists in DB.
   - What's unclear: Whether to refactor `get_recent_logs()` to DB-based, or enrich file-based results.
   - Recommendation: Switch the recent calls query in `dashboard.py` to `ConversationLog` DB records for the top 5 (direct query, includes `session_mode`, `kb_end` for score, `session_start`/`session_end` for duration). This is cleaner and needed for the score column in D-23.

4. **Plan badge data — template context**
   - What we know: `g.org.plan` is set in the `before_request` auth handler and available in all templates via `g`. No changes needed to expose it.
   - What's unclear: Whether `Starter` / `Pro` / `Business` display names map cleanly from `Organisation.plan` values.
   - Recommendation: `Organisation.plan` field stores `'starter'` / `'pro'` / `'business'` (confirmed from PLANS dict). Use `| capitalize` in Jinja for display: `starter` → `Starter`. Acceptable for this phase.
