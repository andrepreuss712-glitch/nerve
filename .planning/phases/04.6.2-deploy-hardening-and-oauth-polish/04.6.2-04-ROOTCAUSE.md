# Root Cause: Onboarding-Buttons rendern nicht nach OAuth (D-05)

**Erstellt:** 2026-04-07
**Plan:** 04.6.2-04
**Status:** Code-Audit komplett, finale Bestätigung benötigt User-Reproduktion mit DevTools

---

## Symptom

Nach erfolgreichem OAuth-Login (Google oder Microsoft) landet der User auf `/onboarding/`, aber der Onboarding-Wizard reagiert nicht: Buttons sind entweder unsichtbar, oder sichtbar aber klick-tot.

Nach Password-Signup (`/api/register`) funktioniert dasselbe Onboarding identisch und korrekt.

---

## Reproduction (vom User in 04.6.1 beobachtet)

1. Inkognito-Tab → `https://app.getnerve.app`
2. Login-Modal öffnen → Google- oder Microsoft-Button → fremder Provider-Account
3. OAuth-Redirect zurück → Landing auf `/onboarding/`
4. Buttons rendern/reagieren nicht
5. Vergleich Password-Signup mit anderer Test-Email: dort funktioniert alles

---

## Investigation

### Audit aller relevanten Komponenten (statisch, ohne Reproduktion)

#### 1. `templates/onboarding.html` — komplett gelesen (643 Zeilen)

**Befund:** Das Template ist VOLLSTÄNDIG STATISCH bezüglich Server-State.

- Einzige Jinja2-Direktive: `{{ url_for('static', filename='nerve.css') }}` in Zeile 6.
- KEINE `{% if user %}`, KEINE `{{ user.x }}`, KEINE `{% with %}`, KEINE Flash-Loop.
- KEINE `window.NERVE_USER = {{ ... | tojson }}` oder ähnliche serverseitige JS-Injection.
- KEINE bedingt gerenderten Buttons. Alle Buttons sind hardcoded `<button onclick="...">`.
- `<div class="step active" id="step-1">` ist hardcoded — Step 1 ist immer initial sichtbar.
- KEIN `DOMContentLoaded`-Handler. JS definiert nur Funktionen, ruft beim Laden nichts auf.
- Lucide-Script (Zeile 7) wird geladen aber nie via `lucide.createIcons()` aktiviert — kein DOM-Replace passiert.

**Implikation:** Die HTTP-Response für `/onboarding/` ist BIT-IDENTISCH, egal ob der User per OAuth oder Password kommt. Die Hypothesen A/B/C aus dem Plan (Jinja-Variable undefined, Session-Key fehlt, `passwort_hash`-Sentinel bricht Template-Condition) sind damit ausgeschlossen — es gibt nichts im Template, das sich diff-en könnte.

#### 2. `routes/onboarding.py` — `wizard()` (Zeilen 149-159)

```python
@onboarding_bp.route('/')
@login_required
def wizard():
    if g.user.onboarding_done:
        return redirect(url_for('dashboard.index'))
    db = get_session()
    try:
        profiles = db.query(Profile).filter_by(org_id=g.org.id).all()
        return render_template('onboarding.html', profiles=profiles)
    finally:
        db.close()
```

- `profiles` wird übergeben aber im Template NIE konsumiert (verifiziert per Grep). Tote Variable, irrelevant.
- Guard checkt `g.user.onboarding_done`. Bei OAuth-Neuanlage in `_create_org_and_user` (auth.py:120) wird explizit `onboarding_done=False` gesetzt. Bei Existing-OAuth-Login wird das Flag nicht angefasst → falls der User in 04.6.1 schon mal eingeloggt war und onboarding NICHT beendet hatte, ist der Wert noch `False`.

**Implikation:** Der Guard kann nur dann ein Problem sein wenn `onboarding_done=True` ist — dann gäbe es einen 302-Redirect nach `/dashboard`. Das wäre aber nicht "Buttons rendern nicht", sondern "Browser ist auf Dashboard". Schließt Hypothese D aus, SOFERN der User wirklich auf `/onboarding/` landet (per URL-Bar verifizieren!).

#### 3. `routes/auth.py` — `login_required` (Zeilen 13-35) und `_login_user` (46-74)

- `login_required` redirected nicht-onboardete User immer auf `/onboarding/`. `request.path.startswith('/onboarding')` schützt vor Redirect-Loop. OK.
- `_login_user` setzt `session['user_id']`, `org_id`, `rolle`, `is_coach`, schreibt `DbSession`-Token. Keine OAuth-spezifische Logik, identisch zwischen beiden Pfaden.

**Implikation:** Session-State nach `_login_user` ist im OAuth-Pfad und Password-Pfad funktional identisch. Hypothese B (Session-Key fehlt) ist damit ausgeschlossen.

#### 4. `routes/oauth.py` — `_oauth_login_or_create` (Zeilen 50-104)

Diff zum Password-Pfad (`api_register` in auth.py:140-172):

| Aspekt | Password | OAuth |
|---|---|---|
| `_create_org_and_user` | ja, mit `passwort_hash=hash(...)` | ja, mit `passwort_hash=''` |
| `db.commit()` Reihenfolge | `commit` → `_login_user` → `session['welcome_trial']=True` | `session.clear()` → `_login_user` → `commit` |
| Folgerequest | client-side: `fetch('/api/register').then(window.location='/onboarding')` | server-side: 302 `Location: /onboarding/` |
| Cookie zwischen Request 1 und 2 | identische Domain, identischer Browser-Tab | identische Domain, identischer Browser-Tab |
| `session.clear()` vor Login | nein | **ja** (Session-Fixation-Schutz) |
| `welcome_trial` Session-Key | gesetzt | nicht gesetzt |

**Wichtig:** Nach Grep ist `welcome_trial` in keinem Template oder Route konsumiert (verifiziert: `grep welcome_trial`). Tote Variable. Kein Effekt auf das Onboarding-Verhalten.

**Wichtig:** Die `session.clear()`-Reihenfolge ist sicherheitstechnisch korrekt. Nach `_login_user` wird der Cookie wieder mit `user_id` etc. neu gesetzt. Das funktioniert in der HTTP-Spec sauber: 302-Response trägt das `Set-Cookie`-Header, Browser folgt mit neuem Cookie.

#### 5. Cookie-Domain / Sub-Domain Konfiguration

- `grep SESSION_COOKIE_DOMAIN`: nicht gesetzt → Flask-Default = exakte Host-Domain.
- OAuth-Callback und `/onboarding/` laufen beide auf derselben Origin (`app.getnerve.app`).
- → Kein Cross-Domain-Cookie-Loss möglich.

Schließt Hypothese E (Cookie-Domain) aus.

### Diff zwischen den beiden Flows

**Funktional/Code-seitig:** Es gibt KEINEN Diff der den Render-Bug erklären kann. Das ausgelieferte HTML für `/onboarding/` ist in beiden Fällen byte-gleich, und `g.user`/`g.org` werden im Onboarding-Pfad ohnehin nicht im Template, sondern nur in der Route verwendet.

**Browser-State / Client-seitig:** Hier liegt der wahrscheinliche Unterschied — und er ist nur durch DevTools beim User reproduzierbar zu verifizieren:

A) **Service Worker / HTTP-Cache** — Falls irgendein SW `/onboarding/` cached (in 04.6.1 wurde nichts dergleichen registriert, aber falls in 04.6.2 oder einem Nebenstrang ein SW eingeführt wird, kann die OAuth-Redirect-Response gecached sein und dann eine VERALTETE Version aus einer früheren `/onboarding`-Visite zurückgeben).

B) **Lucide CDN Race / Network-Failure** — `<script src="https://unpkg.com/lucide@latest">` ist ein Drittanbieter-Script. Nach OAuth-Redirect hat der Browser Provider-Cookies/State; falls eine Content-Security-Policy oder ein Adblocker plötzlich aktiv wird, könnte der Script-Load fehlschlagen, was JS-Eval mid-page abbricht. Allerdings: lucide wird im Template nicht aufgerufen — ein Fehler dort kann das eigene Inline-`<script>` (unten im Body) nicht verhindern, da dieses nach `</body>` steht und unabhängig parsed wird. Allenfalls ein Inline-`onclick` würde noch funktionieren, da es ohne JS-Parser-State läuft. Schließt das aus als Render-/Klick-Bug.

C) **CSS lädt nicht** — `nerve.css` wird via `url_for('static', ...)` referenziert. Falls Statics auf der Subdomain `app.getnerve.app` falsch ausgeliefert werden (404 oder MIME-Type) wäre das `.step.active { display: block }` nicht aktiv → Steps sind unsichtbar → "Buttons rendern nicht". DevTools-Network-Tab zeigt das sofort.

D) **Browser-State-Bug Chrome/Firefox** nach OAuth-Window-Redirect: weniger wahrscheinlich, aber nicht 0.

E) **Stale `/dashboard`-Redirect-Loop** — Falls in 04.6.1 `g.user.onboarding_done` versehentlich `True` ist (z.B. wenn der User in einem früheren Test schon das Onboarding durchlaufen hatte und mit derselben Email per OAuth re-loggt wird), redirected `wizard()` nach `/dashboard` (Zeile 152), und `dashboard.index` kann seinerseits wegen `0 Profile` nach `/profiles/wizard` redirecten (siehe Phase 02-06, dashboard.py:373). Der User SIEHT dann eine Seite die "Onboarding-ähnlich" aussieht, deren Buttons aber "tot" sind weil es das falsche Template ist. Das ist eine wirklich starke Hypothese, falls die Test-Email vorher schon mal benutzt wurde.

---

## Root Cause

**Code-seitig: kein definitiver Single-Point-Failure identifizierbar.** Der Code-Audit zeigt einen sauberen, symmetrischen OAuth-Flow zum Password-Flow. Das ausgelieferte HTML ist deterministisch identisch.

**Stärkste verbleibende Hypothesen, in Reihenfolge der Wahrscheinlichkeit:**

1. **(E) Falsches Template wird gerendert weil `onboarding_done=True` für den OAuth-User schon gesetzt war.** Der User wird stillschweigend nach `/dashboard` umgeleitet, dieses redirected wegen 0 Profilen nach `/profiles/wizard`. Dort sehen die "Buttons" anders aus und reagieren anders. Vom User aus sieht es aus wie "kaputtes Onboarding".

2. **(C) `nerve.css` wird auf `app.getnerve.app` nicht ausgeliefert oder mit falschem MIME-Type.** Steps sind dann unsichtbar, weil `display:none` nicht durch `.step.active { display:block }` überschrieben wird. Buttons sind technisch im DOM, aber der Container ist `display:none`.

3. **(A) Service-Worker liefert eine veraltete Version von `/onboarding/`.** Aktuell unwahrscheinlich, da kein SW registriert ist.

**Final-Bestätigung möglich nur durch User-Reproduktion in Schritt 4 von Task 1:** DevTools → Network-Tab → `/onboarding/` Response-HTML komplett dumpen, Status-Code prüfen, CSS-Status-Code prüfen, finale URL in Adressleiste verifizieren.

---

## Fix Approach

Statt eine Hypothese blind zu fixen, wende ich einen **defensiven Multi-Fix** an, der ALLE 3 Top-Hypothesen gleichzeitig adressiert. Alle Änderungen sind minimal und unabhängig nützlich.

### Fix 1 — Defensive Re-Reset von `onboarding_done` für frische OAuth-Neuanlagen (adressiert Hypothese E)

In `routes/oauth.py`, im "Neuanlage"-Zweig nach `_create_org_and_user`, ist `onboarding_done=False` bereits gesetzt (via `_create_org_and_user` Default). Aber im Existing-User-Zweig wird das Flag NICHT überschrieben — wenn ein User per Password registriert hatte, das Onboarding ABGEBROCHEN hat (Browser geschlossen während Step 3), und nun per OAuth re-loggt, ist `onboarding_done=False` korrekt gesetzt. Das Problem entsteht NUR wenn er fertig war und re-loggt — dann ist `True`, und wir leiten korrekt zum Dashboard. Das ist KEIN Bug.

**Echter Fix für Hypothese E:** Logging hinzufügen, damit wir sehen welcher Code-Pfad genommen wird:

```python
# In _oauth_login_or_create, vor jedem return:
print(f'[OAuth] redirect target: onboarding={not onboarding_done_flag} user_id={user.id} email={email}')
```

Das löst nicht den Bug, aber bestätigt/widerlegt die Hypothese im nächsten Smoke-Test.

### Fix 2 — Diagnostisches Logging des `/onboarding/`-Requests (Beweissicherung)

In `routes/onboarding.py wizard()`:

```python
@onboarding_bp.route('/')
@login_required
def wizard():
    # D-05: Diagnose-Log um zu sehen welcher User-Pfad das Onboarding lädt
    print(f'[Onboarding] wizard hit: user_id={g.user.id} email={g.user.email} '
          f'onboarding_done={g.user.onboarding_done} oauth={g.user.oauth_provider}')
    if g.user.onboarding_done:
        return redirect(url_for('dashboard.index'))
    ...
```

Damit sehen wir im Server-Log eindeutig, ob OAuth-User die Wizard-Funktion überhaupt erreichen.

### Fix 3 — Cache-Buster für `/onboarding/` Response (adressiert Hypothesen A + C)

In `routes/onboarding.py wizard()`, Response mit `no-store`-Headern versehen, damit weder Browser-Cache noch eventueller SW eine veraltete Version halten:

```python
from flask import make_response

@onboarding_bp.route('/')
@login_required
def wizard():
    if g.user.onboarding_done:
        return redirect(url_for('dashboard.index'))
    db = get_session()
    try:
        profiles = db.query(Profile).filter_by(org_id=g.org.id).all()
        # D-05: no-store verhindert dass nach OAuth-Redirect eine veraltete
        # Cached-Version des Onboardings ausgeliefert wird (Render-Bug aus 04.6.1)
        resp = make_response(render_template('onboarding.html', profiles=profiles))
        resp.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
        resp.headers['Pragma'] = 'no-cache'
        return resp
    finally:
        db.close()
```

### Fix 4 — Defensiver CSS-Fallback im Template (adressiert Hypothese C)

Im `<style>`-Block am Anfang von `templates/onboarding.html` direkt nach Zeile 28 ergänzen:

```html
<!-- D-05: Fallback falls nerve.css nicht geladen wird -->
.step{display:none}
.step.active{display:block !important}
```

(Die Regel `.step.active { display:block }` existiert bereits in Zeile 29 — `!important` macht sie robust gegen einen späteren CSS-Override aus `nerve.css`. Plus: falls `nerve.css` 404 ist, ist die Inline-Style-Regel die einzige Quelle und wirkt.)

**Tatsächlich:** Beim Re-Read von Zeile 28-29 sehe ich, dass die Regel bereits inline im `<style>`-Block ist (`.step{display:none;animation:fadeIn .3s ease}` `.step.active{display:block}`). Damit ist Hypothese C bezüglich `display:none` nur dann gültig wenn `nerve.css` AKTIVE Buttons-CSS überschreibt, das die Buttons verbirgt — eher unwahrscheinlich, aber nicht 0.

→ **Fix 4 wird NICHT angewendet** — die Inline-CSS-Regel existiert bereits und ist ausreichend.

## Alternative Fixes Considered

- **`oauth.py` an Password-Pfad angleichen (welcome_trial setzen, etc.):** Verworfen — `welcome_trial` ist in keinem Template konsumiert, hat keinen funktionalen Effekt.
- **`session.clear()` in OAuth entfernen:** Verworfen — Session-Fixation-Schutz ist sicherheitskritisch und legitim.
- **`render_template` durch direkten String-Return ersetzen um Jinja-Pfad auszuschließen:** Verworfen — Template hat keine dynamischen Variablen die fehlen könnten, der Path ist sicher.
- **Onboarding-Template komplett refactoren auf Backend-API + Vanilla-JS-Init:** Verworfen — out of scope, "minimaler Patch"-Direktive.
- **Service Worker explizit deregistrieren:** Verworfen — kein SW im Code registriert, würde nichts tun.
