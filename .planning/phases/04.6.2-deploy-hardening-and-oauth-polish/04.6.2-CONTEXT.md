<decisions>
## Implementation Decisions

### Deploy-Pipeline

**D-01: deploy.sh auf rsync umbauen**
- `scripts/deploy.sh` (oder wo immer das Deploy-Skript liegt) muss auf `rsync` umgestellt werden, damit alle relevanten Ordner automatisch übertragen werden.
- Hintergrund: Beim 04.6.1-Deploy wurde der Ordner `database/` vom Skript nicht mitgenommen — musste manuell nachgeladen werden. Darf nie wieder passieren.
- Excludes sauber definieren: `.git`, `.env`, `__pycache__`, `*.pyc`, `.planning`, `node_modules`, ggf. `database/*.db` (Prod-DB NICHT überschreiben!).
- Include by default statt Include-Liste: Standard ist "alles rüber außer Excludes".
- Locked: Prod-SQLite-Datei (`database/salesnerve.db`) darf beim Deploy NICHT überschrieben werden. Schema-Ordner schon, Daten nicht.

### OAuth Landing & UI

**D-02: Landing Page Login-Modal → app.getnerve.app**
- Der Login-Modal auf der Landing (`templates/landing.html` oder wo der Einstieg liegt) soll auf `https://app.getnerve.app` verlinken statt auf den alten Flow.
- Trennung Marketing (`getnerve.app`) vs. App (`app.getnerve.app`) wird damit im UI vollzogen.
- Bei "Login"-Klick: Redirect zu `app.getnerve.app` (ggf. mit Ziel-Parameter wenn vorhanden).

**D-03: Microsoft Button-Text**
- Button-Label auf "Mit Microsoft 365 anmelden" (exakter Wortlaut).
- Gilt überall wo der MS-Login-Button erscheint (Landing-Modal, app-Login, ggf. Signup).
- Begründung: Klarstellung dass Business-Accounts erwartet werden.

### OAuth-Fehlermeldungen & Flow

**D-04: Deutsche Fehlermeldung für private Microsoft-Konten**
- Wenn ein User versucht sich mit einem persönlichen Microsoft-Konto (Hotmail/Outlook/Live/Consumer) einzuloggen, soll eine klare deutsche Fehlermeldung erscheinen statt generischer Auth-Error.
- Vorschlag (final wording in Planung): "NERVE benötigt ein Microsoft 365 Business-Konto. Private Microsoft-Konten (Hotmail, Outlook, Live) werden nicht unterstützt. Bitte verwende deinen Firmen-Account oder melde dich mit Google an."
- Erkennung: über den MS-Graph Response oder den `tid` (tenant id) Claim — Consumer-Tenant hat eine feste ID (`9188040d-6c67-4c5b-b112-36a304b66dad`).
- Anzeige-Ort: `oauth_error` Query-Param + Landing-Modal Fehler-Banner (wie bereits in 04.6.1 existiert).

**D-05: Onboarding-Buttons fixen nach OAuth-Redirect**
- Bug aus 04.6.1: Nach OAuth-Login landet der User im Onboarding, aber die Buttons rendern nicht / reagieren nicht.
- Root Cause im Plan analysieren (vermutlich: JS erwartet einen State der beim OAuth-Flow nicht gesetzt wird, oder Template-Variable fehlt im OAuth-Pfad).
- Fix: Onboarding muss nach OAuth-Redirect identisch funktionieren wie nach Password-Signup.

**D-06: `prompt=consent` nur beim ersten Login**
- Aktuell wird bei jedem OAuth-Login `prompt=consent` an Google/Microsoft geschickt → User muss jedes Mal Berechtigungen bestätigen (nervig).
- Optimierung: `prompt=consent` nur beim allerersten Login des Users mit diesem Provider. Bei Folge-Logins weglassen (= Silent-SSO).
- Erkennung: `user.oauth_provider`/`oauth_id` bereits in DB → kein `prompt=consent` mehr. Nur wenn der Provider-Account frisch verknüpft wird, Consent anfordern.
- Gilt für Google UND Microsoft gleichermaßen.
