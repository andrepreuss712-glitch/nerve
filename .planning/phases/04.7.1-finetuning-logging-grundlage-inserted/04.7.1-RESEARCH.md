## Architecture Patterns

### Migrations-Pattern (bestehend, beibehalten)

**Verifiziert in `app.py:47-226`:** Migrations laufen idempotent in `_migrate()` über:
```python
for col, typedef in [('new_col', 'VARCHAR(100)'), ...]:
    try:
        conn.execute(text(f'ALTER TABLE users ADD COLUMN {col} {typedef}'))
        conn.commit()
    except Exception:
        pass  # Column exists already
```
Plus `CREATE TABLE IF NOT EXISTS` für neue Tabellen (siehe `planning_feedback_link`-Block ab Zeile 186). **Neue Tabellen werden jedoch bevorzugt über `Base.metadata.create_all(engine)` angelegt** — der `init_db(engine)`-Call in `app.py:44` erzeugt automatisch alle in `models.py` definierten Tabellen, die noch nicht existieren. Wir hängen uns daran an, **kein** manuelles `CREATE TABLE` für die `ft_*`-Tabellen nötig.

**Empfehlung Phase 04.7.1:**
- 4 neue SQLAlchemy-Models in `database/models.py` (Base.metadata.create_all regelt Erstellung).
- `_migrate()` erweitern **nur für die `ALTER TABLE`-Calls** auf `users` (market, language) und `conversation_logs` (market, language).
- Prompt-Seed als eigene idempotente Funktion direkt nach `_migrate()` in `app.py` (analog zum Audit-Log-Trigger-Block Zeile 229-249).

### Model-Konventionen (aus `database/models.py`)

- `Column(Integer, primary_key=True)` – Auto-Increment
- `Column(DateTime, default=utcnow)` – `utcnow` ist module-lokal definiert (timezone-naive UTC)
- FK-Syntax: `Column(Integer, ForeignKey('tablename.id'), nullable=...)`
- JSON-Spalten: im bestehenden Code als `Column(Text)` mit Kommentar `# JSON` — siehe `Profile.daten`, `ConversationLog.gegenargument_details`. **Empfehlung für 04.7.1:** `Column(Text)` für `context_window`/`customer_data`/`profile_data`/`active_learning_cards` — konsistent mit Codebase. PG-Migration wird später über SQLAlchemy-Column-Typänderung auf `JSONB` erfolgen.
- Tabellennamen: lowercase + underscore, plural (`conversation_logs`, `objection_events`).
- Deutsche Feldnamen sind ok für Business-Domain, englische ok für generische Felder — beide Patterns sind präsent.

## Common Pitfalls

### Pitfall 1: DB-Session im Background-Thread
**Was geht schief:** `analyse_loop()` läuft in einem Thread außerhalb des Flask-Request-Contexts. Kein `g.db`, kein scoped-session.
**Wie vermeiden:** Neue Session per Event über `SessionLocal()` öffnen, try/finally `db.close()`. Beispiel-Pattern existiert in `services/audit.py` (nicht gelesen, aber erwartet).
**Warnzeichen:** `RuntimeError: Working outside of application context` oder stale sessions.

### Pitfall 2: `prompt_version` inkonsistent geloggt
**Was geht schief:** Zwischen SELECT (prompt_versions WHERE is_active=1) und INSERT (ft_assistant_events) wird `is_active` nicht durchgängig gehalten → verwechselte Version.
**Wie vermeiden:** Prompt-Version einmal beim App-Start in ein module-level `_ACTIVE_PROMPT_VERSIONS = {'assistant_live': 'v1.0.0'}` dict laden (oder per Request aus `g`). Nicht bei jedem Write neu querien.
**Warnzeichen:** Performance-Einbruch durch N+1 SELECT bei jedem Hint.

### Pitfall 3: `transcript_segment` versehentlich im Cold-Call gefüllt
**Was geht schief:** Im Cold-Call wird der Berater-Text als "transcript_segment" reingeschrieben → DSGVO-Graubereich (Cold-Call-Modus loggt aktuell auch keine Beratertexte in ConversationLog, sondern nur Metadaten).
**Wie vermeiden:** `speaker` + `transcript_segment` nur füllen, wenn `session_mode == 'meeting'` UND Consent gegeben. Im Cold-Call beide auf `NULL` setzen; nur `conversation_phase`, `readiness_score`, `hint_type`, `hint_text` speichern.
**Warnzeichen:** Vertrauen in Briefing-Beispiel `transcript_segment TEXT NOT NULL` — das ist die Stelle, wo D-04/D-05 greifen.

### Pitfall 4: Alte User haben kein `market`-Feld
**Was geht schief:** `ALTER TABLE ... ADD COLUMN market TEXT NOT NULL` scheitert in SQLite ohne DEFAULT.
**Wie vermeiden:** `DEFAULT 'dach' NOT NULL` — SQLite akzeptiert das, alle bestehenden Rows bekommen den Default.
**Warnzeichen:** Migration-Failure beim ersten Deploy.

### Pitfall 5: `init_db` ruft `create_all` nach `_migrate`
**Was geht schief:** `app.py:44` ruft `init_db(engine)` **vor** `_migrate()`. Neue Tabellen werden also schon erzeugt, bevor Migrations laufen — das ist ok für Neustart. Aber: Die `ft_*`-Models müssen beim Import in `models.py` präsent sein, damit `create_all` sie erfasst.
**Wie vermeiden:** Models am Ende von `models.py` anhängen, **nicht** in ein separates Modul auslagern, das nur lazy importiert wird.
**Warnzeichen:** Tabellen fehlen beim ersten Start.

_ACTIVE_PROMPT_CACHE = {}

def get_active_prompt_version(module: str) -> str:
    if module in _ACTIVE_PROMPT_CACHE:
        return _ACTIVE_PROMPT_CACHE[module]
    from database.db import SessionLocal
    from database.models import PromptVersion
    db = SessionLocal()
