# Phase 04.7.1 Plan 05: FT-Lifecycle Write-Hooks + JSONL-Export-CLI Summary

Lifecycle-Write-Hooks fuer ft_call_sessions (INSERT beim Socket start_live_session, UPDATE bei api_beenden) plus ft_objection_events INSERT beim EWB-Button-Klick — plus CLI-Stub fuer spaeteren Fine-Tuning-Export. Damit ist der komplette Write-Pfad von Session-Start bis Export als Minimum-Viable-Lifecycle geschlossen.

## What Was Built

**Task 1 — ft_call_sessions lifecycle:** `services/deepgram_service.py::handle_start_live_session` wurde direkt nach `_open_deepgram_connection(...)` um einen try/except-Block erweitert, der `user_id` aus `flask.session` liest, `market`/`language` per `User`-Lookup aus der DB holt (Fallback `dach`/`de`), eine `FtCallSession`-Row mit `mode`, `user_id`, `market`, `language` und initialen Zaehlern (`hints_shown=0`, `hints_used=0`, `buttons_pressed=0`) anlegt und die erzeugte `ft_session_id` plus `user_id`/`market`/`language` in `ls.state` abspeichert. In `routes/app_routes.py::api_beenden` wurde nach dem bestehenden ObjectionEvent-Commit ein UPDATE-Block eingefuegt: liest `ft_session_id` aus `ls.state`, holt die Row ueber `db_conv.query(FtCallSession)`, setzt `duration_seconds` aus `postcall['dauer_sek']`, `hints_shown`/`hints_used`/`buttons_pressed` (aus `log_entries` + `hilfe_count` + `ewb_clicks`-Laenge), `readiness_score_start`, `readiness_score_end`, `conversation_log_id`, `outcome`, commited und setzt `ls.state['ft_session_id'] = None`. Commit: `48cd165`.

**Task 2 — ft_objection_events + export CLI + lifecycle test:** In `api_ewb_trigger` wurde direkt nach dem bestehenden `record_ewb_click(...)` ein zweiter try/except-Block ergaenzt, der eine frische `get_session()` oeffnet und `FtObjectionEvent` mit `objection_type=einwand_typ`, `recommended_response=antwort`, `model_used='claude-haiku-4-5-20251001'` (exakt wie im bestehenden Haiku-Call) und `prompt_version=get_active_prompt_version('objection_trigger')` schreibt. `scripts/__init__.py` und `scripts/export_ft_jsonl.py` wurden neu angelegt. Das CLI nutzt `argparse` (`--market`, `--out`, `--table assistant|objection`), oeffnet `dbmod.SessionLocal()`, filtert per `market` und serialisiert jede Row via `__table__.columns` als JSON-Zeile. `tests/test_ft_lifecycle.py` seedet Organisation + User + FtCallSession + je einen FtAssistantEvent und FtObjectionEvent, monkeypatcht `database.db.SessionLocal` und ruft `main()` zweimal auf (beide Tabellen) — prueft `rc==0`, Zeilen-Anzahl und Feldwerte. Commit: `4a3d763`.

## Deviations from Plan

**1. [Rule 3 — Blocking] Plan referenziert `services/audio_service.py`, real existiert die Datei nicht**
- **Issue:** Plan-Text sagt `services/audio_service.py` fuer den `start_live_session` Handler, aber seit Phase 04.1 liegt der Handler in `services/deepgram_service.py::register_audio_handlers.handle_start_live_session`. Der Plan hat dies sogar explizit als Alternative erwaehnt ("oder wo auch immer der Handler start_live_session lebt").
- **Fix:** INSERT-Hook in `services/deepgram_service.py` eingefuegt (korrekte Stelle).
- **Commit:** 48cd165

**2. [Rule 1 — Bug] Test-User ohne `org_id` schlug an `NOT NULL`-Constraint fehl**
- **Issue:** Initialer Test-Draft seedete User ohne Organisation. SQLAlchemy wirft `IntegrityError: NOT NULL constraint failed: users.org_id`.
- **Fix:** Test seedet jetzt erst `Organisation(name='Test Org')`, dann `User(org_id=org.id, ...)`.
- **Commit:** 4a3d763

**3. [Rule 2 — Missing functionality] `db_conv`-Scope nicht verfuegbar in api_ewb_trigger**
- **Issue:** Plan skizziert `db_conv.add(FtObjectionEvent(...))` in `api_ewb_trigger`, aber der Handler oeffnet bisher keine DB-Session (nutzt nur in-memory `ls.state`).
- **Fix:** Frischer `get_session()`-Scope im FT-Hook geoeffnet, im `finally` geschlossen.
- **Commit:** 4a3d763

**4. [Rule 2] `model_used` Literal `'claude-haiku-4-5-20251001'` statt Plan-Literal `'claude-haiku-4-6'`**
- **Issue:** Plan nennt `claude-haiku-4-6` als model_used, der reale Code in `api_ewb_trigger` ruft aber `claude-haiku-4-5-20251001` auf.
- **Fix:** Literal exakt gematcht (wie im Plan gefordert: "EXAKT wie im Haiku-Call oben im Handler").
- **Commit:** 4a3d763
