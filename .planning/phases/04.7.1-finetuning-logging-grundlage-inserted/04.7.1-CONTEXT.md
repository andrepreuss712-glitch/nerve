# Phase 04.7.1: FineTuning Logging Grundlage - Context

**Gathered:** 2026-04-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Strukturierte Logging-Grundlage für Fine-Tuning Trainingsdaten von Tag 1. Liefert 4 Tabellen produktiv:
`ft_call_sessions`, `ft_assistant_events`, `ft_objection_events`, `prompt_versions` — plus Markt-Feld (`dach`/`us`) auf `users` und allen Session-Tabellen.

**In Scope:**
- SQLAlchemy-Models für die 4 Tabellen (PG-kompatibel)
- Markt-Spalten (`market`, `language`) auf `users` + Session-Tabellen, Default `'dach'`/`'de'`
- Synchrones Schreiben der Log-Events aus dem Live-Assistent-/Einwand-Button-Pfad
- Seed der aktiv genutzten System-Prompts als `v1.0.0` in `prompt_versions`
- Auflösung der aktuellen Prompt-Version über `is_active=1`

**Out of Scope (bewusst verschoben):**
- `ft_training_events`, `ft_learning_card_events` → nach Early Access Start
- JSONL-Export-Scripts und Qualitäts-Filter → "nach 3 Monaten"
- UI für Markt-Einstellung im Settings-Screen → Phase 4.16.2
- Tatsächliches Fine-Tuning oder Training-Pipeline

</domain>

<decisions>
## Implementation Decisions

### Write-Path & Performance
- **D-01:** Synchrones Schreiben im Live-Assistent-Pfad. Kein Async-Queue, kein Redis, kein Background-Worker. Begründung: 50 Early-Access-User, Latenz unkritisch, No-Over-Engineering-Prinzip. Falls später Performance-Problem: refactor in eigener Phase.

### Anonymisierung
- **D-02:** Rohdaten landen unverändert in den `ft_*`-Tabellen. Anonymisierung/PII-Scrubbing passiert erst **beim Export** (Phase nach Monat 3). DSGVO-Compliance wird über Zugangskontrolle (Datenbank-Zugriff nur André/Superadmin) abgedeckt, nicht über Destruktion der Rohdaten.
- **D-03:** Cold-Call-Modus loggt weiterhin **kein** Kundentranskript (bestehende DSGVO-Architektur-Entscheidung aus CLAUDE.md — unangetastet). Meeting-Modus loggt nur nach Consent (bleibt wie bisher).

### Schema-Design
- **D-04:** `transcript_segment` in `ft_assistant_events` wird **nullable** — kein Platzhalter-String. Im Cold-Call-Modus ist NULL ehrlich (kein Kundentranskript verfügbar). Gleiches Prinzip für `context_window` wenn keine Kundenseite vorhanden.
- **D-05:** Abweichung vom Briefing: Im Spec ist `transcript_segment TEXT NOT NULL` — wir ändern zu `nullable=True`. Begründung steht in D-04.
- **D-06:** SQLAlchemy-Models statt raw `CREATE TABLE`. PostgreSQL-Kompatibilität von Tag 1 (JSON-Spalten als `JSON`/`JSONB`, Autoincrement via `Integer primary_key`, FKs explizit). SQLite bleibt das aktuelle Entwicklungs-Backend, aber nichts im Schema darf PG blockieren.

### Markt-Trennung
- **D-07:** `ALTER users ADD COLUMN market TEXT NOT NULL DEFAULT 'dach'` und `language TEXT NOT NULL DEFAULT 'de'`. Alle bestehenden User erben `'dach'` / `'de'`. Kein Settings-UI in dieser Phase — kommt in Phase 4.16.2.
- **D-08:** `market` auf `ft_call_sessions` wird beim Insert aus `users.market` abgeleitet (nicht manuell gesetzt). Assistant/Objection-Events leiten den Markt über `session_id` → `ft_call_sessions.market` ab.

### Prompt-Versionierung
- **D-09:** Initialer Seed: Alle aktuell produktiv genutzten System-Prompts werden als `v1.0.0` in `prompt_versions` eingefügt (Live-Assistent, Einwand-Classifier, und weitere, die der Researcher im Code finden muss). Genau **ein** Eintrag pro `module` hat `is_active=1`.
- **D-10:** Aktuelle Prompt-Version wird zur Write-Zeit via `SELECT version FROM prompt_versions WHERE module=? AND is_active=1` aufgelöst und als String in `model_used`/`prompt_version`-Spalten der Event-Tabellen gespeichert. Kein ENV, kein Config-File.

### Foreign Keys & bestehende Tabellen
- **D-11:** Researcher-Aufgabe (vor Planning): Welche der referenzierten Tabellen existieren heute im Code — `call_sessions`, `training_sessions`, `learning_cards`, `users`? Für nicht existierende Tabellen: FK **nullable** lassen, keine Stub-Tabellen erzeugen. Stub-Erzeugung gehört in die Phase, die das Feature baut (z.B. `learning_cards` → Phase 4.11).

### Claude's Discretion
- Migrations-Mechanismus (Alembic vs. manuelles SQL-Script): Claude wählt das, was bereits im Repo etabliert ist. Falls nichts etabliert ist: Alembic einführen.
- Konkrete Modul-Namen für `prompt_versions.module`-Spalte (z.B. `'assistant_live'`, `'objection_classifier'`) — Researcher schlägt Liste vor, basierend auf dem, was im Code gefunden wird.
- Test-Strategie für die Models (pytest + in-memory SQLite oder gegen Dev-DB) — Claude wählt pragmatisch.

</decisions>

<specifics>
## Specific Ideas

- "Rohdaten behalten, DSGVO über Zugangskontrolle" — André will Datenqualität für späteres Fine-Tuning nicht durch frühe Destruktion verlieren. Anonymisierung ist ein Export-Schritt, kein Insert-Schritt.
- "NULL ist ehrlich" — Für Cold-Call-Events wird `transcript_segment` explizit NULL gelassen statt mit einem Platzhalter gefüllt. Das macht später Export-Filter trivial.
- "Kein Over-Engineering" — 50 User bedeuten: synchroner DB-Write ist vollkommen ausreichend. Erst bei echtem Latenzproblem async machen.

</specifics>
